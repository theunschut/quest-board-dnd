# Phase 54 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none — server-rendered ASP.NET Core 10 MVC + Razor views; no `components.json`/`package.json` in repo (confirmed) |
| Preset | not applicable |
| Component library | Bootstrap 5 (existing project-wide dependency, no change) |
| Icon library | Font Awesome (`fas fa-*`, existing project-wide dependency, no change) |
| Font | inherited from site-wide layout, unchanged by this phase |

This phase touches two existing Razor view files (`Details.cshtml`, `Details.Mobile.cshtml`) and one existing controller action (`QuestController.JoinFinalizedQuest`). It introduces **zero new components, zero new CSS classes, and zero new visual language** — it is a like-for-like port of an already-shipped desktop pattern into the mobile view, using mobile's own already-established CSS conventions. Per CLAUDE.md's UI/UX Design Guidelines, the desktop `modern-card`/`modern-card-header`/`modern-card-body` classes are the standard for *new desktop views* — but this project's mobile views use a separate, already-established convention (`quest-section-card-mobile`, `quest-section-heading`, defined in `quests.mobile.css`) that this phase must follow instead, per Pitfall 3 in 54-RESEARCH.md.

---

## Spacing Scale

Declared values (must be multiples of 4): project already uses Bootstrap's spacing utilities (`mb-2`, `mb-3`, `mt-2`, `gap-2`, `p-12px` via the existing `quest-section-card-mobile` `padding: 12px`) — no new spacing scale is introduced by this phase.

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon-to-text gaps (`me-1`) |
| sm | 8px | `gap-2` between stacked buttons/forms (Bootstrap `d-grid gap-2`) |
| md | 12px | Card internal padding (existing `.quest-section-card-mobile { padding: 12px }` — a pre-existing exception to the 8pt scale, already baked into mobile's design system; do not change) |
| md-alt | 16px | `mb-3` section-to-section gaps (matches sibling "Update Your Vote"/"Choose a Date" cards) |
| lg | 24px | Not used in this phase |

Exceptions: **12px card padding** is a pre-existing mobile convention (`.quest-section-card-mobile`, `.participant-list-mobile`, `.quest-header-card-mobile` all use it) — reuse verbatim, do not round to 16px. **44px minimum touch-target height** applies to all 3 join buttons (`min-height: 44px`, matching the existing `.calendar-vote-row .btn` convention in `quests.mobile.css` lines 135-140) — this is the phase's one explicit touch-target exception, consistent with the project's established mobile convention.

## Implementation Notes (Executor Reference)

These notes translate the design contract above into the concrete markup/structure decisions an executor needs, consistent with 54-CONTEXT.md D-01/D-02 and 54-RESEARCH.md's verified file/line references.

### 1. New mobile "Join This Quest" card — structure

Insert immediately after the waitlist section (`Details.Mobile.cshtml`, after line 254's closing `</div>` and its `}`) and before the "DM controls link" comment/block (currently starting at line 257). Follow the exact container pattern already used by the "Update Your Vote" (lines 102-141) and "Choose a Date" (lines 143-179) sibling cards:

```html
@if (userCanJoin)
{
    <div class="quest-section-card-mobile mb-3">
        <h6 class="quest-section-heading mb-3">
            <i class="fas fa-user-plus me-1"></i>Join This Quest
        </h6>

        <div class="alert alert-info">
            <strong>Joining as:</strong> @Model.Player?.Name (@Model.Player?.Email)
        </div>

        <div class="alert alert-success">
            <p class="mb-1"><strong>Quest Date:</strong> @Model.Quest?.FinalizedDate?.ToString("dddd, MMMM dd, yyyy 'at' h:mm tt")</p>
            <p class="mb-1"><strong>Player Slots:</strong> @selectedPlayersCount/@Model.Quest?.TotalPlayerCount filled</p>
            @if (hasSpace)
            {
                <p class="mb-0"><i class="fas fa-check-circle text-success me-1"></i>Space available for players!</p>
            }
            else
            {
                <p class="mb-0"><i class="fas fa-info-circle text-warning me-1"></i>Player slots full &mdash; joining as a Player will place you on the waitlist. You can also join as Assistant DM or Spectator.</p>
            }
        </div>

        <div class="mb-3">
            <label class="form-label">Select Character (Optional)</label>
            <select id="finalizedQuestCharacterMobile" class="form-select">
                <option value="">-- Join without character --</option>
                @foreach (var character in ViewBag.UserCharacters as List<QuestBoard.Domain.Models.Character> ?? new List<QuestBoard.Domain.Models.Character>())
                {
                    var classList = string.Join(", ", character.Classes.Select(c => $"{c.Class} {c.ClassLevel}"));
                    <option value="@character.Id">@character.Name - Level @character.Level (@classList)</option>
                }
            </select>
            <small class="form-text">Choose which character you'll be playing, or join without one.</small>
        </div>

        <div class="d-grid gap-2">
            <form asp-action="JoinFinalizedQuest" method="post" id="joinPlayerFormMobile">
                <input type="hidden" name="questId" value="@Model.Quest?.Id" />
                <input type="hidden" name="selectedRole" value="0" />
                <input type="hidden" name="characterId" id="playerCharacterIdMobile" value="" />
                <button type="submit" class="btn btn-primary w-100">
                    <i class="fas fa-users me-2"></i>Join as Player
                    <small class="d-block mt-1" style="font-size: 0.8em;">Participate in the quest (counts toward player limit)</small>
                </button>
            </form>

            <form asp-action="JoinFinalizedQuest" method="post" id="joinAssistantFormMobile">
                <input type="hidden" name="questId" value="@Model.Quest?.Id" />
                <input type="hidden" name="selectedRole" value="2" />
                <input type="hidden" name="characterId" id="assistantCharacterIdMobile" value="" />
                <button type="submit" class="btn btn-success w-100">
                    <i class="fas fa-user-shield me-2"></i>Join as Assistant DM
                    <small class="d-block mt-1" style="font-size: 0.8em;">Help the DM run the session</small>
                </button>
            </form>

            <form asp-action="JoinFinalizedQuest" method="post" id="joinSpectatorFormMobile">
                <input type="hidden" name="questId" value="@Model.Quest?.Id" />
                <input type="hidden" name="selectedRole" value="1" />
                <input type="hidden" name="characterId" id="spectatorCharacterIdMobile" value="" />
                <button type="submit" class="btn btn-info w-100">
                    <i class="fas fa-eye me-2"></i>Join as Spectator
                    <small class="d-block mt-1" style="font-size: 0.8em;">Watch the session (auto-approved)</small>
                </button>
            </form>
        </div>
    </div>

    <script>
        document.getElementById('finalizedQuestCharacterMobile').addEventListener('change', function () {
            const characterId = this.value;
            document.getElementById('playerCharacterIdMobile').value = characterId;
            document.getElementById('assistantCharacterIdMobile').value = characterId;
            document.getElementById('spectatorCharacterIdMobile').value = characterId;
        });
    </script>
}
```

**Element ID collision guard:** desktop's `Details.cshtml` uses unqualified IDs (`finalizedQuestCharacter`, `playerCharacterId`, etc.). Since desktop and mobile views never render on the same page simultaneously (mutually exclusive view selection via `MobileViewLocationExpander`), a plain copy would not technically collide at runtime — but use the `...Mobile` suffix shown above anyway for clarity/debuggability and to avoid any confusion if a future refactor ever renders partials together. This is a naming-hygiene recommendation, not a functional requirement.

**`hasSpace` / `selectedPlayersCount` variables:** these must be computed inside `Details.Mobile.cshtml` before this block — mirror desktop's exact computation (`Details.cshtml` lines 37-38): `selectedPlayersCount = Model.Quest?.PlayerSignups.Where(ps => ps.IsSelected && ps.Role == SignupRole.Player).Count() ?? 0` and `hasSpace = selectedPlayersCount < Model.Quest?.TotalPlayerCount`. `userCanJoin` mirrors desktop line 40: `User.Identity?.IsAuthenticated == true && !isPlayerSignedUp` (mobile already has `isPlayerSignedUp` computed at line 12).

**Antiforgery:** no explicit `@Html.AntiForgeryToken()` call needed inside these three forms — `asp-action` tag helpers auto-inject it, matching desktop's existing forms exactly (confirmed in 54-RESEARCH.md Pattern 1).

### 2. D-06 copy — apply to both platforms

- **Desktop** (`Details.cshtml` line 314): replace `"Player slots full, but you can join as Assistant DM or Spectator!"` with the locked D-06 string above.
- **Mobile** (new card, shown above): use the same locked string from first render — do not write a "full = rejected" message and then have to fix it later.

### 3. Touch targets & responsive spacing

- All 3 join buttons: Bootstrap `.btn.w-100` (full-width) stacked via `d-grid gap-2` — matches the existing desktop button-group pattern and satisfies the 44px minimum touch-target height already established by `.calendar-vote-row .btn` in `quests.mobile.css`. No new CSS rule is required; Bootstrap's default `.btn` padding on `w-100` buttons already exceeds 44px total height once the sub-label `<small>` is included — if visual QA on a real device finds any button under 44px, add `min-height: 44px` inline or via a new scoped rule, but do not shrink the sub-label text below 0.8em to compensate.
- Card spacing: `mb-3` after the card (matches sibling cards' `mb-3`), `mb-3` internal for the character-select block (matches desktop's `mb-3`).
