## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Authenticated mobile browser → `POST /Quest/JoinFinalizedQuest` (via the new mobile forms) | Untrusted form input crosses into the shared, `[Authorize]`-protected join action. The forms are new; the action they submit to is unchanged and owned by plan 01. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-54V-01 | Spoofing / CSRF | The 3 new mobile `<form asp-action="JoinFinalizedQuest">` forms | mitigate | Forms use the `asp-action` tag helper, which auto-injects the antiforgery token exactly like desktop's existing forms; the target action retains `[ValidateAntiForgeryToken]`. Acceptance criteria forbid a new endpoint and forbid adding a redundant `@Html.AntiForgeryToken()`, keeping behavior identical to the already-covered desktop forms (`AntiForgeryTokenCoverageTests`). |
| T-54V-02 | Elevation of Privilege / Broken Access Control | Rendering the Join card to users who should not see it | mitigate | The card renders only when `userCanJoin = User.Identity?.IsAuthenticated == true && !isPlayerSignedUp`. Server-side authorization is still enforced by the action's `[Authorize]` + duplicate-join guard regardless of what the view renders — the card gate is UX, not the security boundary. Negative tests assert the card is absent for signed-up and unauthenticated users. |
| T-54V-03 | Information Disclosure | Character `<select>` options leaking other users' characters | mitigate | The select is populated from `ViewBag.UserCharacters`, the same user-scoped collection desktop already uses (populated by the shared `Details` GET action); this plan changes no controller/ViewBag data, so no new disclosure surface is created. |
| T-54V-SC | Tampering | npm/pip/cargo installs | accept | No package installs in this phase (RESEARCH.md "Package Legitimacy Audit: Not applicable"). No install task exists; nothing to gate. |

**Note:** This plan adds only server-rendered Razor markup + a copy change + tests; it does not modify the controller, ViewModel, or any authorization/antiforgery attribute. The behavioral security surface (capacity recompute, `[Authorize]`, `[ValidateAntiForgeryToken]`, character ownership) is unchanged here and covered by plan 01's threat model.
