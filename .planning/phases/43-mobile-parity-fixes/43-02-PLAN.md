<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| server→rendered HTML | The Razor view renders the badge conditionally and emits a fixed literal string ("Session Recap Available"). The guard reads `quest.Recap` only to decide whether to render — the recap text itself is NOT emitted into the badge (the badge label is a hard-coded literal). |
| browser→static CSS | The browser fetches `quest-log.mobile.css` (author-controlled static file). No user input crosses into it. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-43-03 | Injection (XSS) | Recap badge markup in `Index.Mobile.cshtml` | accept | The badge label is a fixed literal ("Session Recap Available"), not interpolated user input — no `@quest.Recap` or other user-controlled value is written into the badge. The `!string.IsNullOrWhiteSpace(quest.Recap)` guard only reads the field as a boolean condition. Even if it did render, Razor `@` output is HTML-encoded by default. This markup is a verbatim port of already-shipped, already-safe desktop markup (`Index.cshtml`). No new injection surface. |
| T-43-04 | Tampering | `quest-log.mobile.css` static asset | accept | Author-controlled static file, no runtime interpolation, no package installs, no new dependency. Nothing to sanitize. |
| T-43-SC | Tampering | npm/pip/cargo installs | accept | No package installs in this plan (markup + CSS only); supply-chain vector not applicable. No `[ASSUMED]`/`[SUS]` packages introduced. |

No high-severity threats. This is a port of already-rendered, safely-encoded server-side markup (with a hard-coded literal label, not user input) plus a static CSS rule — it introduces zero new attack surface.
</threat_model>
