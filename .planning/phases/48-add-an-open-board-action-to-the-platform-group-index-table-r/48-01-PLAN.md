<objective>
Add an "Open Board" action button to the Platform area's Group index table so a SuperAdmin can jump straight into any group's quest board from `/platform/group`, without the current detour through `/groups/pick`. The button reuses the existing `GroupPickerController.SelectGroup` POST action verbatim (D-01) — no new controller, service, or entity code — and is added to both the desktop (`Index.cshtml`) and mobile (`Index.Mobile.cshtml`) views (D-04), positioned to the left of the existing "Members" and "Edit" actions.

Purpose: Removes a friction point in the SuperAdmin group-management workflow. Setting a group active and viewing its board currently requires navigating out to the group picker; this collapses that to a single click from the management table.
Output: Two modified Razor views. No new symbols.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser → GroupPickerController.SelectGroup (POST) | Authenticated SuperAdmin submits a group-selection form; `groupId` is attacker-controllable input crossing into the app. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-48-01 | Spoofing / Elevation | The new POST form → `SelectGroup` | mitigate | Form is emitted only inside `Areas/Platform/Views/Group/Index*.cshtml`, which renders under a `[Authorize(Policy = "SuperAdminOnly")]`-gated Platform GroupController. `SelectGroup` itself is `[Authorize]`-gated. No new anonymous surface is introduced — this phase adds a caller, not a new endpoint. |
| T-48-02 | Tampering (CSRF) | The new POST form → `SelectGroup` | mitigate | Form includes `@Html.AntiForgeryToken()`; `SelectGroup` carries `[ValidateAntiForgeryToken]`. Cross-site forged POSTs are rejected, identical to the existing GroupPicker form. |
| T-48-03 | Tampering / IDOR | `groupId` hidden input → `SelectGroup` | accept (pre-existing, out of scope) | `SelectGroup` looks up the group by id and sets it active WITHOUT verifying the caller is a member of that group — it returns `NotFound()` only if the group does not exist. This is a PRE-EXISTING property of `SelectGroup` (used by `/groups/pick` today), NOT introduced by this phase. In this phase's context the only caller is a SuperAdmin (who legitimately may open any group's board), so no new risk arises. Fixing the general per-group membership check on `SelectGroup` is explicitly out of scope for this markup-only phase. |
| T-48-04 | Information Disclosure | Redirect to a group's quest board | accept | The redirect target is the standard Quest Index for the selected group; a SuperAdmin is already authorized to view any group. No new data is exposed beyond what the existing group picker already permits. |

**Note on `SelectGroup`'s missing per-group membership check:** confirmed as a pre-existing condition unrelated to this phase. This phase adds a new *caller* of an already-authorized action from within a SuperAdmin-only view; it does not change what `SelectGroup` authorizes, does not widen its access, and does not introduce a new endpoint. No package installs occur (no supply-chain threat), so no `T-48-SC` row is required.
</threat_model>
