<objective>
Build the shared, collision-aware user-creation method `IUserService.CreateOrAddToGroupAsync` in the Domain layer, plus a `CreateOrAddToGroupResult` return type that tells the caller which of four outcomes occurred (new account, added-to-group, added-to-group-stranded-account, already-member). This is the single method both today's group-admin Create User form (Plan 03) and Phase 40's future platform entry point will call — consolidating logic that currently only exists inline in `AdminController.CreateUser`.

The method performs collision detection and membership mutation only. It does NOT build callback URLs or enqueue email jobs (those are MVC-only concerns owned by the controller in Plan 03) — it returns enough information for the controller to send the right email.

Purpose: Establish the shared collision behavior once, in the layer both current and future callers can reach, so Phase 40's new entry point cannot diverge from this one (the recurring duplicated-logic-diverges bug class flagged in STATE.md).
Output: `CreateOrAddToGroupResult` model, `CreateOrAddToGroupAsync` interface member + implementation, and unit tests for all four outcomes.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| group-admin browser → AdminController (Plan 03) → UserService.CreateOrAddToGroupAsync | An authenticated Admin submits an arbitrary email that may collide with any existing platform user's account, across group boundaries. The Domain method mutates group membership. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-39-01 | Elevation of Privilege | `CreateOrAddToGroupAsync` collision-add path (silent privilege grant) | accept (by milestone spec) + mitigate (bounded scope) | The method grants ONLY the group membership + role explicitly chosen by an already-authenticated Admin acting within their own group scope (`groupId` supplied by the caller). It never touches ASP.NET Core Identity roles, never sets a password, never confirms email, and never modifies the existing account's Name/Email (D-07). No cross-group escalation is possible because the caller owns `groupId`. The "silent add, notify after" behavior is an explicit REQUIREMENTS.md out-of-scope decision (no consent interstitial); the notification email (Plan 02/03) is the after-the-fact control. Verified by unit tests asserting no account/role mutation on the collision branch. |
| T-39-02 | Spoofing / Account Takeover | Existing-user collision branch | mitigate | The method resolves the existing account strictly by `GetIdByEmailAsync(email)` and returns the EXISTING account's own `UserId`/`Email`/`Name`; it never re-creates or overwrites the account, so a colliding submission cannot hijack or duplicate an account. The `AlreadyMember` branch creates no second membership row (unique composite index on `UserId,GroupId` is the backstop). CREATE-03 already-member and D-01 stranded-account paths are both covered by passing unit tests. |
| T-39-03 | Tampering | Membership row creation | mitigate | Membership is written only via `IGroupRepository.AddMemberAsync`, which does an existence-check-then-throw ahead of the DB unique index — no upsert, no silent overwrite of an existing role. |
</threat_model>
