<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| server → Resend email relay | The AddedToGroup email is sent to the colliding user's registered address via the existing email service. Content is server-composed. |
| TempData["Warning"] → rendered HTML in Users.cshtml | Flash message text is composed server-side by the controller (Plan 03) and rendered in the Admin view. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-39-04 | Information Disclosure | AddedToGroup email content | mitigate | The email carries no set-password token, no callback URL, and no credential — its CTA is a plain `/Account/Login` link (D-05), so an intercepted or forwarded email cannot be used to seize the account. It names the group and role only; no PII beyond the recipient's own membership. |
| T-39-05 | Cross-Site Scripting (Tampering) | `@TempData["Warning"]` rendering in Users.cshtml | accept | Razor HTML-encodes `@TempData["Warning"]` by default, matching the existing Success/Error blocks. The message text is server-composed from a controller-supplied user Name (Plan 03), not raw request input, so the existing encoding is sufficient — no new sink introduced. |
| T-39-06 | Spoofing | Log In CTA link | mitigate | The CTA points to the app's own `/Account/Login` route (built by the controller in Plan 03 via `Url.Action`), never an externally supplied URL, so the email cannot be weaponized to redirect users to a phishing login page. |
</threat_model>
