<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| group-admin browser â†’ AdminController.CreateUser (POST) | An authenticated Admin submits an email that may collide with any existing platform user. The action triggers a group-membership grant and an outbound email. |
| AdminController â†’ Hangfire job â†’ Resend email relay | The controller enqueues either a WelcomeEmailJob (with a live SetPassword token) or a GroupMembershipAddedEmailJob (token-free) depending on account state. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-39-07 | Elevation of Privilege | Silent collision auto-add (no consent step) | accept (milestone spec) + mitigate (scope + notify + audit) | The action is behind `[Authorize(Policy = "AdminOnly")]` and `[ValidateAntiForgeryToken]`; the grant is bounded to the Admin's own active group (`groupId` from `activeGroupContext`, never cross-group) and to a `GroupRole` only â€” no Identity role, no password, no email-confirm. The affected user is notified after the fact (AddedToGroup email), which is the milestone's chosen control in place of a consent interstitial (REQUIREMENTS.md out-of-scope). The blocking human-verify checkpoint explicitly validates this path. |
| T-39-08 | Information Disclosure | Stranded-account resend issues a live SetPassword token | mitigate | The fresh SetPassword token is generated ONLY when the colliding account is unconfirmed (`EmailConfirmed == false`, i.e. it never had a usable credential), and is delivered solely to that account's own registered email â€” identical to the existing `SendConfirmationEmail` resend path. A confirmed account NEVER receives a token via this flow (it gets the token-free AddedToGroup email instead), so a collision submission cannot mint a password-reset link for an established account. |
| T-39-09 | Repudiation | Silent membership change | accept | Membership grants flow through the existing repository/EF write path; no dedicated audit log exists in this app today and adding one is out of scope for this bugfix milestone. The after-the-fact email provides the affected user a record that the change occurred. |
| T-39-SC | Tampering | npm/pip/cargo installs | n/a | This phase installs no packages â€” no new dependencies are added (all reused symbols already exist in the solution). No package-legitimacy checkpoint required. |
