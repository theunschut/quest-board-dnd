# Phase 34.3 Plan 02: QuestController Inline Role-Check Migration Summary

**Migrated all 12 inline `IsInRole("Admin")`/`IsInRoleAsync(User, "Admin"|"DungeonMaster")` checks in `QuestController` to the shared `GetEffectiveGroupRoleAsync` helper, fixing the Manage Quest card disappearing and 8 Forbid() sites incorrectly blocking Admins who aren't the quest's own DM.**

## Accomplishments
- Index role filtering, Details `ViewBag.CanManage`, and Manage `ViewBag.IsAdmin`/`ViewBag.IsAuthorized` now derive from `GetEffectiveGroupRoleAsync` (GroupRole via UserGroups) instead of the empty `AspNetUserRoles`
- All 8 ownership-OR-admin `Forbid()` guards (Edit GET/POST, Delete, Finalize, Open, SendReminder, CreateFollowUp GET/POST) fixed — a real Admin who is not the quest's DM is no longer incorrectly forbidden
- Ownership comparisons (`currentUser.Equals(quest.DungeonMaster)` / `isQuestDm`) preserved inline and unchanged at every site — only the admin-role term was migrated
- Anonymous-reachable sites (Index, Details) verified to keep `RequireActiveGroupId()` strictly inside their existing null-guards, so anonymous visitors never trigger a 500
- Zero `IsInRole("Admin")`/`IsInRoleAsync(User, "Admin"|"DungeonMaster")` occurrences remain in `QuestController.cs`

## Decisions Made
- Index's two separate `IsInRoleAsync` calls (Admin, DungeonMaster) were consolidated into one `GetEffectiveGroupRoleAsync` call per the plan's explicit instruction, then mapped to `Role.Admin`/`Role.DungeonMaster` — this is why the file shows 11 `GetEffectiveGroupRoleAsync` call sites covering all 12 original inline checks, not a 1:1 count.
- `RequireActiveGroupId()` is called once per guarded action (hoisted immediately above each `Forbid()` check for the 8 ownership sites) rather than reused across multiple checks, keeping each site's guard self-contained and matching the plan's per-site action text.
