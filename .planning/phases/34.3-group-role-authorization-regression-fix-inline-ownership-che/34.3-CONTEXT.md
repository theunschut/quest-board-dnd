# Phase 34.3: Group Role Authorization Regression Fix - Context

**Gathered:** 2026-07-02
**Status:** Ready for planning

<domain>
## Phase Boundary

Every inline ownership/role check across `QuestController`, `QuestLogController`, `DungeonMasterController`, and `Admin/AccountController` that still calls `User.IsInRole("Admin")` / `User.IsInRole("DungeonMaster")` / `userService.IsInRoleAsync(user, "Admin"|"DungeonMaster")` is migrated to the `GroupRole`-aware pattern already proven in `AdminHandler`/`DungeonMasterHandler`:

```csharp
if (context.User.IsInRole("SuperAdmin")) { /* bypass */ }
var groupId = activeGroupContext.RequireActiveGroupId();
var role = await userService.GetGroupRoleAsync(context.User, groupId);
// role == GroupRole.Admin / GroupRole.DungeonMaster
```

Phase 27's `AddGroupSchema` migration deleted the Player/DungeonMaster/Admin rows from `AspNetUserRoles` (moved to `UserGroups.GroupRole`), but only the policy-based handlers were updated in Phase 29 — roughly 23 inline call sites were missed and now silently always evaluate `isAdmin`/`isDungeonMaster` as `false` for real Admins/DMs.

**Confirmed call sites (23 total, via RIP FindCallers + grep, cross-verified):**
- `QuestController.cs` — 12 sites: lines 42-43 (Index role filter), 127 (Edit), 179 (Edit POST), 237 (Delete), 282 (Manage `CanManage`), 611 (Finalize), 648 (Open), 682 (SendReminder), 739 (Manage `IsAdmin`), 760 (CreateFollowUp GET), 815 (CreateFollowUp POST)
- `Admin/AccountController.cs` — 4 sites: lines 152-153 (Profile), 171-172 (Edit)
- `DungeonMaster/DungeonMasterController.cs` — 3 sites: lines 32 (Profile `CanEdit`), 52 (Edit GET), 82 (Edit POST)
- `QuestBoard/QuestLogController.cs` — 2 sites: lines 56 (Details `CanEditRecap`), 93 (UpdateRecap)
- `Program.cs:302` and `Authorization/AdminDashboardAuthFilter.cs:14` — 2 sites gating `/hangfire`, same broken pattern, not named in the roadmap's file list but discovered during codebase scout (see Decisions).

**Wiring note:** `QuestController` already has `IActiveGroupContext` injected. `QuestLogController`, `DungeonMasterController`, and `AccountController` do not — the fix must add that constructor dependency to those three.

<decisions>
## Implementation Decisions

### Hangfire dashboard scope
- **D-01:** `Program.cs:302` and `AdminDashboardAuthFilter.cs:14` gate `/hangfire` with `!IsInRole("Admin") && !IsInRole("SuperAdmin")` — same root-cause bug, but not named in the roadmap's scoped file list. Fix in scope for this phase: **remove the dead `Admin` check entirely**, simplify both to SuperAdmin-only. Rationale: Phase 29's decision was explicit that Hangfire is SuperAdmin-only and group Admins should NOT access it (PROJECT.md Key Decisions). Migrating this Admin check to be GroupRole-aware (as if it were a QuestController-style fix) would actually grant group Admins new access to background job infrastructure — reversing that decision. Removing the dead clause matches both the current effective behavior (Admin has been dead since Phase 27) and the documented intent exactly.

### Duplication vs. shared helper
- **D-02:** Introduce **one small reusable helper** (exact placement/naming left to planning) that all ~23 call sites use for the SuperAdmin-bypass + active-group + `GetGroupRoleAsync` check, rather than duplicating the pattern inline at every site. Rationale: several sites (`QuestController.Index`, `AccountController.Profile`/`Edit`) currently call `IsInRoleAsync` twice — once for Admin, once for DungeonMaster. A shared helper lets these derive both booleans from a single `GetGroupRoleAsync` lookup instead of two separate role checks, and keeps the 23 sites consistent. Matches the Phase 34.2 `ControllerExtensions` precedent for shared MVC-boilerplate helpers.

### Null-group failure behavior
- **D-03:** `QuestController`, `QuestLogController`, and `DungeonMasterController` use `activeGroupContext.RequireActiveGroupId()` (fail-hard, throws `InvalidOperationException` on null) — safe because `GroupSessionMiddleware` (Phase 31) redirects any authenticated non-SuperAdmin request with a null `ActiveGroupId` to `/groups/pick` before it reaches these controllers.
- **D-04:** `AccountController.Profile`/`Edit` treat a null `ActiveGroupId` as **"no role, show no badges"** (fail-soft) instead of throwing. Rationale: `GroupSessionMiddleware` exempts the entire `/Account` path prefix (to avoid a login redirect loop with `Login`/`ForgotPassword`), so these two actions are genuinely reachable by an authenticated user whose session-scoped `ActiveGroupId` has expired or isn't set yet — this is the exact session-recovery scenario Phase 31 was built for, not a SuperAdmin-only edge case. `IsAdmin`/`IsDungeonMaster` stay tied to `ActiveGroupId` (not decoupled into a group-independent concept) because `GroupRole` is inherently per-group in this schema — every other place that shows this (Manage Quest card, QuestLog recap-edit link, `DungeonMasterController.CanEdit`) ties it to the active group, and decoupling the profile badges would mean answering a different, broader question ("Admin of *any* group?") that's out of scope for this regression fix. User confirmed this is an acceptable edge case.

### Regression test coverage
- **D-05:** Add integration tests mirroring the existing `AdminHandlerIntegrationTests` pattern (`QuestBoard.IntegrationTests/Controllers/AdminHandlerIntegrationTests.cs`) for the newly-fixed inline sites. Rationale: this exact class of bug (broken inline auth checks) went undetected until manual pre-ship testing precisely because no automated test covered it — locking in the fix with tests prevents recurrence on the next `AspNetUserRoles`-adjacent change.

### Claude's Discretion
- Exact name/location/signature of the shared helper from D-02 (e.g., extension method vs. `IUserService` method) — left to planning.
- Test file organization/naming for D-05 (one file per controller vs. one combined file) — left to planning.

</decisions>
