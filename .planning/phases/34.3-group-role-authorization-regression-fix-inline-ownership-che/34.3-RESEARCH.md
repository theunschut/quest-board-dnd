# Phase 34.3: Group Role Authorization Regression Fix - Research

**Researched:** 2026-07-02
**Domain:** ASP.NET Core MVC authorization migration (inline role checks -> GroupRole-aware helper)
**Confidence:** HIGH

## Summary

This phase migrates ~24 inline `User.IsInRole("Admin"|"DungeonMaster")` / `userService.IsInRoleAsync(...)` call sites that were never updated when Phase 27 moved per-group roles out of `AspNetUserRoles` into `UserGroups.GroupRole`. Every one of these sites currently evaluates `isAdmin`/`isDungeonMaster` as permanently `false` for real Admins/DMs, because `AspNetUserRoles` no longer contains Player/DungeonMaster/Admin rows (only SuperAdmin). The fix pattern already exists and is proven in production: `AdminHandler`/`DungeonMasterHandler` (SuperAdmin bypass -> active group -> `GetGroupRoleAsync` comparison).

All code, tests, and DI infrastructure needed to execute this fix already exist in the codebase ‚Äî this is a mechanical, low-risk migration, not new architecture. `IActiveGroupContext` is registered once in `Program.cs` via standard constructor injection (no per-controller DI registration needed). `ActiveGroupContextExtensions.RequireActiveGroupId()` (Phase 34.2) is an unused seam waiting for its first caller. The integration test factory's `MutableGroupContext` already defaults to `ActiveGroupId = 1`, so `RequireActiveGroupId()` succeeds out-of-the-box in tests with zero extra setup.

**One material finding beyond CONTEXT.md's 23-site inventory:** a 24th site exists in `QuestBoard.Repository/IdentityService.cs:113` (`AdminResetPasswordAsync`), calling `userManager.IsInRoleAsync(adminEntity, "Admin")` directly via ASP.NET Identity's `UserManager`. It is architecturally different from the other 23 (Repository layer, not Service layer; redundant defense-in-depth behind an already-correct `[Authorize(Policy = "AdminOnly")]` gate, not the primary access gate) and needs a distinct decision in planning ‚Äî see Open Questions.

**Primary recommendation:** Add one shared extension-method helper (e.g., `GetGroupRoleFlagsAsync` on `IUserService`, or an extension method near `ActiveGroupContextExtensions`) that returns `(bool IsAdmin, bool IsDungeonMaster)` from a single `GetGroupRoleAsync` call with the SuperAdmin bypass baked in, then replace each of the 23 named controller-layer call sites with a call to that helper. Handle the Hangfire sites (2) by deleting the dead `Admin` check per D-01. Handle the 24th `IdentityService` site as a separate, smaller decision (see Open Questions) since it doesn't fit the same DI shape.

## User Constraints (from CONTEXT.md)

### Locked Decisions

**Hangfire dashboard scope (D-01):** `Program.cs:302` and `AdminDashboardAuthFilter.cs:14` gate `/hangfire` with `!IsInRole("Admin") && !IsInRole("SuperAdmin")` ‚Äî same root-cause bug, but not named in the roadmap's scoped file list. Fix in scope for this phase: **remove the dead `Admin` check entirely**, simplify both to SuperAdmin-only. Rationale: Phase 29's decision was explicit that Hangfire is SuperAdmin-only and group Admins should NOT access it. Migrating this Admin check to be GroupRole-aware would actually grant group Admins new access to background job infrastructure ‚Äî reversing that decision. Removing the dead clause matches both current effective behavior and documented intent.

**Duplication vs. shared helper (D-02):** Introduce **one small reusable helper** (exact placement/naming left to planning) that all ~23 call sites use for the SuperAdmin-bypass + active-group + `GetGroupRoleAsync` check, rather than duplicating the pattern inline at every site. Several sites (`QuestController.Index`, `AccountController.Profile`/`Edit`) currently call `IsInRoleAsync` twice ‚Äî once for Admin, once for DungeonMaster. A shared helper lets these derive both booleans from a single `GetGroupRoleAsync` lookup instead of two separate role checks, and keeps the 23 sites consistent. Matches the Phase 34.2 `ControllerExtensions` precedent for shared MVC-boilerplate helpers.

**Null-group failure behavior (D-03/D-04):**
- D-03: `QuestController`, `QuestLogController`, and `DungeonMasterController` use `activeGroupContext.RequireActiveGroupId()` (fail-hard, throws `InvalidOperationException` on null) ‚Äî safe because `GroupSessionMiddleware` (Phase 31) redirects any authenticated non-SuperAdmin request with a null `ActiveGroupId` to `/groups/pick` before it reaches these controllers.
- D-04: `AccountController.Profile`/`Edit` treat a null `ActiveGroupId` as **"no role, show no badges"** (fail-soft) instead of throwing. Rationale: `GroupSessionMiddleware` exempts the entire `/Account` path prefix (to avoid a login redirect loop with `Login`/`ForgotPassword`), so these two actions are genuinely reachable by an authenticated user whose session-scoped `ActiveGroupId` has expired or isn't set yet. `IsAdmin`/`IsDungeonMaster` stay tied to `ActiveGroupId` (not decoupled into a group-independent concept) because `GroupRole` is inherently per-group in this schema.

**Regression test coverage (D-05):** Add integration tests mirroring the existing `AdminHandlerIntegrationTests` pattern (`QuestBoard.IntegrationTests/Controllers/AdminHandlerIntegrationTests.cs`) for the newly-fixed inline sites. This exact class of bug went undetected until manual pre-ship testing precisely because no automated test covered it.

## Canonical Pattern to Replicate

```csharp
// Source: QuestBoard.Service/Authorization/AdminHandler.cs (verified current, 2026-07-02)
// Step 1: SuperAdmin bypass ‚Äî reads claims directly, no DB call
if (context.User.IsInRole("SuperAdmin"))
{
    context.Succeed(requirement);
    return;
}

// Step 2: Null group guard
if (activeGroupContext.ActiveGroupId is not { } groupId)
{
    context.Fail();
    return;
}

// Step 3: Group role check
var role = await userService.GetGroupRoleAsync(context.User, groupId);
if (role == GroupRole.Admin)
    context.Succeed(requirement);
else
    context.Fail();
```

```csharp
// Source: QuestBoard.Service/Authorization/DungeonMasterHandler.cs (verified current, 2026-07-02)
// Same shape; DungeonMasterOnly is satisfied by EITHER DungeonMaster or Admin GroupRole
var role = await userService.GetGroupRoleAsync(context.User, groupId);
if (role == GroupRole.Admin || role == GroupRole.DungeonMaster)
    context.Succeed(requirement);
else
    context.Fail();
```

Note: `AdminHandler`/`DungeonMasterHandler` use the graceful `ActiveGroupId is not { } groupId` null-guard (returns `context.Fail()`), NOT `RequireActiveGroupId()`. Per D-03/D-04, the 23 inline controller sites use `RequireActiveGroupId()` instead (fail-hard via exception) because they are controller actions guarded by `GroupSessionMiddleware`, not `IAuthorizationHandler`s that run inside the ASP.NET Core authorization pipeline before that middleware necessarily applies. **Exception:** `AccountController` sites use fail-soft (D-04) because `/Account` is `GroupSessionMiddleware`-exempt.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| SuperAdmin bypass logic | A second, slightly-different bypass condition per call site | Reuse the exact `context.User.IsInRole("SuperAdmin")` check, copied verbatim from `AdminHandler`/`DungeonMasterHandler` | Any variation (e.g., checking a different claim, or checking case-sensitively differently) risks a second silent regression identical in kind to the one this phase fixes |
| Group role lookup | A new repository query or a cached/denormalized role flag | `userService.GetGroupRoleAsync(ClaimsPrincipal, int)` ‚Äî already exists, already used by both handlers | This method is the single source of truth for "what GroupRole does this user hold in this group" ‚Äî a second implementation would immediately diverge in edge-case handling (e.g., not-a-member returns null) |
| Null active-group handling | Ad-hoc `?? 1` fallbacks (the exact anti-pattern STATE.md documents Phase 29/30 removed) | `RequireActiveGroupId()` (fail-hard, controller actions) or the graceful `ActiveGroupId is not { }` pattern (fail-soft, AccountController) | `?? 1` fallbacks were deliberately removed in Phase 30 because they silently leak GroupId=1 data across tenants in a multi-group deployment; reintroducing them here would be a second, worse regression |

**Key insight:** every piece this phase needs already exists and is proven correct in production via the two `IAuthorizationHandler`s. The only genuinely new code is the shared helper (D-02) that adapts this proven pattern to controller-action call sites instead of `IAuthorizationHandler.HandleRequirementAsync`.

## Shared Helper Design (D-02) ‚Äî Options Analysis

CONTEXT.md leaves exact placement/signature to planning. Research surfaced these constraints from actual call-site shapes above:

**Constraint 1 ‚Äî mixed input types.** Most sites have `ClaimsPrincipal User` available (`ControllerBase.User`). `AccountController.Profile`/`Edit` instead call `userService.IsInRoleAsync(user, "Admin")` where `user` is the **domain `User` model** already fetched via `GetUserAsync(User)` ‚Äî not the `ClaimsPrincipal`. A single helper signature must accept whichever is idiomatic per site, or the helper should take `ClaimsPrincipal` uniformly (since `IUserService.GetGroupRoleAsync(ClaimsPrincipal, int)` already exists) and `AccountController` should be updated to pass `User` (the `ClaimsPrincipal` property inherited from `Controller`) instead of the already-fetched domain `User` model ‚Äî this is a minor behavior-neutral simplification, not a functional change, since both resolve to the same underlying identity.

**Constraint 2 ‚Äî two failure modes needed (D-03 fail-hard vs D-04 fail-soft).** The helper cannot unilaterally decide fail-hard vs fail-soft ‚Äî either:
- (a) the helper takes the *already-resolved* `int? groupId` (caller decides `RequireActiveGroupId()` vs `ActiveGroupContext.ActiveGroupId` before calling), or
- (b) two helper overloads/variants exist, or
- (c) the helper takes `IActiveGroupContext` + a `bool requireGroup` flag.

Option (a) is simplest and keeps the null-handling decision visible at each call site (matching D-03/D-04's per-controller distinction), at the cost of one extra line per call site (`var groupId = activeGroupContext.RequireActiveGroupId();` or `activeGroupContext.ActiveGroupId is not { } groupId ? ... : ...` before the helper call).

**Constraint 3 ‚Äî return shape.** Several sites need both `isAdmin` and `isDungeonMaster` (or a derived `Role`/`GroupRole` enum) from one lookup (D-02's explicit rationale). A tuple `(bool IsAdmin, bool IsDungeonMaster)` or the raw `GroupRole?` (letting the caller compare `== GroupRole.Admin` etc.) both satisfy this ‚Äî returning the raw `GroupRole?` is more flexible (also supports `QuestController.Index`'s three-way `Role` mapping) and requires less bespoke tuple-unpacking code at each site.

**Recommended shape** (synthesizing the above, final naming left to planner):

```csharp
// Candidate placement: QuestBoard.Domain/Interfaces/IUserService.cs (new method)
// or QuestBoard.Service/Extensions/ (new static extension class, mirroring ControllerExtensions.cs)
Task<GroupRole?> GetEffectiveGroupRoleAsync(ClaimsPrincipal user, int groupId);
// Internally: if (user.IsInRole("SuperAdmin")) return GroupRole.Admin; // SuperAdmin bypass
//             return await GetGroupRoleAsync(user, groupId);
```

Callers then do `var role = await userService.GetEffectiveGroupRoleAsync(User, activeGroupContext.RequireActiveGroupId());` and compare `role == GroupRole.Admin` / `role == GroupRole.Admin || role == GroupRole.DungeonMaster` inline ‚Äî this collapses the SuperAdmin-bypass boilerplate into the helper while leaving the null-group-handling decision (D-03 vs D-04) at the call site, satisfying all three constraints above. This mirrors the existing `IUserService.GetGroupRoleAsync`/`GetGroupRoleByIdAsync` naming convention (`Task<GroupRole?> Get...Async`) rather than introducing a boolean-tuple return shape that doesn't exist elsewhere in the codebase.

**Alternative considered:** placing this as a `ClaimsPrincipal`/`IActiveGroupContext` extension method (mirroring `ActiveGroupContextExtensions.RequireActiveGroupId()` and the Phase 34.2 `ControllerExtensions` precedent) instead of an `IUserService` interface method. Tradeoff: an extension method can't easily call `userService.GetGroupRoleAsync` without taking `IUserService` as a parameter (`this ClaimsPrincipal user, IUserService userService, int groupId`), which is a less idiomatic C# extension-method shape than a plain interface method. An `IUserService` interface method is the more consistent choice given `GetGroupRoleAsync`/`GetGroupRoleByIdAsync` already live there.

## Common Pitfalls

### Pitfall 1: Collapsing ownership-OR-admin checks into the helper
**What goes wrong:** ~14 of the 23 sites are `!currentUser.Equals(X) && !User.IsInRole("Admin")` ‚Äî an ownership check ANDed with a negated admin check. If the shared helper is designed to also take "is this the owner" as a parameter and return a single combined bool, it silently couples two orthogonal concerns and makes the helper untestable in isolation.
**Why it happens:** Tempting to "finish the refactor" by also extracting the ownership-OR-admin pattern itself, since it repeats 14 times.
**How to avoid:** The helper (per D-02) covers ONLY the SuperAdmin-bypass + GroupRole lookup. Ownership comparison (`currentUser.Equals(quest.DungeonMaster)`, `currentUser.Equals(targetUser)`) stays inline at each call site exactly as today ‚Äî only the trailing `!User.IsInRole("Admin")` is replaced with `role != GroupRole.Admin` (or equivalent) sourced from the helper.
**Warning signs:** A helper signature that takes a `User owner`/`User currentUser` parameter is a sign scope crept beyond D-02.

### Pitfall 2: Calling the helper for anonymous/unauthenticated users
**What goes wrong:** `QuestController.Details`, `QuestController.Index`, `DungeonMasterController.Profile`, and `QuestLogController.Details` are all reachable without `[Authorize]`. All four currently guard the role-check with `currentUser != null` (or an outer `if (userEntity != null)` block). If the migrated code calls `activeGroupContext.RequireActiveGroupId()` or the shared helper unconditionally (outside that guard), an anonymous visitor hits an `InvalidOperationException` (500 error) instead of simply seeing `CanManage = false`.
**Why it happens:** Easy to reflexively hoist the new helper call above the existing null-check while refactoring, especially since `RequireActiveGroupId()` "feels like" it should always be safe to call.
**How to avoid:** Preserve the exact existing `currentUser != null && ...` / `if (userEntity != null) { ... }` short-circuit structure at each of these 4 unauthenticated-reachable sites ‚Äî only swap what's inside the guard.
**Warning signs:** Any call to the new helper (or to `RequireActiveGroupId()`) that is NOT nested inside an existing `if (currentUser != null)` / `if (user != null)` check on an action without `[Authorize]`.

### Pitfall 3: `RequireActiveGroupId()` throwing inside `AccountController` (violates D-04)
**What goes wrong:** `AccountController.Profile`/`Edit` are reachable with a null `ActiveGroupId` specifically because `/Account` is `GroupSessionMiddleware`-exempt (verified: `GroupSessionMiddleware.cs:45`, `ExemptPathPrefixes` includes `ControllerNameOf<AccountController>()`). If these two actions use `RequireActiveGroupId()` (matching the other 3 controllers) instead of the graceful null-check, a session-expired authenticated user gets a 500 error on their own profile page instead of seeing no role badges.
**Why it happens:** Copy-pasting the same fix pattern across all 4 controllers for consistency, missing that `AccountController` is the one controller NOT protected by `GroupSessionMiddleware`.
**How to avoid:** `AccountController` must use `if (activeGroupContext.ActiveGroupId is not { } groupId) { /* no role, no badges */ }` (or equivalent), never `RequireActiveGroupId()`.
**Warning signs:** `RequireActiveGroupId()` appearing anywhere in `AccountController.cs`.

### Pitfall 4: Missing test seed causes false negatives in D-05 integration tests
**What goes wrong:** `AuthenticationHelper.CreateAuthenticatedClientWithUserAsync` already seeds a `UserGroups` row for GroupId=1 based on the `roles` array passed in (verified: `AuthenticationHelper.cs:118-138`), and `WebApplicationFactoryBase` registers `MutableGroupContext` (default `ActiveGroupId = 1`) as the test `IActiveGroupContext` (verified: `MutableGroupContext.cs:12`, `WebApplicationFactoryBase.cs:69`). New D-05 tests do NOT need to manually seed `UserGroups` or set `ActiveGroupId` ‚Äî calling `AuthenticationHelper.CreateAuthenticatedClientWithUserAsync(_factory, ..., roles: ["Admin"])` already does both. Writing tests that manually construct `UserGroupEntity` rows (duplicating what the helper already does) risks a GroupId mismatch (e.g., seeding GroupId=2 while `MutableGroupContext` defaults to 1) that produces a confusing false-negative test failure.
**Why it happens:** Not realizing `CreateAuthenticatedClientWithUserAsync` already does GroupRole seeding as of Phase 29 (see `STATE.md`: "Phase 29 Plan 01: AuthenticationHelper must seed UserGroups rows for DM/Admin test users").
**How to avoid:** Use the existing `AuthenticationHelper.CreateAuthenticatedClientWithUserAsync(..., roles: [...])` / `CreateAuthenticatedAdminClientAsync` / `CreateAuthenticatedDMClientAsync` / `CreateAuthenticatedSuperAdminClientAsync` helpers exactly as `AdminHandlerIntegrationTests.cs` does ‚Äî do not hand-roll group seeding.
**Warning signs:** Manual `context.UserGroups.Add(...)` calls in new test files.

### Pitfall 5: `Manage`'s `ViewBag.IsAdmin` semantic differs subtly from `IsAuthorized`
**What goes wrong:** `QuestController.Manage` (line 739-741) computes `isAdmin` once, then uses it TWO ways: folded into `ViewBag.IsAuthorized = isQuestDm || isAdmin` (an access gate) AND exposed standalone as `ViewBag.IsAdmin` (likely used by the view to conditionally render admin-only UI elements within the management page, distinct from the page-access gate). If the migration accidentally only fixes `IsAuthorized` and leaves a stale/differently-scoped `IsAdmin`, the page becomes accessible but admin-only buttons inside it silently stay hidden ‚Äî a subtler, harder-to-notice version of the same regression.
**Why it happens:** Only checking whether the page loads (IsAuthorized) during manual testing, not scrutinizing every ViewBag flag consumed by the Manage view.
**How to avoid:** Confirm `Views/QuestBoard/Manage.cshtml` (and its Mobile variant, if any) consumes `ViewBag.IsAdmin` and verify it renders correctly for a real Admin post-fix, not just that the page loads.

## Open Questions

1. **Should the 24th `IdentityService.AdminResetPasswordAsync` site be fixed in this phase, and if so, how?**
   - What we know: It's the same broken-since-Phase-27 pattern (`userManager.IsInRoleAsync(entity, "Admin")` after `AspNetUserRoles` no longer holds Admin rows for non-SuperAdmins). It currently makes `AdminController.ResetPassword` (POST) always fail with "Admin user not found or not authorized" for any real Admin ‚Äî even though the outer `[Authorize(Policy = "AdminOnly")]` on `AdminController` already correctly lets them reach the action. It is NOT named in CONTEXT.md's 23-site inventory or D-01 through D-05.
   - What's unclear: Whether this was deliberately excluded from the phase's scope (since it's Repository-layer, not one of the 4 named controllers) or simply missed by the original audit. It doesn't fit the `IActiveGroupContext`-based fix pattern cleanly since `IdentityService` sits in the Repository layer.
   - Recommendation: Two clean options for planning to choose between: (a) **simplest fix, in scope**: since the outer `AdminController` policy already correctly restricts callers to Admin/SuperAdmin, this inner check is now provably redundant ‚Äî just delete the `!await userManager.IsInRoleAsync(adminEntity, "Admin")` clause from `AdminResetPasswordAsync` entirely (keep only the `adminEntity == null` null-check), removing dead/misleading code without needing any `IActiveGroupContext` plumbing in the Repository layer. This mirrors the D-01 Hangfire precedent (delete dead Admin check rather than migrate it) and is the lower-risk option. (b) **defer to a follow-up phase**: explicitly flag it as an acknowledged gap for now, since it is a redundant secondary gate (the primary gate already works correctly) rather than a full access-control bypass. Recommend option (a) ‚Äî it is small, low-risk, consistent with D-01's established precedent in this same phase, and actually fixes a real (if masked) bug for the cost of deleting one boolean clause.

2. **View-layer verification for `ViewData["IsAdmin"]`/`ViewData["IsDungeonMaster"]` and `ViewBag.IsAdmin` consumers.**
   - What we know: `Views/Account/Profile.cshtml` and `Profile.Mobile.cshtml` consume `ViewData["IsAdmin"]`/`ViewData["IsDungeonMaster"]` (confirmed via grep). `Views/QuestBoard/Manage.cshtml` presumably consumes `ViewBag.IsAdmin` (not read in this research pass ‚Äî file not opened).
   - What's unclear: Exact Razor markup consuming these flags, to confirm no additional silent-fallback logic exists at the view layer that would mask or duplicate the controller-layer fix.
