## Duplication vs. shared helper

| Option | Description | Selected |
|--------|-------------|----------|
| Small reusable helper | One helper every site calls once, deriving isAdmin/isDungeonMaster locally from a single GetGroupRoleAsync lookup | ✓ |
| Inline at every site | Duplicate the check literally at each of the 23 sites, mirroring AdminHandler's exact style | |

**User's choice:** Small reusable helper (Recommended).
**Notes:** Several sites (`QuestController.Index`, `AccountController.Profile`/`Edit`) currently call `IsInRoleAsync` twice (once for Admin, once for DungeonMaster) — a shared helper lets these derive both booleans from one `GetGroupRoleAsync` lookup instead of two role checks. Matches the Phase 34.2 `ControllerExtensions` precedent.

## Null-group failure behavior

| Option | Description | Selected |
|--------|-------------|----------|
| Graceful fallback in AccountController only | AccountController.Profile/Edit treat null ActiveGroupId as "no role" instead of throwing; other 3 controllers use RequireActiveGroupId() | ✓ |
| RequireActiveGroupId() everywhere, no exceptions | All ~23 sites throw on null, including AccountController | |

**User's choice:** Graceful fallback in AccountController only (Recommended), after discussion.
**Notes:** User initially pushed back, believing only SuperAdmin could have a null ActiveGroupId. Clarified the distinction between DB group membership (guaranteed for all migrated users) and session-scoped `ActiveGroupId` (can be null/expired independent of DB state — this is the exact session-recovery scenario Phase 31 was built for). Also confirmed `GroupSessionMiddleware` exempts the entire `/Account` path prefix to avoid a login redirect loop, so `AccountController.Profile`/`Edit` are genuinely reachable in this state, unlike the other three controllers.

User then asked whether the IsAdmin/IsDungeonMaster badges should be decoupled from ActiveGroupId entirely (profile independent of active group). Advised against it: `GroupRole` is inherently per-group in this schema (a user could be Admin in one group, Player in another), and every other role-gated feature in the app (Manage Quest card, QuestLog recap-edit, DungeonMasterController.CanEdit) ties to the active group — decoupling the profile badges would require answering a different, broader question ("Admin of any group?") that's new scope beyond this regression fix. User agreed and confirmed the edge case is acceptable.
