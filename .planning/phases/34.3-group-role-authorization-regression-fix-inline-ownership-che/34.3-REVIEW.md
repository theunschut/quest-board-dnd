# Phase 34.3: Code Review Report

**Reviewed:** 2026-07-02T14:39:18Z
**Depth:** standard
**Files Reviewed:** 15
**Status:** issues_found

## Summary

This phase migrates DM/Admin ownership checks off the (now largely empty) ASP.NET Core Identity
role table and onto per-group `GroupRole` lookups (`GetEffectiveGroupRoleAsync`), and adds a
SuperAdmin bypass. The migration is applied correctly and consistently for the `Id`-based
ownership checks (`currentUser.Equals(quest.DungeonMaster)`), and the regression test suite is
strong for the "non-owner Admin is not incorrectly Forbidden" scenarios described in the plan.

However, the review surfaced two systemic classes of BLOCKER-severity bugs that the stated
migration goal does not fully address:

1. **Name-based DM ownership checks remain in four call sites** (`QuestController.Manage`,
   `CreateFollowUp` GET/POST, `QuestLogController.Manage`-equivalent `UpdateRecap`/`Details`),
   comparing `currentUser.Name == quest.DungeonMaster?.Name` instead of using `Id` or `.Equals()`.
   `User.Name` has no uniqueness constraint at the database or application level — any
   authenticated user can rename themselves (via `AccountController.Edit`) to collide with a
   target DM's display name and inherit that DM's authorization on affected endpoints.
2. **`activeGroupContext.RequireActiveGroupId()` is called unconditionally in several action
   bodies that are reachable by an authenticated SuperAdmin**, whose `ActiveGroupId` is
   documented and enforced elsewhere in the codebase (`GroupSessionMiddleware`,
   `DungeonMasterHandler`) to legitimately be `null`. `QuestController.Index` was given a
   special-cased SuperAdmin short-circuit, but the same fix was not propagated to
   `QuestController.Details`, `Manage`, `Edit`, `Delete`, `Finalize`, `Open`, `SendReminder`,
   `CreateFollowUp` (x2), `QuestLogController.Details`/`UpdateRecap`, or any action in
   `DungeonMasterController`. These will throw `InvalidOperationException` for a SuperAdmin with
   no active group. The existing `Delete_SuperAdmin_IsNotForbidden` regression test does not catch
   this because the shared test fixture defaults `ActiveGroupId = 1` (non-null).

A third, narrower issue: `AccountController.ConfirmEmailChange` decodes an unauthenticated,
user-supplied base64url token without the try/catch that the sibling `SetPassword` action uses
for the same operation, so a malformed/tampered link throws instead of failing gracefully.

## Critical Issues

### CR-01: Name-based DM ownership checks allow authorization bypass via display-name collision

**File:** `QuestBoard.Service/Controllers/QuestBoard/QuestController.cs:288,748,769,824`
**File:** `QuestBoard.Service/Controllers/QuestBoard/QuestLogController.cs:58,95`

**Issue:** Several ownership/authorization checks compare `User.Name` instead of `User.Id`:

```csharp
// QuestController.cs:288 (Details — drives ViewBag.CanManage)
var isQuestDm = currentUser?.Name == quest.DungeonMaster?.Name;

// QuestController.cs:748 (Manage — drives ViewBag.IsAuthorized, gates Finalize/Open/etc. UI)
var isQuestDm = currentUser.Name.Equals(quest.DungeonMaster?.Name, StringComparison.OrdinalIgnoreCase);

// QuestController.cs:769, 824 (CreateFollowUp GET/POST — directly gates Forbid())
var isQuestDm = currentUser.Name.Equals(original.DungeonMaster?.Name, StringComparison.OrdinalIgnoreCase);
if (!isQuestDm && !isAdmin) return Forbid();

// QuestLogController.cs:58 (Details — drives ViewBag.CanEditRecap)
var isQuestDm = currentUser?.Name == quest.DungeonMaster?.Name;

// QuestLogController.cs:95 (UpdateRecap — directly gates Forbid())
var isQuestDm = currentUser.Name.Equals(quest.DungeonMaster?.Name, StringComparison.OrdinalIgnoreCase);
if (!isQuestDm && !isAdmin) return Forbid();
```

`User.Name` has no uniqueness constraint: the only DB index on the `Users`/`AspNetUsers` table is
on `NormalizedUserName`/`NormalizedEmail` (Identity's login fields), not the domain `Name` display
field. `AdminController.CreateUser` does not call `IUserService.ExistsAsync(name)` before
creating a user, and `AccountController.Edit` (`AccountController.cs:206`) lets any authenticated
user set `user.Name = model.Name` with no uniqueness check at all:

```csharp
user.Name = model.Name;   // AccountController.cs:206 — no ExistsAsync check
...
await userService.UpdateAsync(user);
```

Combined, this means: DM B can rename themselves (via `/Account/Edit`) to exactly match DM A's
display `Name`, then reach `QuestController.CreateFollowUp`, `QuestLogController.UpdateRecap`, or
see `Manage`'s admin-only controls for any quest owned by DM A — despite not being DM A and not
being a group Admin. This directly contradicts the intent of this phase (verified by the sibling
`Id`-based checks at `QuestController.cs:132,185,244,619,657,692`, which use the correct
`currentUser.Equals(quest.DungeonMaster)` pattern in the very same file).

**Fix:** Replace every `Name`-based comparison with an `Id`-based comparison, consistent with the
rest of the file:
```csharp
var isQuestDm = currentUser.Id == quest.DungeonMaster?.Id;
// or, where a full User is available (matches the pattern already used at QuestController.cs:132):
var isQuestDm = currentUser.Equals(quest.DungeonMaster);
if (!isQuestDm && role != GroupRole.Admin)
{
    return Forbid();
}
```

### CR-02: `RequireActiveGroupId()` called unconditionally in SuperAdmin-reachable actions — throws for SuperAdmin with no active group

**File:** `QuestBoard.Service/Controllers/QuestBoard/QuestController.cs:131,184,243,289,618,656,691,724,749,770,825`
**File:** `QuestBoard.Service/Controllers/QuestBoard/QuestLogController.cs:59,96`
**File:** `QuestBoard.Service/Controllers/DungeonMaster/DungeonMasterController.cs:32,61,92`

**Issue:** The codebase has an established, documented invariant: SuperAdmin users legitimately
have `ActiveGroupId == null` (see `GroupSessionMiddleware.cs:16-17`: "SuperAdmin passes through —
a null ActiveGroupId is correct by design"; `DungeonMasterHandler.cs:16-21` explicitly checks
`IsInRole("SuperAdmin")` and short-circuits *before* touching `ActiveGroupId`). `Extensions/ActiveGroupContextExtensions.cs:10-12`'s own XML doc warns: "Do NOT use this on the
SuperAdmin/see-all/seeding paths, where a null ActiveGroupId is intentional... calling it there
would incorrectly turn valid 'see all' behavior into an error."

`QuestController.Index` (`QuestController.cs:41-47`) was correctly updated in this phase to
special-case SuperAdmin and avoid calling `RequireActiveGroupId()` for that role:
```csharp
var role = User.IsInRole("SuperAdmin")
    ? GroupRole.Admin
    : await userService.GetEffectiveGroupRoleAsync(User, activeGroupContext.RequireActiveGroupId());
```

But every other action in the same controller (and in `QuestLogController` and
`DungeonMasterController`) still calls `activeGroupContext.RequireActiveGroupId()`
unconditionally whenever a user is authenticated, e.g.:
```csharp
// QuestController.cs:289 (Details — GET, no [Authorize] attribute, reachable by SuperAdmin)
var isAdmin = currentUser != null && await userService.GetEffectiveGroupRoleAsync(User, activeGroupContext.RequireActiveGroupId()) == GroupRole.Admin;

// DungeonMasterController.cs:32 (Profile — [Authorize] class-level only, reachable by SuperAdmin)
if (currentUser != null)
{
    role = await userService.GetEffectiveGroupRoleAsync(User, activeGroupContext.RequireActiveGroupId());
}
```

For `[Authorize(Policy = "DungeonMasterOnly")]`-gated actions (`Edit`, `Delete`, `Finalize`,
`Open`, `SendReminder`, `Manage`, `CreateFollowUp` x2, `UpdateRecap`), `DungeonMasterHandler`
grants SuperAdmin access via its own bypass *without* requiring a non-null `ActiveGroupId` — so a
SuperAdmin with no selected group passes the `[Authorize]` filter and then crashes inside the
action body on `RequireActiveGroupId()`, which throws:
```csharp
public static int RequireActiveGroupId(this IActiveGroupContext context) =>
    context.ActiveGroupId
        ?? throw new InvalidOperationException("Active group context is not initialized. This request requires a selected group.");
```

`DungeonMasterController.Profile` has no policy gate at all beyond controller-level `[Authorize]`,
so any authenticated SuperAdmin viewing *any* DM's profile page (`GET /DungeonMaster/Profile/{id}`)
will 500.

This is untested: `QuestControllerAuthorizationRegressionTests.Delete_SuperAdmin_IsNotForbidden`
is the only SuperAdmin regression test in the reviewed files, and it runs against the shared test
fixture's default `TestGroupContext.ActiveGroupId = 1` (confirmed via
`AccountControllerIntegrationTests.Profile_WhenActiveGroupIdIsNull_...`, which explicitly sets
`ActiveGroupId = null` and then restores it to `1` in a `finally` block — proving `1` is the
ambient default for every other test). No test exercises a SuperAdmin with `ActiveGroupId == null`
against `QuestController` or `DungeonMasterController`, so this regression is currently invisible
to the suite.

**Fix:** Apply the same short-circuit pattern used in `Index` everywhere `GetEffectiveGroupRoleAsync`
is invoked on a path reachable by SuperAdmin, e.g.:
```csharp
var role = User.IsInRole("SuperAdmin")
    ? GroupRole.Admin
    : await userService.GetEffectiveGroupRoleAsync(User, activeGroupContext.RequireActiveGroupId());
```
or, better, push the bypass into `GetEffectiveGroupRoleAsync`'s caller contract by adding a
group-id-optional overload/helper so call sites don't need to special-case `RequireActiveGroupId()`
at all when the group id isn't otherwise needed. At minimum, add an integration test that
authenticates as SuperAdmin with `ActiveGroupId = null` and hits each of the affected
`DungeonMasterOnly`-gated actions and `QuestController.Details` / `DungeonMasterController.Profile`.

### CR-03: `AccountController.ConfirmEmailChange` — unguarded token decode on anonymous, user-controlled input

**File:** `QuestBoard.Service/Controllers/Admin/AccountController.cs:244-247`
**Issue:**
```csharp
[HttpGet]
public async Task<IActionResult> ConfirmEmailChange(int userId, string newEmail, string token)
{
    var decodedToken = Encoding.UTF8.GetString(WebEncoders.Base64UrlDecode(token));
    var result = await identityService.ChangeEmailAsync(userId, newEmail, decodedToken);
    ...
```
This action has no `[Authorize]` attribute and is invoked directly from an email link with
attacker-influenceable query-string parameters (`userId`, `newEmail`, `token`). Unlike the sibling
`SetPassword` POST action (`AccountController.cs:78-107`), which wraps the equivalent
`WebEncoders.Base64UrlDecode` + reset call in a `try/catch` that logs and redirects with a
friendly "link may be expired or invalid" message, `ConfirmEmailChange` performs no exception
handling. `WebEncoders.Base64UrlDecode` throws `FormatException` on malformed input (e.g. a
tampered or hand-crafted `token` value with invalid base64url characters or padding), so a request
to `/Account/ConfirmEmailChange?userId=1&newEmail=x&token=%25%25invalid` bypasses the intended
"Email confirmation failed" soft-fail path entirely and instead surfaces the generic exception
handler page, with no `logger.LogError` call to record the attempt (compare to `SetPassword`,
which logs the exception with the affected `userId`).

**Fix:** Mirror the `SetPassword` pattern:
```csharp
[HttpGet]
public async Task<IActionResult> ConfirmEmailChange(int userId, string newEmail, string token)
{
    try
    {
        var decodedToken = Encoding.UTF8.GetString(WebEncoders.Base64UrlDecode(token));
        var result = await identityService.ChangeEmailAsync(userId, newEmail, decodedToken);

        if (result.Succeeded)
            TempData["SuccessMessage"] = "Email address updated. Please sign in with your new address.";
        else
            TempData["ErrorMessage"] = "Email confirmation failed. The link may have expired.";
    }
    catch (Exception ex)
    {
        logger.LogError(ex, "ConfirmEmailChange failed for userId {UserId}", userId);
        TempData["ErrorMessage"] = "Email confirmation failed. The link may have expired.";
    }

    return RedirectToAction(nameof(Login));
}
```

## Warnings

### WR-01: `GetGroupRoleAsync` vs `GetEffectiveGroupRoleAsync` naming is a foot-gun for future call sites

**File:** `QuestBoard.Domain/Interfaces/IUserService.cs:46-55`
**Issue:** The interface now exposes two very similarly named methods with materially different
semantics: `GetGroupRoleAsync` (raw membership lookup, returns `null` for SuperAdmin unless they
happen to also be a group member) and `GetEffectiveGroupRoleAsync` (SuperAdmin-aware, treats
SuperAdmin as `GroupRole.Admin` unconditionally). All of the controllers reviewed correctly call
the `Effective` variant for authorization decisions, but `DungeonMasterHandler` (referenced, not
in review scope) calls the non-`Effective` `GetGroupRoleAsync` directly after its own separate
SuperAdmin short-circuit — meaning the two code paths independently re-implement the SuperAdmin
bypass rather than sharing it through `GetEffectiveGroupRoleAsync`. This duplication is exactly
the kind of drift that produced CR-02: one caller's bypass logic evolves (or is added correctly)
while a sibling caller's does not.
**Fix:** Consider consolidating the SuperAdmin short-circuit into a single well-tested location
(e.g. have `DungeonMasterHandler`/`AdminHandler` call `GetEffectiveGroupRoleAsync` instead of
duplicating the `IsInRole("SuperAdmin")` check), and/or rename `GetGroupRoleAsync` to something
that makes the "raw, non-SuperAdmin-aware" contract unmistakable at call sites (e.g.
`GetRawGroupRoleAsync` or add an XML doc `<remarks>` warning against use in authorization checks).

### WR-02: Inconsistent DM-ownership comparison style across the same controller increases risk of future regressions

**File:** `QuestBoard.Service/Controllers/QuestBoard/QuestController.cs`
**Issue:** Independent of CR-01's correctness bug, the file mixes three different equality styles
for "is the current user this quest's DM": `currentUser.Equals(quest.DungeonMaster)` (7 call
sites), `currentUser?.Name == quest.DungeonMaster?.Name` (1 call site), and
`currentUser.Name.Equals(quest.DungeonMaster?.Name, StringComparison.OrdinalIgnoreCase)` (3 call
sites). Even setting aside the security issue, three different idioms for the same semantic check
within one class is a maintainability hazard — the next contributor has no clear "correct" pattern
to copy.
**Fix:** Once CR-01 is fixed, standardize on a single private helper, e.g.:
```csharp
private static bool IsQuestOwner(User currentUser, User? dungeonMaster) =>
    dungeonMaster != null && currentUser.Id == dungeonMaster.Id;
```
and use it at every call site instead of `Equals`, `Name ==`, or `Name.Equals(...)`.

### WR-03: `User.Equals`/`GetHashCode` implement full value equality, which is surprising for an entity with an `Id`

**File:** `QuestBoard.Domain/Models/User.cs:26-39`
**Issue:** `User.Equals` compares `Id`, `Name`, `Email`, `HasKey`, and `EmailConfirmed` — i.e.
value equality rather than identity (entity `Id`) equality. This happens to work correctly for the
`currentUser.Equals(quest.DungeonMaster)` call sites in this diff because both operands are
loaded via full entity projections with matching fields, but it is fragile: if either loader is
later changed to a partial/lighter projection (e.g. dropping `EmailConfirmed` or `HasKey` from a
list view's DTO mapping), every `.Equals()`-based ownership check silently starts returning
`false` for the same physical user, converting what should be a "DM can manage their own quest"
path into an incorrect `Forbid()`. This is not a new defect introduced by this diff, but this diff
newly depends on `.Equals()` semantics more heavily for authorization, which raises the blast
radius of a future change to this method.
**Fix:** Add an `Id`-only equality overload (or use `currentUser.Id == quest.DungeonMaster?.Id`
directly, per WR-02) for authorization-sensitive comparisons, reserving full-value `Equals` (if
still needed elsewhere) for a differently-named method or explicit comparer.

## Info

### IN-01: `DungeonMasterController.Profile` computes `role` even when it is only used for `CanEdit`, duplicating logic already covered by `currentUser.Equals(user)`

**File:** `QuestBoard.Service/Controllers/DungeonMaster/DungeonMasterController.cs:29-33,41`
**Issue:** Minor: the `role` lookup is only consumed by the single `CanEdit` boolean expression.
Once CR-02 is fixed (adding a SuperAdmin short-circuit here too), consider extracting a small
private helper shared with `EditProfile`'s duplicated `role != GroupRole.Admin` check
(`DungeonMasterController.cs:61-65,92-96`) to avoid tripling the "is this Admin-or-self" logic
across three actions in the same controller.
**Fix:** Not required for correctness; a small `IsSelfOrAdminAsync(ClaimsPrincipal, User target, int groupId)` helper would reduce duplication once CR-02 is addressed.

### IN-02: `AccountController.Edit` silently drops a `Name` collision instead of surfacing it to the user

**File:** `QuestBoard.Service/Controllers/Admin/AccountController.cs:206`
**Issue:** Tied to CR-01: even after fixing the `Name`-based authorization checks, allowing
unrestricted duplicate `Name` values is still a UX smell — two DMs with an identical displayed
name are indistinguishable in quest listings, DM profile links (`DungeonMasterController.Profile`
`Name` display), and email notifications that reference `quest.DungeonMaster?.Name` (see
`QuestService.cs` email dispatch code, out of this review's file scope but visible via grep).
**Fix:** Consider adding a `ModelState.AddModelError` when `userService.ExistsAsync(model.Name)`
returns true for a name different from the user's current one, mirroring how email uniqueness is
already enforced by Identity's `RequireUniqueEmail`.
