## Fixed Issues

### CR-01: Name-based DM ownership checks allow authorization bypass via display-name collision

**Files modified:** `QuestBoard.Service/Controllers/QuestBoard/QuestController.cs`, `QuestBoard.Service/Controllers/QuestBoard/QuestLogController.cs`
**Commit:** `5b5ff48`
**Applied fix:** Replaced every `Name`-based DM comparison (`currentUser?.Name == quest.DungeonMaster?.Name` and `currentUser.Name.Equals(..., StringComparison.OrdinalIgnoreCase)`) with `Id`-based comparisons, at `QuestController.Details` (GET), `QuestController.Manage`, `QuestController.CreateFollowUp` (GET and POST), `QuestLogController.Details` (GET), and `QuestLogController.UpdateRecap`. This closes the display-name collision authorization bypass described in the finding. Committed together with CR-02 because both fixes touched the same statements at several of these call sites (the CR-02 helper replaced the `RequireActiveGroupId()` calls immediately adjacent to the CR-01 ownership checks).

### CR-02: `RequireActiveGroupId()` called unconditionally in SuperAdmin-reachable actions — throws for SuperAdmin with no active group

**Files modified:** `QuestBoard.Service/Controllers/QuestBoard/QuestController.cs`, `QuestBoard.Service/Controllers/QuestBoard/QuestLogController.cs`, `QuestBoard.Service/Controllers/DungeonMaster/DungeonMasterController.cs`
**Commit:** `5b5ff48`
**Applied fix:** Added a private `GetEffectiveRoleAsync()` helper to each of the three controllers, mirroring the SuperAdmin short-circuit pattern already used in `QuestController.Index` (`User.IsInRole("SuperAdmin") ? GroupRole.Admin : await userService.GetEffectiveGroupRoleAsync(User, activeGroupContext.RequireActiveGroupId())`), and replaced every unconditional `activeGroupContext.RequireActiveGroupId()` role-check call site with a call to this helper — `QuestController.Edit` (GET/POST), `Delete`, `Details` (GET), `Finalize`, `Open`, `SendReminder`, `Manage`, `CreateFollowUp` (GET/POST); `QuestLogController.Details` (GET), `UpdateRecap`; `DungeonMasterController.Profile`, `EditProfile` (GET/POST). One call site in `SendReminder` (the `reminderJobDispatcher.EnqueueSessionReminder(id, activeGroupContext.RequireActiveGroupId(), ...)` call) needed a different fix, since it requires a concrete group id for the reminder job rather than a role check with an Admin bypass — a SuperAdmin sending a reminder genuinely has no "current group" to hand to that job. Changed it to use `quest.GroupId` (the quest's own group, populated as a plain scalar column on `GetQuestWithDetailsAsync`) instead of the ambient active-group context, which both fixes the SuperAdmin crash and is more correct for any caller regardless of role.

### CR-03: `AccountController.ConfirmEmailChange` — unguarded token decode on anonymous, user-controlled input

**Files modified:** `QuestBoard.Service/Controllers/Admin/AccountController.cs`
**Commit:** `bc791ab`
**Applied fix:** Wrapped the `WebEncoders.Base64UrlDecode` + `identityService.ChangeEmailAsync` call in a `try/catch`, mirroring the sibling `SetPassword` action's pattern exactly — on exception, logs via `logger.LogError(ex, "ConfirmEmailChange failed for userId {UserId}", userId)` and sets the same "Email confirmation failed. The link may have expired." `TempData["ErrorMessage"]` before redirecting to `Login`, instead of letting a malformed/tampered token throw past the intended soft-fail path.

### WR-01: `GetGroupRoleAsync` vs `GetEffectiveGroupRoleAsync` naming is a foot-gun for future call sites

**Files modified:** `QuestBoard.Domain/Interfaces/IUserService.cs`
**Commit:** `ea3ec29`
**Applied fix:** Added an XML doc `<remarks>` block to `GetGroupRoleAsync`'s interface declaration explicitly warning that it does not apply the SuperAdmin bypass and pointing callers making authorization decisions to `GetEffectiveGroupRoleAsync` instead (or an explicit short-circuit, as `DungeonMasterHandler`/`AdminHandler` already do). Chose the documentation option over consolidating/renaming: `GetEffectiveGroupRoleAsync(ClaimsPrincipal, int groupId)` requires a non-nullable concrete group id, so `DungeonMasterHandler`/`AdminHandler` would still need their own null-group guard before they could call it — consolidating would require a larger interface change (e.g. a new `int?`-accepting overload) that goes beyond what this Warning-severity naming/documentation finding calls for. A rename was also rejected: `GetGroupRoleAsync` has call sites in `UserService.cs`, both authorization handlers, and `UserServiceTests.cs`, which is a larger blast radius than warranted here.

### WR-02: Inconsistent DM-ownership comparison style across the same controller increases risk of future regressions

**Files modified:** `QuestBoard.Service/Controllers/QuestBoard/QuestController.cs`
**Commit:** `1f74666`
**Applied fix:** Added a private static `IsQuestOwner(User currentUser, User? dungeonMaster)` helper (`dungeonMaster != null && currentUser.Id == dungeonMaster.Id`) exactly as suggested in the finding, and replaced all 10 call sites in `QuestController.cs` — the 7 `.Equals()`-based sites (`Edit` GET/POST, `Delete`, `Finalize`, `Open`, `SendReminder`) and the 3 CR-01-fixed `Id ==` inline sites (`Details`, `Manage`, `CreateFollowUp` GET/POST) — with calls to this single helper, so every "is the current user this quest's DM" check in the file now uses one idiom.

### WR-03: `User.Equals`/`GetHashCode` implement full value equality, which is surprising for an entity with an `Id`

**Files modified:** `QuestBoard.Service/Controllers/DungeonMaster/DungeonMasterController.cs`
**Commit:** `9649277`
**Applied fix:** Replaced the three remaining `.Equals()`-based self/ownership checks in `DungeonMasterController` (`Profile`'s `CanEdit` computation, and the identical guard in both `EditProfile` GET and POST) with direct `Id` comparisons (`currentUser.Id == user.Id` / `currentUser.Id != targetUser.Id`), removing the dependency on `User`'s full value equality (`Id`, `Name`, `Email`, `HasKey`, `EmailConfirmed`) for authorization decisions in this controller. Chose direct `Id` comparison over adding a new `Id`-only equality overload on the shared `User` model, per the fix's own suggested alternative — this is a smaller, more localized change that doesn't risk altering `.Equals()`/`GetHashCode()` behavior for any other (non-authorization) caller of `User` elsewhere in the codebase.
