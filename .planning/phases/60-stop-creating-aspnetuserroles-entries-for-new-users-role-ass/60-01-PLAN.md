<objective>
Stop new-user creation from writing rows to `AspNetUserRoles`, and remove the now-fully-dead per-group Identity-role read/write API surface that lingered after Phase 27/34.3 moved per-group roles to `UserGroups.GroupRole`.

This is a correctness fix plus dead-code deletion. Per-group role assignment (Player/DungeonMaster/Admin) is already handled exclusively by `UserService.SetGroupRoleAsync` writing `UserGroups.GroupRole` — that path is correct and is NOT touched. The system-wide `SuperAdmin` ASP.NET Core Identity role IS still real and IS NOT touched.

Purpose: `IdentityService.CreateUserAsync` currently calls `userManager.AddToRoleAsync(entity, "Player")` on every new account, re-creating stale `AspNetUserRoles` rows that nothing reads. The parallel dead API (`AddToRoleAsync` / `RemoveFromRoleAsync` / `IsInRoleAsync` / `GetRolesAsync` on both service interfaces) has zero non-internal callers and should be removed to prevent future misuse.

Output: Modified `IdentityService.cs`, `IIdentityService.cs`, `UserService.cs`, `IUserService.cs`, and the test helper `AuthenticationHelper.cs`; no new files; no migration.
</objective>
