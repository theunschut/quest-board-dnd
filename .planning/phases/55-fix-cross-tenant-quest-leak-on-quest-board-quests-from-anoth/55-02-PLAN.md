<objective>
Close the confirmed root cause: `GroupSessionMiddleware` currently exempts SuperAdmin from the "must have an active group" gate on EVERY route, so a SuperAdmin with a null `ActiveGroupId` reaches `/quests` and sees every group's data merged. Reorder the guard clauses so the exempt-path check runs first (D-02 — SuperAdmin still bypasses on `/platform`/`/Error`/GroupPicker/Account) and remove the blanket SuperAdmin bypass so SuperAdmin is gated on group-scoped routes exactly like every other role (D-01).

Purpose: This is the middleware half of the fix — the primary gate. Combined with Plan 01's fail-closed filter, a SuperAdmin can never render a board in the ambiguous "every group merged" state. Also correct the now-stale `CONCERNS.md` rationale so a future reader doesn't re-introduce the bypass.
Output: Reordered middleware, rewritten class doc comment, a rewritten existing integration test (asserting the new/opposite behavior), a new SuperAdmin `[Theory]` variant, and corrected `CONCERNS.md`.
</objective>
