<objective>
Close the residual time-axis gap (D-06): a user removed from a group mid-session keeps board access until they naturally re-select a group, because their non-null `ActiveGroupId` still matches a real group (so Plan 01's filter and Plan 02's null-gate both pass). Add interval-gated re-validation inside `GroupSessionMiddleware`: whenever `ActiveGroupId` is written, stamp a `ActiveGroupValidatedAtUtc` timestamp; on each request, if more than 5 minutes have elapsed, re-check membership via `GetGroupRoleByIdAsync` and gate the user out (clear the group, redirect/409) if they are no longer a member. SuperAdmin is excluded (no membership to go stale).

Purpose: This is the same class of correctness gap as D-01/D-03, on the time axis. The mechanism reuses the *concept* of Phase 41's `SecurityStampValidatorOptions.ValidationInterval` (5-minute bounded staleness) but is implemented as an independent session-timestamp check inside the middleware, because `ActiveGroupId` lives in Session, not in the auth cookie's claims (RESEARCH.md Pitfall 1).
Output: new `SessionKeys.ActiveGroupValidatedAtUtc` constant, timestamp stamping at both `ActiveGroupId` write-sites in `GroupPickerController`, the re-validation block in `GroupSessionMiddleware`, and a new middleware unit test covering the stale/fresh/removed-member branches.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Long-lived session (ActiveGroupId set at t=0) → later request at t=N | A membership granted at selection time may have been revoked; the authorization decision must be re-checked as time passes, not assumed permanent |
| Session-stored timestamp → re-validation trigger | An attacker-controlled or corrupted timestamp must not disable the re-check; unparseable/absent timestamps force a re-validation, not a skip |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-55-10 | Elevation of Privilege | Stale authorization — a removed group member retains board access until they re-select a group | mitigate | Interval-gated re-validation via `GetGroupRoleByIdAsync`; a removed member is gated out (session cleared, redirect/409) within a 5-minute bounded staleness window, mirroring Phase 41's `SecurityStampValidatorOptions` intent for the analogous account-disabled-mid-session case |
| T-55-11 | Tampering | A corrupted/absent `ActiveGroupValidatedAtUtc` used to indefinitely skip re-validation | mitigate | `needsRevalidation` treats null OR unparseable timestamps as "must re-validate now" — fail toward re-checking, never toward skipping |
| T-55-12 | Denial of Service | A membership DB round-trip on every request under load | accept | Re-check is interval-gated (once per 5 min per active user), not per-request; negligible at this app's scale — the same tradeoff Phase 41 accepted for its 5-minute security-stamp interval |
| T-55-SC | Tampering | npm/pip/cargo installs | accept | No package installs — reuses the existing `IUserService` primitive and BCL `DateTime`/`ISession`; RESEARCH.md Package Legitimacy Audit records zero packages evaluated |
