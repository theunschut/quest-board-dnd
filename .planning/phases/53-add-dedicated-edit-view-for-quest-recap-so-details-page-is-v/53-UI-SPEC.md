## Design System

| Property | Value |
|----------|-------|
| Tool | none — ASP.NET Core MVC + Razor views, Bootstrap 5 utility classes, project-custom CSS. No `components.json`/shadcn/npm frontend toolchain exists in this repo; the shadcn gate is not applicable. |
| Preset | not applicable |

This phase adds **zero new CSS classes**. It reuses `.modern-card`/`.modern-card-header`/`.modern-card-body` (desktop), `.quest-edit-card-mobile` (mobile, exact class from `quest-edit.mobile.css`), and `.recap-display-box` (both) verbatim from existing views. No new stylesheet file is needed.

## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon-to-text gaps (`me-2` on FontAwesome icons — Bootstrap's `me-2` = 8px, see exception below) |
| sm | 8px | `me-2`/`mb-2` utility spacing between inline elements |
| md | 16px | `mb-3` between form groups (Bootstrap default), `.modern-card-body` padding, `.quest-edit-card-mobile` padding |
| lg | 24px | `mb-4` between major page sections |
| xl | 32px | Not used in this phase |
| 2xl | 48px | Not used in this phase |
| 3xl | 64px | Not used in this phase |

Exceptions: This phase uses Bootstrap's built-in spacing utility classes (`mb-3`, `me-2`, `mb-0`, etc.) rather than raw pixel values, exactly as `Quest/Edit.cshtml` and `Details.cshtml` already do. Bootstrap's default spacer unit is `1rem` (16px) with `mb-3` = 16px, `me-2` = 8px, `mb-4` = 24px — all multiples of 4, no deviation from the codebase's established pattern. Do not introduce inline `style="margin:..."` — use the same Bootstrap utility classes the mirrored `Quest/Edit.cshtml` view already uses.

| Display | Not applicable — no display-scale text in this phase | — | — |

## Color

This phase makes zero new color decisions — it strictly reuses the existing theme already applied to every other `modern-card` view (parchment-on-dark-glass, warm brown/gold accent). Documented for traceability:

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | Page background gradient (defined globally in `site.css`, unchanged by this phase) | Full-page backdrop behind all cards |
| Secondary (30%) | `rgba(255, 255, 255, 0.15)` glass card surface (`.modern-card` desktop) / `rgba(255, 255, 255, 0.15)` (`.quest-edit-card-mobile`) | The Edit Recap card itself (desktop + mobile), the read-only Details card |
| Accent (10%) | `#8B4513`→`#A0522D` brown gradient (`.btn-primary`) | **Reserved for exactly:** the "Add Recap"/"Edit Recap" entry-point button on Details, and the "Save Recap" submit button on the new Edit Recap page. Per D-03, Save Recap must be `btn-primary` (not `btn-warning`, which is what the mirrored `Quest/Edit.cshtml` uses for its "Update Quest" button — do not copy that detail; D-03 explicitly locks Save Recap as primary/right, matching the color already used by the current inline desktop Save Recap button today). |
| Destructive | `#e74c3c`→`#c0392b` (`.btn-danger`) | Not used in this phase — there is no destructive action (Cancel is a neutral navigation action, not a delete/destroy). |

Accent reserved for: the recap entry-point button on Details ("Add Recap" / "Edit Recap") and the "Save Recap" primary submit button on the Edit Recap page. The Cancel button on the Edit Recap page uses `.btn-secondary` (gray gradient), never the accent color — this keeps the 10%-accent budget meaningful by not tinting every button on the page.

## View-by-View Contract

### 1. `QuestLog/Details.cshtml` (desktop) — Session Recap section becomes read-only + entry point

Replace lines 100–136 (the `@if ((bool)ViewBag.CanEditRecap) { <form>...</form> } else { ...read-only... }` block) with:

```html
<!-- Session Recap -->
<div class="mb-4">
    <h5 class="text-white"><i class="fas fa-book me-2"></i>Session Recap</h5>

    @if (!string.IsNullOrWhiteSpace(Model.Quest.Recap))
    {
        <div class="recap-display-box">
            @Model.Quest.Recap
        </div>
    }
    else
    {
        <p class="text-muted">No recap has been written for this quest yet.</p>
    }

    @if ((bool)ViewBag.CanEditRecap)
    {
        <a href="@Url.Action("EditRecap", "QuestLog", new { id = Model.Quest.Id })" class="btn btn-primary mt-3">
            @if (!string.IsNullOrWhiteSpace(Model.Quest.Recap))
            {
                <i class="fas fa-edit me-2"></i>@:Edit Recap
            }
            else
            {
                <i class="fas fa-plus me-2"></i>@:Add Recap
            }
        </a>
    }
</div>
```

Key contract points:
- The read-only display (`.recap-display-box` / empty-state `<p class="text-muted">`) renders **unconditionally** now — no more `CanEditRecap` branch gating what's *displayed*, only gating whether the button appears underneath (D-01).
- Button sits directly under the recap content/empty-state, inside the same `<div class="mb-4">` — not a new section, not moved to the Quick Actions sidebar (D-01).
- `mt-3` spacing between the recap box/empty-state text and the button (matches the existing `mb-3`/`mt-3` rhythm used elsewhere in this same card).
- Button is `btn-primary` (accent color, single CTA — this is the one interactive element in an otherwise read-only section, consistent with the accent's 10% reservation).
- Icon swaps with label per D-02: `fa-plus` for "Add Recap" (empty state), `fa-edit` for "Edit Recap" (has content) — reuses the same `!string.IsNullOrWhiteSpace(Model.Quest.Recap)` condition already in scope, no new ViewBag/ViewModel field needed.

### 3. `QuestLog/EditRecap.cshtml` (desktop, new file) — mirrors `Quest/Edit.cshtml` structure

```html
@using QuestBoard.Service.ViewModels.QuestLogViewModels
@model EditRecapViewModel
@{
    ViewData["Title"] = $"Edit Recap: {Model.Quest.Title}";
}

<div class="row">
    <div class="col-lg-8 col-md-7">
        <div class="card modern-card">
            <div class="card-header modern-card-header">
                <h2 class="mb-0">
                    <i class="fas fa-book me-2"></i>
                    @ViewData["Title"]
                </h2>
            </div>
            <div class="card-body modern-card-body">
                <div asp-validation-summary="All" class="text-danger mb-3"></div>
                <form asp-action="EditRecap" asp-route-id="@Model.Id" method="post">
                    <input type="hidden" asp-for="Id" />

                    <div class="mb-3">
                        <label for="recap" class="form-label">
                            Write or update the session recap:
                        </label>
                        <textarea class="form-control"
                                  id="recap"
                                  name="recap"
                                  rows="10"
                                  placeholder="Describe what happened during this quest...">@Model.Recap</textarea>
                        <small class="form-text text-muted">
                            Share the story of this adventure with your players!
                        </small>
                    </div>

                    <hr />

                    <div class="d-flex justify-content-between">
                        <a href="@Url.Action("Details", "QuestLog", new { id = Model.Id })" class="btn btn-secondary">
                            <i class="fas fa-times me-2"></i>Cancel
                        </a>
                        <button type="submit" class="btn btn-primary">
                            <i class="fas fa-save me-2"></i>Save Recap
                        </button>
                    </div>
                </form>
            </div>
        </div>
    </div>
</div>
```

Key contract points (all locked, not discretionary):
- Card header icon: `fa-book` — matches the icon already used for "Session Recap" on the Details page (visual continuity between the two pages for the same concept), **not** `fa-edit` (that's reserved as the entry-point button's icon when content exists, per D-02 — using it again on the page header would be redundant within the same user flow).
- No right-column "tips" sidebar (unlike `Quest/Edit.cshtml`'s `col-lg-4` tips card) — recap editing is a single free-text field with no multi-step gotchas worth calling out; adding a tips panel here would be inventing UI surface CONTEXT.md never asked for. Use a single full-width-within-row `col-lg-8 col-md-7` column (kept narrower than full 12 columns for readability, matching the proportion `Quest/Edit.cshtml` already uses for its own main form column).
- `<hr />` before the button row — required by CLAUDE.md convention, present here (Quest/Edit.cshtml's mobile variant has this too but its desktop variant does not consistently use `<hr>` before buttons; this phase's new view follows the CLAUDE.md-documented convention explicitly, which takes precedence).
- Button layout: `d-flex justify-content-between` — Cancel (`btn-secondary`, left) / Save Recap (`btn-primary`, right) — per D-03, this is a locked decision, not a stylistic default.
- Cancel is an `<a>` tag (plain navigation), not a `<button>` — matches `Quest/Edit.cshtml`'s existing Cancel implementation exactly (no `onclick`/JS confirmation, no unsaved-changes warning — out of scope).
- Textarea: 10 rows, verbatim placeholder and help text carried over per CONTEXT.md Claude's Discretion guidance. Drop the inline `style="background-color: ... !important"` override that exists in today's `Details.cshtml` recap textarea (lines 108–113) — `.modern-card .form-control` in `site.css` (lines 1142–1150) already provides the same high-contrast white-background/dark-text treatment globally, making the inline override redundant now that this textarea lives in its own dedicated `.modern-card`. Use `class="form-control"` with no inline style, exactly like `Quest/Edit.cshtml`'s own `Quest.Description` textarea.
- `text-muted` on the help text `<small>` — replaces today's `text-white` (which only made sense when the textarea sat inside the dark-background Details card without the full `.modern-card .form-text` treatment). `.modern-card .form-text` already sets the correct faded-parchment color; explicit `text-muted`/`text-white` overrides are unnecessary noise once this lives in its own modern-card, so omit the color utility class entirely and let the card's own CSS govern it (shown above as no extra class beyond `form-text`).

Key contract points:
- Reuses `quest-edit.mobile.css` verbatim (`.quest-edit-card-mobile`, `.form-label`, `.form-text` rules already defined at lines 5–43) — no new mobile stylesheet needed; this class was written generically enough (not `.quest-edit-*` field-specific) to apply directly to this new view. **Do not** create a `recap-edit.mobile.css` — that would duplicate identical rules.

- Button row: `d-flex gap-2` with `flex-fill` on both buttons (Cancel left, Save Recap right) — matches `Quest/Edit.Mobile.cshtml`'s exact button-row pattern (lines 101–108), which already satisfies the secondary-left/primary-right convention at mobile width. This differs visually from desktop's `justify-content-between` (buttons don't hug opposite edges) but is the established mobile idiom already used by the sibling Quest/Edit.Mobile.cshtml view — do not deviate to `justify-content-between` on mobile, that would break consistency with every other mobile edit form in this codebase.
- No separate "Back to Quest" link needed beyond Cancel — `Quest/Edit.Mobile.cshtml` labels its equivalent button "Back to Quest" with a `fa-arrow-left` icon instead of "Cancel"/`fa-times`; this phase uses **"Cancel"** with `fa-times` per D-03's explicit copy lock, not "Back to Quest" — do not copy that specific label choice from the mirrored template, only its structural/layout pattern.
