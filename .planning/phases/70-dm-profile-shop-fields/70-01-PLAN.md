<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser → DungeonMasterController (EditProfile POST) | Authenticated DM submits untrusted Bio free-text |
| Browser → MarkdownController (POST /markdown/preview) | Bio editor Preview toggle sends untrusted Bio text for server-side render |
| Stored Bio → Profile read views (Html.Markdown render sink) | Untrusted stored text is converted to HTML for display |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-70-01 | Tampering (XSS) | Bio render sink on Profile.cshtml / Profile.Mobile.cshtml | high | mitigate | Rendering goes exclusively through `Html.Markdown()` → `IMarkdownService.RenderToHtml(..., Web)` (Markdig + Ganss.Xss HtmlSanitizer, built and unit-tested in Phase 65). No `Html.Raw` on the Bio value is introduced. `<script>`, `javascript:` URLs, and attribute-injection payloads are already stripped by the sanitizer. |
| T-70-02 | Spoofing | POST /markdown/preview reachable from the Bio editor | medium | mitigate | Endpoint already enforces `[Authorize]` + antiforgery; `window.markdownAntiforgeryToken` is supplied by the `_QuestFormScripts.cshtml` include added to both write forms. No new endpoint or auth change. |
| T-70-03 | Denial of Service | Oversized Bio input | low | accept | Server-side `[StringLength(2000)]` on `EditDMProfileViewModel.Bio` is unchanged and bounds stored length; dropping the client-side maxlength attribute does not remove server enforcement. |
| T-70-SC | Tampering | npm/NuGet installs | low | accept | No new packages this phase — Markdig / Ganss.Xss / EasyMDE reused verbatim (slopcheck `[OK]` recorded Phase 66). No install task exists in this plan. |
</threat_model>
