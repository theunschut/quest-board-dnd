# Phase 70 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

**Scope note:** This phase mechanically repeats Phase 66's Markdown editor + rendering pattern onto DM Profile Bio (`EditProfile.cshtml`/`.Mobile`, read on `Profile.cshtml`/`.Mobile`) and Shop Item Description (`ShopManagement/Create.cshtml`/`.Mobile`, `Edit.cshtml`/`.Mobile`, read on `_ShopItemDetailsContent.cshtml` and `Shop/Details.Mobile.cshtml`). ## Typography

No new typography surface for the editor/rendered-content path — DM Profile Bio and Shop Item Description render through the exact same `.markdown-content` wrapper Quest/Character/Contact fields already use (`Html.Markdown()` in `HtmlHelperExtensions.cs` wraps every field's output identically in `<div class="markdown-content">`, regardless of which field or view called it). The table documents the current as-built state, inherited unchanged.

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 16px (1rem) | 400 regular | 1.6 |
| Label | 16px (1rem) | 700 bold | 1.2 |
| Heading | stepped 24px→14px (h1 1.5rem down to h6 0.875rem) | 700 bold | 1.2 |
| Display | not used | — | — |

**Accepted Deviation — Heading row (6 sizes, exceeds standard 4-size guideline), carried forward unchanged:**

Same accepted exception documented in `67-UI-SPEC.md`/`68-UI-SPEC.md`/`69-UI-SPEC.md`: the 6-level `.markdown-content h1`–`h6` stepped scale (24/22/20/18/16/14px) plus Body/Label (16px) totals 6 unique font sizes. This is already-shipped, operator-approved, class-scoped CSS from Phase 66 that applies app-wide — not introduced by, or specific to, this phase's fields. Phase 70 does not modify this CSS and does not re-litigate the deviation.

**Heading color — resolved for all 4 render surfaces this phase touches (primary technical finding of this research pass, and a correction of `70-CONTEXT.md`'s stated assumption):**

`70-CONTEXT.md` assumed `.modern-card-body > .markdown-content h1..h6 { color: #1a1a1a; text-shadow: none; }` (`markdown-content.css`, added Phase 66/67) "already covers DM Profile's desktop About card and Shop's desktop Description card" because both are `.modern-card` → `.modern-card-body` with `.markdown-content` as a direct child. Verified against the actual CSS this session: `site.css` line 1094's pre-existing `.modern-card h1, h2, h3, h4, h5, h6 { color: #F4E4BC !important; text-shadow: ... !important; }` matches any heading nested anywhere inside `.modern-card` regardless of depth, and — because it carries `!important` against a non-`!important` compensating override — it **always wins**, exactly as `68-UI-SPEC.md` already established for Character Details. This means the direct-child structural condition `70-CONTEXT.md` flagged as load-bearing is **not actually load-bearing**: the compensating override is dead CSS on every `.modern-card` page regardless of nesting depth, this phase included.

- **DM Profile desktop About card (`Views/DungeonMaster/Profile.cshtml`):** Bio will render as `<div class="markdown-content">@Html.Markdown(Model.Bio)</div>`, a direct child of `.card-body.modern-card-body` inside `.card.modern-card`. Net effect: headings and body text render in the same gold-with-shadow (`#F4E4BC`) as every other field on this card (Name, role label). **This is consistent, not invisible** — matches the established, operator-accepted pattern from every prior phase. **No new desktop CSS needed.**
- **Shop Description desktop card (`Views/Shared/_ShopItemDetailsContent.cshtml`):** identical shape — `.card-body.modern-card-body` → `.card.modern-card`, same `!important` rule applies, same gold-with-shadow outcome. **No new desktop CSS needed.** This applies identically whether the partial renders as a full page (`Shop/Details.cshtml`) or inside the Bootstrap modal (`isModal` branch) — both paths share the same `.modern-card` wrapper, confirmed by `70-CONTEXT.md`'s "Shop Details is read-only in both its rendering contexts" note.
- **DM Profile mobile Bio card (`Views/DungeonMaster/Profile.Mobile.cshtml`, `dm-profile.mobile.css`):** `.dm-profile-bio-text` (the existing `<p>` class, carried onto the new wrapper `<div>`) sets `color: #F4E4BC` and `text-shadow` **without `!important`**. `.markdown-content h1..h6`'s own explicit (also non-`!important`, but directly-targeted rather than inherited) color rule takes precedence over the inherited `.dm-profile-bio-text` color for heading elements specifically — explicit same-element rules always beat inherited values regardless of `!important`. Net effect: headings render at `.markdown-content`'s default gold-with-shadow (`#F4E4BC`), matching `.dm-profile-bio-text`'s own color exactly — visually seamless, **zero new CSS needed**. Body text (`<p>`, `<li>` from Markdown) inherits `.dm-profile-bio-text`'s color via normal CSS inheritance (not a per-tag catch-all list), so — unlike Character's/Contact's mobile `:not(.badge)` catch-alls — **there is no `li`-omission gap here**: inheritance covers every descendant tag automatically. Confirm visually during human verification since this is a styling surface none of the generic selectors were written against, per `70-CONTEXT.md`.
- **Shop Description mobile (`Views/Shop/Details.Mobile.cshtml`, `shop-details.mobile.css`):** same shape as DM Profile mobile — `.parchment-text-muted` (carried onto the new wrapper `<div>` in place of the current `<p class="mt-2 parchment-text-muted">`) sets `color: rgba(244, 228, 188, 0.7) !important` at the div level; `.markdown-content h1..h6`'s explicit color rule still wins for headings (explicit beats inherited regardless of `!important`, since `!important` only matters when two rules target the *same* element — here the inherited value never had a competing explicit declaration on the heading itself). Body text and list items inherit the muted opacity via normal inheritance — again no `li` gap, because `.parchment-text-muted` is a single class selector whose `color` cascades to all descendants, not a per-tag enumerated list. **Zero new CSS needed**, confirm visually during human verification.

**Other inherited notes (unchanged):**

- Heading scale is `.markdown-content h1 { 24px }` → `h6 14px` (`markdown-content.css`) — applies automatically to any Bio/Description heading syntax a user types.
- Blockquote: `border-left: 3px solid rgba(255, 193, 7, 0.5)`, `padding-left: 12px`, italic — unchanged, inherited automatically.
- Links: unmodified Bootstrap default (`var(--bs-link-color)`, underlined) — unchanged.
- **Pre-wrap cleanup:** `.markdown-content { white-space: normal }` (Phase 67, `markdown-content.css`) already neutralizes the doubled-spacing mechanism the moment Bio/Description route through `Html.Markdown()`. Per `70-CONTEXT.md`, remove (not move) the 3 inline `style="white-space: pre-wrap;"` attributes: `DungeonMaster/Profile.cshtml:51`, `_ShopItemDetailsContent.cshtml:136`, `Shop/Details.Mobile.cshtml:21`. `Profile.Mobile.cshtml`'s `.dm-profile-bio-text` carries `white-space: pre-wrap` in its own CSS rule (`dm-profile.mobile.css:62`, not inline) — leave that rule in place per `70-CONTEXT.md`'s precedent (Phase 64/68's established pattern: `.markdown-content`'s own `white-space: normal` override neutralizes it regardless, and the class isn't shared with any other single-line-only element that would be harmed by removing it, so removing it is optional cleanup, not required).
- **Item Properties CSS unaffected:** the `.item-description` teaser on `Shop/Index.cshtml` and the `<small class="text-muted">` teasers on `ShopManagement/Index.cshtml` do **not** route through `Html.Markdown()`/`.markdown-content` — they display an `ExtractPlainText()`-derived plain-text substring (D-01/D-02), so none of the heading/blockquote/list typography above applies to them. Their existing typography (`.item-description`: 0.9rem/1.4 line-height, `shop.css`; table `<small>`: Bootstrap default) is unchanged by this phase.

**Required-field asterisk (new this phase — first field in this milestone to need it):**

**CSS specificity fix (editor/Preview text-color leak) — verified not applicable to any of this phase's 6 write forms, no new work required:**

Checked this phase's 3 mobile write-form CSS files (`dm-editprofile.mobile.css`, `shop-management-create.mobile.css`, `shop-management-edit.mobile.css`) directly: **none of them contain a generic `p`/`span`/`li` catch-all rule at all** — each only targets specific classes (`.form-label`, `h5`, `.form-text`, `small`), never a bare `span` or `span:not(.badge)` selector that could match CodeMirror's syntax-highlighting spans or `.editor-preview` output. This is a materially different (and simpler) pattern than Character/Contact's mobile CSS. **No leak exists on any of this phase's write forms, mobile or desktop — the already-landed tripled-class global fix is not even being tested here, since there is nothing to lose against.** Still include a live-browser spot-check of one mobile Bio editor and one mobile Description editor's text color (typing bold/heading syntax) in this phase's verification checkpoint as final confirmation, consistent with every prior phase's practice.
