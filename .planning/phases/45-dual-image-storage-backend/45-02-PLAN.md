<objective>
Widen the three image repositories and services to store both an original and a cropped byte array atomically, add distinct original/cropped read methods (with the `CroppedImageData ?? OriginalImageData` fallback resolved at the query level), and guard the two highest-risk correctness pitfalls **end-to-end through the real service call path**: (Pitfall 4) an unrelated-field edit with no new photo must not wipe the cropped image; (Pitfall 5) a genuinely new original re-upload must clear a stale crop. The critical design point this plan fixes: the Service layer must receive an explicit "a new original file was uploaded this request" signal from the controller boundary — it cannot infer it from `model.ProfilePicture`, which is never null on a no-photo-change edit (the existing bytes are round-tripped through the Edit form). Written tests-first (RED) against the target API, then implemented to green.

Purpose: Delivers the atomic dual-image storage path and independently-retrievable original/cropped values (IMAGE-03, success criteria #2 and #4) with regression coverage for Pitfall 4 (fetch-and-preserve) and Pitfall 5 (clear-stale-crop) that is actually reachable in production, not just via a direct repository unit test.
Output: Three widened repositories + interfaces + services (with an explicit new-upload signal), extended CharacterRepositoryTests/ContactRepositoryTests, a new DungeonMasterProfileRepositoryTests file, and service-level regression tests for Pitfall 4 AND Pitfall 5 across Character/Contact/DM.
