```csharp
// Returns the character's original (unmodified) profile image bytes, or null if none set.
Task<byte[]?> GetCharacterOriginalPictureAsync(int id, CancellationToken token = default);

// Returns the cropped/display image, falling back to the original when no crop was ever saved. Null if neither set.
Task<byte[]?> GetCharacterCroppedPictureAsync(int id, CancellationToken token = default);

// Atomically sets/replaces/clears BOTH columns together. Null originalImageData clears both.
Task UpdateProfileImageAsync(int characterId, byte[]? originalImageData, byte[]? croppedImageData, CancellationToken token = default);
```

Naming decision (Claude's Discretion, D-02 area): RENAME the existing `GetCharacterProfilePictureAsync` -> `GetCharacterOriginalPictureAsync` outright (and add `GetCharacterCroppedPictureAsync`), updating all call sites — cleaner long-term than keeping a "ProfilePicture" name that no longer clearly means "original". Apply the same rename convention to DM (`GetProfilePictureAsync` -> `GetOriginalPictureAsync` + add `GetCroppedPictureAsync`) and Contact (`GetContactImageAsync` -> `GetContactOriginalImageAsync` + add `GetContactCroppedImageAsync`). Controllers that call these are updated in Plan 03 — coordinate the rename so both plans agree on the final method names (record the chosen names in this plan's SUMMARY so Plan 03 uses them exactly).

Current upsert pattern (CharacterRepository.cs:129-155) — widen to two params, set BOTH props on the same tracked entity, one SaveChangesAsync (RESEARCH Pattern 2). DM's UpsertProfileImageAsync (DungeonMasterProfileRepository.cs:49-74) and Contact's UpdateProfileImageAsync follow the identical shape.

Fallback read (RESEARCH Pattern 3): resolve `c.ProfileImage.CroppedImageData ?? c.ProfileImage.OriginalImageData` INSIDE the LINQ `Select` (query-level COALESCE), not in application code. Character/Contact root at the owner DbSet (group filter applies); DM roots at `DbContext.DungeonMasterProfileImages` directly (no group filter — preserve this existing distinction).

Service call site (CharacterService.cs:66-71 / ContactService.cs:21-26) — Pitfall 4: these call the upsert on EVERY edit. DM's UpsertProfileAsync (DungeonMasterProfileService.cs:17-38) branches lazy-create vs update and threads a `removeImage` bool — widen both branches, preserving the `imageBytes != null || removeImage` gating.
