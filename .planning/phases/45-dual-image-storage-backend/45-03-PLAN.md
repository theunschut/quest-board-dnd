<objective>
Extract the copy-pasted image-upload validation (MIME allowlist, extension allowlist, 5 MB size limit) into one shared `IImageValidationService` in the Domain layer, register it, and replace all five inline validation blocks across the three upload controllers with a single call that validates both the original and (when present) the cropped file. Repoint each controller's image-serving action to the original-read service method renamed in Plan 02. **Wire the Character and Contact Edit POST actions to Plan 02's new `UpdateAsync(model, hasNewOriginalUpload, token)` overload** — passing the exact `IFormFile != null && .Length > 0` signal that already gates each action's `CopyToAsync`, so the Pitfall-5 clear-stale-crop branch built in Plan 02 is actually reached in production, not dead code. Closes the pre-existing DM-profile gap where MIME/extension were never checked (per D-03). Written tests-first (RED) for the new service, plus a controller-level regression test proving a genuine new-original Edit POST clears a stale crop.

Purpose: Delivers D-03's validation-duplication cleanup and the server-side input-validation boundary (ASVS V5), without adding any image-processing library (IMAGE-02); AND completes the Plan 02 -> Plan 03 wiring so the dual-image atomicity guarantee (a new original never coexists with a stale crop) holds through the real controller path (IMAGE-03, success criterion #2). Zero view/form changes (D-04) — the declarative `[MaxFileSize]`/`[AllowedExtensions]` ViewModel attributes stay untouched as the client-side UX layer.
Output: New `IImageValidationService` + implementation + DI registration, a new `ImageValidationServiceTests`, three controllers whose upload actions call the shared validator and whose serving actions use Plan 02's renamed read methods, the Character/Contact Edit POST actions calling the new `hasNewOriginalUpload` overload, and a controller-level integration regression test for the clear-stale-crop path.
</objective>
