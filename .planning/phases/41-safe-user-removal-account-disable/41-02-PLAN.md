<objective>
Build the backend service primitives (SAFE-02, SAFE-03) that disable and re-enable an account by reusing ASP.NET Core Identity's `LockoutEnd` mechanism — no new schema, no new column. Add the three `IIdentityService`/`IdentityService` methods and the app-wide `SecurityStampValidatorOptions` config that bounds how quickly a disabled user's active session is force-expired.

Purpose: Disable/enable is an account-wide concept modeled entirely on Identity's existing `LockoutEnd` field (D-09). Disabling sets `LockoutEnd = DateTimeOffset.MaxValue` (the permanent sentinel) and bumps the security stamp so any already-issued cookie is invalidated (D-10); enabling clears `LockoutEnd` to null (D-12). Shortening `SecurityStampValidatorOptions.ValidationInterval` from the 30-minute Identity default to 5 minutes (D-11) bounds the active-session-revocation window. This plan is the foundation the UsersController (Plan 03) and Login messaging (Plan 04) both consume.

Output: Three new `IIdentityService` methods (`DisableUserAsync`, `EnableUserAsync`, `GetLockoutEndAsync`) following the existing thin-wrapper style, plus a standalone `Configure<SecurityStampValidatorOptions>` block in `Program.cs`.
</objective>
