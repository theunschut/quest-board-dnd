### MGMT-02: Groups Index (`/platform/Group/Index`)

Card header icon: `fa-layer-group text-danger`
Card title: `Group Management`

Primary visual anchor: Groups data table

Table columns: `Name` | `Members` | `Created` | `Actions`

Action buttons per row (right-aligned, `text-end`):
- `<a class="btn btn-sm btn-info me-2">` + `fa-users me-1` → "Members" → links to Members page

## Interaction Contracts

### Form validation

- All forms use model binding (`asp-for`, `asp-validation-for`) with Bootstrap server-side validation display.
- No client-side JavaScript validation in platform views (consistent with existing admin pages).
- `[ValidateAntiForgeryToken]` on every POST action.
- `[Required]`, `[StringLength(100)]` on `GroupName` (enforced at ViewModel level).

### Table actions

- All destructive row actions (Delete group, Remove member) use `<form method="post">` inline forms — no JavaScript confirm dialogs in the members table (consistent with `Admin > Quests` delete pattern).
- Delete Group is a separate confirmation page, not a modal and not JavaScript — matches the form-based pattern declared in D-14.
- The Delete link on the Groups index is rendered only when `MemberCount == 0` (Razor condition). This prevents the page from rendering at all for non-empty groups, acting as a UI guard alongside the server-side guard.

### Navigation flow

```
/platform/Group/Index
  ├── "Create Group" → /platform/Group/Create → POST → redirect to Index
  ├── "Edit" → /platform/Group/Edit/{id} → POST → redirect to Index
  ├── "Delete" (0-member groups only) → /platform/Group/Delete/{id} → POST → redirect to Index
  └── "Members" → /platform/Group/Members/{id}
        ├── "Remove Member" inline form → POST → redirect to Members/{id}
        └── "Add to Group" form → POST → redirect to Members/{id}
```

### Error presentation

- Validation errors: `<span asp-validation-for>` inline beneath the field.
- Business errors (duplicate name, non-empty delete, duplicate membership): `TempData["Error"]` rendered as `alert-danger` at the top of `card-body`.
- Success confirmations: `TempData["Success"]` rendered as `alert-success` at the top of `card-body`.
