Deliver the change across the controller, the ViewModel, and all four view files (desktop + mobile Edit, desktop + mobile Manage) in a single plan so no view ships without its mobile twin.

Purpose: Close the standing backlog item (#61) so DMs can fix a typo or adjust rewards on a finalized quest without the destructive Open (which resets every player selection). Implements CONTEXT.md decisions D-01 through D-04.
Output: 6 modified source files; the 8 RED tests from 61-01 go GREEN.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser → QuestController.Edit (POST) | Untrusted form fields (Title, Description, TotalPlayerCount, etc.) cross into the controller and then the persistence layer. |
| browser → QuestController.Edit (GET/POST) | A finalized quest owned by another DM in the same/another group could be targeted by ID. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-61-01 | Elevation of Privilege | Edit GET/POST authorization (relaxed IsFinalized guard) | mitigate | The removed block was NOT an authorization gate — it was a state gate. The real authorization (`[Authorize(Policy = "DungeonMasterOnly")]` + in-action `IsQuestOwner(currentUser, quest.DungeonMaster) || role == GroupRole.Admin`) at Edit GET lines 155-160 and POST lines 210-215 is left fully intact and runs BEFORE the removed block. Relaxing the finalized-state check does not widen who may edit; a non-owner/non-Admin still gets Forbid(). |
| T-61-02 | Tampering | D-01 Total Player Count floor guard | mitigate | Server-side ModelState guard in Edit POST rejects any TotalPlayerCount below the current selected-Player count (computed from `existingQuest.PlayerSignups`, the persisted server state — never from the posted form), preventing an over-capacity state that would corrupt the seat/waitlist invariant. The guard reasons over server-side existing selections, so a forged form cannot bypass it. |
| T-61-03 | Tampering | Mass-assignment / over-posting on reused EditQuestViewModel | mitigate | ProposedDates and FinalizedDate are never bound from the finalized-quest edit path: the controller passes `proposedDates: null` and `updateProposedDates: false` for finalized quests, so any ProposedDates values a client injects into the POST body are discarded server-side. FinalizedDate and IsSelected are never model-bound at all (not on EditQuestViewModel/QuestViewModel), so the roster and chosen date cannot be tampered via this form. |
| T-61-04 | Tampering | Cross-tenant quest edit by ID | mitigate | QuestEntity is protected by the existing group-scoped EF Core global query filter (fail-closed, hardened in Phase 55); GetQuestWithDetailsAsync returns null for an out-of-group quest, yielding NotFound() before any edit. This phase adds no new data-access path and does not weaken the filter. |
| T-61-05 | Information Disclosure | D-02 email suppression | accept | No email is sent on finalized-quest detail edits (updateProposedDates: false gates the date-changed email off). This is the intended behavior and matches how the same fields already behave on non-finalized quests — no notification threat introduced. |
</threat_model>
