<objective>
Rate-limit the repeatable manual admin email-send buttons — "Resend Welcome/Confirmation Email" (`SendConfirmationEmail`) and `EditUser`'s email-change confirmation — to protect the Resend relay's 100/day quota from accidental button-mashing, partitioned per target user so no single recipient's inbox is spammed regardless of which admin triggers it.

Purpose: `SendConfirmationEmail` currently has no rate limit (only the anonymous `ForgotPassword`/`SetPassword` flows do, from Phase 32). Because `userId`/`Id` are POST form fields (not route values), the `[EnableRateLimiting]` attribute cannot partition by them — its policy factory runs before MVC model binding. The correct approach is a programmatic `PartitionedRateLimiter<int>` checked inside the action body after the id is bound.

Output:
- A singleton `PartitionedRateLimiter<int>` in `Program.cs`, fixed-window 3/hour, keyed `$"email-resend:{userId}"`
- `AttemptAcquire(userId)` guard as the first statement of `AdminController.SendConfirmationEmail`
- `AttemptAcquire(model.Id)` guard as the first statement inside `EditUser`'s `emailChanged && !string.IsNullOrEmpty(model.Email)` branch
- `CreateUser` left untouched (explicitly exempt, D-08)

Implements EMAIL-RATE-01, EMAIL-RATE-02, EMAIL-RATE-03, EMAIL-RATE-04.
</objective>
