<objective>
Rewrite the admin user-creation flow to be passwordless and welcome-email-driven. `AdminController.CreateUser` POST creates the account with no password (via Plan 01's passwordless `CreateAsync`), generates a password-reset token (Plan 01's `GeneratePasswordResetTokenForUserAsync`), and enqueues `WelcomeEmailJob` (Plan 02) with a callback URL to `/Account/SetPassword` — replacing the old `GenerateEmailConfirmationAsync` + `ConfirmationEmailJob` path. `SendConfirmationEmail` is retargeted the same way (becomes the "Resend welcome email" action). `CreateUserViewModel` loses its `Password` property; the `CreateUser` views lose the password input; the `Users.cshtml` row button (still gated on `EmailConfirmed == false`) is relabeled and points at the Welcome-email action.

Purpose: This is the admin-facing half of the phase and closes the loop on D-01/D-06/D-07/D-14. It fixes the transient build break Plan 01 introduced in `AdminController.CreateUser` by updating that caller to the new passwordless signature. It ALSO removes the last two callers of `IIdentityService.GenerateEmailConfirmationAsync` (in Task 2). Removing those two call sites is this plan's only role in the D-05 retirement — the actual deletion of the now-orphaned methods (`GenerateEmailConfirmationAsync` and `ConfirmEmailAsync` on both interface + implementation, plus `AccountController.ConfirmEmail`) is owned by Plan 05 in Wave 3, which runs only after BOTH this plan and Plan 03 have fully merged. This plan does NOT touch `IIdentityService.cs` / `IdentityService.cs` / `AccountController.cs` — that keeps it conflict-free with Plan 03 in the same wave (zero `files_modified` overlap). Depends on Plan 01 (signatures) and Plan 02 (WelcomeEmailJob), so it runs in Wave 2 (parallel with Plan 03 — no file overlap).
Output: `AdminController` + `CreateUserViewModel` + three views updated. No deletions of interface/service methods here. `dotnet build` and targeted integration tests green.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| admin (authenticated) → CreateUser POST | Admin-supplied email/name; account created passwordless; reset token issued |
| CreateUser/SendConfirmationEmail → email transport | Welcome link (userId + reset token) leaves the app in an email |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-32-13 | Spoofing | Welcome-link token purpose | mitigate | CreateUser + SendConfirmationEmail use GeneratePasswordResetTokenForUserAsync (ResetPassword purpose) so SetPassword can verify it; acceptance criteria forbid GenerateEmailConfirmationAsync (RESEARCH.md Pitfall 2) |
| T-32-14 | Elevation of Privilege | Passwordless account before SetPassword | mitigate | Account has null PasswordHash until SetPassword runs; PasswordSignInAsync fails gracefully (PWFLOW-03, verified in Plan 01/05). GroupRole + "Player" Identity role assigned at creation as before, but no sign-in possible without a password |
| T-32-15 | Information Disclosure | Welcome email recipient | accept | Welcome email is sent only to the admin-supplied address of the account being created; no third-party disclosure |
| T-32-16 | Access Control | CreateUser / SendConfirmationEmail actions | accept | AdminController is already protected by the AdminOnly policy (unchanged by this phase); these actions inherit that gate |
| T-32-SC | Tampering | npm/pip/cargo installs | mitigate | Zero new packages this plan |
</threat_model>
