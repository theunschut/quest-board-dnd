## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Service → Repository | Controllers (untrusted request context) reach these methods; userId/email values originate from request input upstream |
| Token issuance → email transport | Raw reset token produced here is later carried in an email link across an untrusted channel |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-32-01 | Spoofing | `GeneratePasswordResetTokenForUserAsync` (wrong token purpose) | mitigate | Method uses `GeneratePasswordResetTokenAsync` (purpose "ResetPassword") only; verified by acceptance criterion grep. A mismatched purpose would silently fail at consumption (Pitfall 2) — prevented here at the source |
| T-32-02 | Elevation of Privilege | `ConfirmEmailDirectlyAsync` sets EmailConfirmed without a token | accept | Only reachable from `SetPassword` POST AFTER a valid password-reset token was verified by `ResetPasswordAsync` (Plan 03 ordering). Standalone this method trusts its caller; the caller gates it behind token verification. No public/anonymous route reaches it directly |
| T-32-03 | Information Disclosure | Null-return on unknown userId | accept | `GeneratePasswordResetTokenForUserAsync` returns null for unknown users; the enumeration-safety decision (never reveal existence) is enforced at the controller layer (Plan 03, D-11), not here |
| T-32-SC | Tampering | npm/pip/cargo installs | mitigate | No package installs in this plan (RESEARCH.md Package Legitimacy Gate: zero new packages) |
