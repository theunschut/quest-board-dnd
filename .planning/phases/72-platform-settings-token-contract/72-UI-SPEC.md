# Phase 72 — UI Design Contract

> Visual and interaction contract for two sibling Omphalos settings pages: (1) the SuperAdmin-only instance-wide "Integrations" page (`/platform/Integrations`) and (2) the **new** Group Admin-only per-group override page living alongside the group-scoped Admin area. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

**Update (2026-07-11):** `72-CONTEXT.md` was revised to add the group-override surface (D-07–D-10) on top of a storage-schema change (fixed-column singleton row to a generic key-value table with cascade fallback). The instance-wide page's design below (Design System through Registry Safety) is **unchanged and re-confirmed** — the storage change is invisible at the UI layer for that page. The new **"Group-Override Settings Page"** section further down is net-new scope added by this revision; everything above it was approved before the revision and is carried forward as-is, not re-litigated.

**Note on scope:** Phase 72 is 100% server-rendered Razor (ASP.NET Core MVC + Bootstrap, no SPA/component framework), so the shadcn initialization gate in the base template does not apply. Every visual decision below is either pulled directly from CONTEXT.md's locked decisions (D-01–D-10) or matched against a verified, currently-in-use precedent elsewhere in this codebase — no new visual language is introduced. No user prompts were needed; all design-contract questions were answerable from upstream artifacts + direct codebase inspection.

## Typography

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 16px | 400 | 1.5 |
| Label | 16px | 600 | 1.5 |
| Heading | 20px | 600 | 1.2 |
| Display | 32px | 600 | 1.2 |

- **Body (16px/400/1.5):** paragraph copy, table/badge-adjacent text, `.modern-card p`/`.form-text` helper copy (`site.css` root `body` uses Bootstrap's default `1rem`/`1.5` — confirmed, no override).
- **Label (16px/600/1.5):** `.form-label` — `modern-card .form-label` explicitly forces `font-weight: 600` (`site.css:1125-1131`); applies to "Omphalos URL", "Shared Secret", "Integration Enabled" labels.
- **Heading (20px/600/1.2):** Bootstrap default `<h5>` size — used for any sub-section heading within the card if one is needed (none required by this phase's single-form layout, declared for consistency with the rest of the design system, e.g. `EditUser.cshtml`'s "Password Management" `<h5>`).
- **Display (32px/600/1.2):** Bootstrap default `<h2>` size — the card title itself (`<h2 class="mb-0">Integrations</h2>`), matching `Group/Index.cshtml`/`Users/Index.cshtml`/`EditUser.cshtml` exactly. Color/text-shadow (parchment `#F4E4BC` + dark shadow) is applied automatically by `.modern-card h1-h6` (`site.css:1094-1105`) — no additional styling needed.
- **Exception — small/meta text (14px, 400, 1.5):** `.form-text` helper lines and secret-status caption text ("•••• Configured" / "Not configured yet") use Bootstrap's `0.875rem` small-text convention at weight 400 — confirmed by `.modern-card .form-text` (`site.css:1133-1139`), which explicitly overrides the global `.form-text` weight-600 rule (`site.css:602-607`) down to `font-weight: 400` inside every card on this page. Status badges (`bg-success`/`bg-secondary`, e.g. "Configured"/"Not configured") are excluded from this row's weight budget: `.modern-card .badge` (`site.css:1205-1209`) sets `font-weight: 500` as pre-existing component chrome — the same treatment already applied to every `.modern-card .btn` (`site.css:1212-1217`, also weight 500) site-wide, independent of the 2-weight running-text budget (400/600) this contract declares.

## Color

This app does not use a flat single-hex 60/30/10 palette — it layers Bootstrap semantic colors under a photographic-background + glassmorphism system. The table below maps the standard 60/30/10 contract onto that reality, using only colors already established and in production use elsewhere in the Platform area (no new hex values introduced).

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#8B4513` (saddle brown, representative of the fixed background image + `.navbar` gradient `rgba(139,69,19,.5)→rgba(205,133,63,.5)`, `site.css:501-511`) | Page backdrop, navbar chrome — pre-existing site-wide, unmodified by this phase |
| Secondary (30%) | `#F4E4BC` (parchment) on `rgba(255,255,255,0.15)` blurred glass (`.modern-card`, `site.css:1045-1052`) + `rgba(255,255,255,0.95)` opaque fields (`.modern-card .form-control`, `site.css:1142-1150`) | The Integrations card surface itself: card body background, `Omphalos URL`/`Shared Secret` input backgrounds |
| Accent (10%) | `#ffc107` (Bootstrap `btn-warning`) | Reserved for exactly one element: the **"Save Settings"** submit button. Matches `Group/Edit.cshtml`'s `btn-warning` "Save Changes" — the closest architectural analog (same Platform area, same "modify an existing config entity" semantics), and the majority pattern across the codebase's "Edit existing entity" forms (`Group/Edit`, `Contacts/Edit`, `Quest/Edit`, `Characters/Edit`, `DungeonMaster/EditProfile` all use `btn-warning`, vs. only `EditUser`/`Account/Edit` using `btn-success`). The existing global gold focus-ring glow (`.form-control:focus`, `site.css:40-43`) applies automatically to this page's inputs — no additional work, not counted against the reserved-for list. |
| Destructive | `#dc3545` (Bootstrap `btn-danger` / `.modern-card .alert-danger`) | Reserved for exactly one element: the **"Generate Secret"** button — `btn-danger` (filled) + `fa-key`, directly matching `Users/Index.cshtml:86`'s "Disable" button precedent (`btn btn-sm btn-danger`, filled — same "consequence-explaining destructive action" semantics, and consistent with root `CLAUDE.md`'s "filled colored buttons (not outline)" UI rule) — plus its `confirm()` dialog copy (see Copywriting Contract). This is a credential-rotation action, not a data-delete action, but it immediately invalidates the previously-working secret, which is the same risk class the codebase already treats as filled `btn-danger`. |

**Accent reserved for:** the "Save Settings" submit button only. Not applied decoratively to the header icon, badges, or the toggle checkbox — those use their own established semantic colors below.

**Status badges (semantic, not part of the accent/destructive budget):** `bg-success` = "Configured" (secret) / green toggle icon for "Enabled"; `bg-secondary` = "Not configured" — directly matching `Users/Index.cshtml`'s existing `Confirmed`/`Unconfirmed`, `Active`/`Disabled` badge convention (`bg-success`/`bg-secondary`, never a new color pair).

**Explicitly not used on this page:** gold/`#ffc107` beyond the one reserved Save-button use — the app's other gold accents (navbar brand glow, shop UI) belong to a different visual register (Cinzel hero surfaces) and would be inconsistent with the `modern-card` form pattern this page follows.

### Cascade Communication (the core new design problem)

This is the one genuinely new interaction this page introduces: communicating whether the group is currently running its **own override** or **falling back to the instance-wide default** — to a Group Admin who may not know an instance-wide default exists at all (per objective #3). A single status banner at the top of the card, above the form fields, resolves this in three states:

| State | Badge | Icon | Banner Copy |
|-------|-------|------|--------------|
| Override active | `bg-success` "Override Active" | `fa-circle-check` | **"This group has its own Omphalos configuration, set below. It takes priority over the instance-wide default — even if a SuperAdmin changes the platform default later, this group keeps using its own settings until you clear the override."** |
| Inherited (instance default configured) | `bg-info text-dark` "Inherited from Instance Default" | `fa-arrow-turn-down` | **"This group has no override — it's currently using the Omphalos configuration set platform-wide by a SuperAdmin. Fill in the fields below and save to give this group its own configuration instead."** |
| Not configured anywhere | `bg-secondary` "Not Configured" | `fa-circle-exclamation` | **"Omphalos integration isn't configured for this group or platform-wide, so DM Tool/Session Notes links won't appear for this group yet. Set the fields below to configure it for this group, or ask a SuperAdmin to set the platform-wide default."** |

Design rules for this banner:
- It is a plain `alert`-free `text-muted` paragraph directly under the card header (not a Bootstrap `.alert` box) — this app has no `.alert` precedent inside `modern-card` bodies for informational (non-error) banners; the badge carries the visual weight, the paragraph carries the explanation. This matches the instance-wide page's existing "Configure the connection..." intro-line treatment (plain paragraph, `text-muted`, no card-within-card).
- The badge is the single element on this page allowed to use `bg-info` — reserved for exactly the "Inherited" cascade state, nowhere else on either page.
- **Never reveal the instance-wide default's actual URL or secret** on this page, even to a Group Admin looking at the "Inherited" state — only whether one exists and is enabled. Group Admins do not have visibility into platform-wide configuration values; only whether their group is currently covered by one. This mirrors the instance-wide page's own masked-secret discipline (Pattern 3), extended across the scope boundary.
- The three-state banner assumes a corresponding boolean/enum the group-override page's GET action must resolve (does a group override row-set exist; does an instance-wide default row-set exist and is it enabled) — a data/service concern for planning, not this UI-SPEC, but the visual contract above is unambiguous about what must be knowable to render it.

### Form Fields & Atomic-Override Model

Per D-08's explicit note that "the UI is expected to treat the three [settings] as one unit per scope rather than exposing field-by-field overrides" — this page does **not** offer per-field "override just the URL" controls. The form is a single atomic unit: either the group has no override (fields render blank/unchecked, all three inherited together), or it has one (fields render prefilled with the group's own saved values, all three overridden together).

| Field | Behavior (identical widget/label to instance-wide page) | New behavior specific to this page |
|-------|-----------------------------------------------------------|-------------------------------------|
| Omphalos URL | Same `[Url]`-validated text input, same label/placeholder/helper text as the instance-wide page | When no override exists: placeholder text reads `"Not set — inherits the instance-wide URL"` (italic, `text-muted`) instead of the instance-wide page's generic example placeholder |
| Shared Secret | Same masked `type="password"` input, same blank-preserve guard (Pattern 3), same "Generate Secret" button/flow, same secret-status caption pattern | Secret-status caption gains a third state to match the cascade banner: **configured (override)** → `"•••• Configured for this group — leave blank to keep the current override secret."` / **not configured (inheriting)** → `"No group-specific secret set — inherits the instance-wide secret, if one is configured."` |
| Integration Enabled | Same plain `form-check` checkbox (never `form-switch`, per RESEARCH.md Pattern 5), same icon/label | When no override exists, the checkbox renders **unchecked** but with a `text-muted` helper line directly beneath it: `"Unchecked here means 'no group override' — the group's actual enabled/disabled state currently comes from the instance-wide default shown above, not from this checkbox."` This prevents the unchecked-by-default state from being misread as "this group has explicitly disabled the integration." |
