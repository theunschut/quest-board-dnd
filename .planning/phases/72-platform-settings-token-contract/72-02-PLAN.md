<objective>
Write the canonical HMAC-SHA256 token-format contract that Phase 73 (Quest Board, signs) and Phase 74 (Omphalos, verifies) both implement against. This phase writes the contract down; it implements no token code. This is the "+ Token Contract" half of the phase and is independent of the settings-storage redesign — it carries over largely as the deleted 72-05 plan intended.

    The document must contain these sections, unambiguous enough that Phase 73 and Phase 74 can be built independently:

    1. Purpose and scope — one signed, short-lived redirect token; both sides are first-party under common operational control; this is not JWT and deliberately adds no JWT library.

    2. Payload field list and types (exact, per D-01/D-02): `nonce` (string, a GUID such as `Guid.NewGuid().ToString()`; uniqueness is the only requirement — Phase 74 keys replay tracking on it; unpredictability NOT required); `userId` (int — Quest Board `UserEntity.Id`; NEVER Name/UserName/Email, per TOKEN-04 and the Phase 34.3 display-name precedent); `questId` (int); `questTitle` (string, free text — safe inside JSON, the reason for moving off delimited concatenation); `questDate` (string, ISO-8601 round-trip "o" format, UTC — pass-through for Omphalos session naming, not used for security logic); `expiry` (long, Unix seconds UTC — NOT an ISO string, avoids timezone-parse ambiguity for the one security-relevant field; 300-second / 5-minute TTL per TOKEN-03).

    3. Encoding and wire format: payload serialized to JSON then UTF-8 bytes; token is `Base64UrlEncode(payloadBytes) + "." + Base64UrlEncode(signatureBytes)`; name `Microsoft.AspNetCore.WebUtilities.WebEncoders.Base64UrlEncode`/`Base64UrlDecode` as the exact encoding API both sides must use (do not hand-roll base64url).

    4. Signing sequence (Phase 73): serialize payload object to JSON UTF-8 bytes; compute `HMACSHA256.HashData(UTF8(secret), payloadBytes)`; Base64URL-encode payload bytes and signature bytes; join with a literal `.`.

    5. Verification sequence (Phase 74) — the byte-exact rule that makes JSON safe (RESEARCH.md Pitfall 1): split on `.`; Base64URL-decode BOTH segments back to raw bytes; recompute `HMACSHA256.HashData(UTF8(secret), decodedPayloadBytes)`; compare to the decoded signature bytes using `System.Security.Cryptography.CryptographicOperations.FixedTimeEquals` (constant-time — never `==`/`SequenceEqual`); ONLY after the signature check passes, `JsonSerializer.Deserialize` the same decoded payload bytes; then check `expiry` against current time with a small clock-skew allowance (±30s) and check `nonce` against Phase 74's used-token store for replay. State explicitly: the verifier NEVER re-serializes parsed fields to recompute the HMAC input — it operates only on the exact transmitted bytes (this is the JWT structural pattern, applied without a JWT library).

    6. Shared-secret provisioning: the secret is generated in Quest Board (server-side CSPRNG) and copied manually by an operator into Omphalos's configuration (D-03); there is no server-to-server sync channel between the apps (D-04, informational — satisfied by absence). Each Omphalos deployment verifies only against its own single configured secret regardless of how many Quest Board groups point at it.

    7. Cross-repo delivery note: this file is the single canonical copy (D-06); Phase 74's PR references it by link and does not duplicate it.

    Do NOT include runnable production code files — worked call-sequence snippets inside this markdown design doc are acceptable and expected, but no `.cs` file is created by this plan.
