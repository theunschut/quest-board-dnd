**Existing two-phase logic to move (source of the new method)** — `QuestController.CreateFollowUp` lines 849-891 (see below). The domain-layer analog method (`CreateFollowUpQuestAsync`, same file lines 174-224) shows the target conventions:
- Guard clauses throw `InvalidOperationException` with a user-facing message (caught by the controller and surfaced via `TempData["Error"]`) — reuse this same exception type/convention for the new method's failure paths, so the controller's existing `catch (InvalidOperationException ex)` handling (line 858) keeps working unchanged.
- `async Task<int>` return type pattern (returns the new quest's Id) is used by `CreateFollowUpQuestAsync`; the new combined method should return the same shape since it's a superset of that method's responsibility (create then immediately apply date/title/description).
- Async calls always pass `CancellationToken token = default` last-position parameter, forwarded to every repository call — see lines 176, 202, 220.

**Rollback pattern to preserve when moving into the service** (currently in controller, `QuestController.cs` lines 868-891):
```csharp
// Apply the proposed dates and title/description edits from the form
// (CreateFollowUpQuestAsync creates the quest shell without dates; dates come from the form)
try
{
    await questService.UpdateQuestPropertiesWithNotificationsAsync(
        newQuestId,
        viewModel.Title,
        viewModel.Description,
        viewModel.ChallengeRating,
        viewModel.TotalPlayerCount,
        viewModel.DungeonMasterSession,
        updateProposedDates: true,
        viewModel.ProposedDates,
        token);
}
catch
{
    // Roll back the shell quest so the unique FK is freed and retries are possible
    var orphan = await questService.GetQuestWithDetailsAsync(newQuestId, token);
    if (orphan != null)
        await questService.RemoveAsync(orphan);
    throw;
}
```
When extracted into `QuestService.CreateFollowUpQuestWithDetailsAsync(...)`, this becomes internal calls to `CreateFollowUpQuestAsync(...)` followed directly by `UpdateQuestPropertiesWithNotificationsAsync(...)` (both already exist on `QuestService`/`repository` — no new repository methods needed), with the same try/catch-and-rollback-via-`RemoveAsync` shape, just operating on `this`/`repository` instead of the injected `questService` proxy.

**Error handling pattern used elsewhere in this file** (line 87-91, `RemoveAsync` override):
```csharp
public override async Task RemoveAsync(Quest model, CancellationToken token = default)
{
    var quest = await repository.GetQuestWithManageDetailsAsync(model.Id, token);
    if (quest == null) return;
    // Manual cleanup required since Quest->PlayerSignup is NoAction to avoid cascade cycles
    ...
```

Null checks return early rather than throwing where the caller doesn't need to know ("nothing to do"); throw `InvalidOperationException` only where the caller must react (see `CreateFollowUpQuestAsync` lines 177-186).

### `QuestBoard.Service/Controllers/QuestBoard/QuestController.cs` — `CreateFollowUp` (controller, request-response) — thinning target

**Analog:** itself, before/after — plus other already-thin actions in the same file for the target shape.

**Current state (lines 802-894)** — mixes real business logic (two-phase create+rollback, lines 849-891) with legitimate MVC concerns (auth guard, `ModelState` validation, `TempData` messaging). Per D-01, only the two-phase logic (lines 849-891) is "real business logic" and should move to `QuestService`; the surrounding guard clauses (lines 812-847, DM/Admin authorization check, follow-up-already-exists check, not-finalized check, `ModelState`/`ProposedDates` validation) are legitimately MVC-layer concerns (auth, input validation, view re-render) and stay in the controller per D-01's "leave thin actions as-is" principle.

**Post-refactor shape should mirror this existing already-thin action in the same controller** (`Open`, or similar simple-delegate action elsewhere in `QuestController.cs`) — parse input, single service call, redirect/TempData. The refactored `CreateFollowUp` body (after extraction) becomes:
```csharp
try
{
    var newQuestId = await questService.CreateFollowUpQuestWithDetailsAsync(
        id, viewModel.Title, viewModel.Description, viewModel.ChallengeRating,
        viewModel.TotalPlayerCount, viewModel.DungeonMasterSession, viewModel.ProposedDates, token);

    return RedirectToAction("Manage", new { id = newQuestId });
}
catch (InvalidOperationException ex)
{
    TempData["Error"] = ex.Message;
    ViewBag.PreApprovedPlayers = original.PlayerSignups
        .Where(ps => ps.IsSelected)
        .Select(ps => new { ps.Player.Name })
        .ToList();
    return View(viewModel);
}
```

**Auth pattern to keep in controller (unchanged)** (lines 799-816):
```csharp
[HttpPost]
[ValidateAntiForgeryToken]
[Authorize(Policy = "DungeonMasterOnly")]
public async Task<IActionResult> CreateFollowUp(int id, FollowUpQuestViewModel viewModel, CancellationToken token = default)
{
    var original = await questService.GetQuestWithDetailsAsync(id, token);
    if (original == null)
        return NotFound();

    var currentUser = await userService.GetUserAsync(User);
    if (currentUser == null)

        return Forbid();
```
