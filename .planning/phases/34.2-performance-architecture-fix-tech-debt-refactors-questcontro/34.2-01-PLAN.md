<objective>
Consolidate the fragile two-phase follow-up-quest create+rollback logic (currently living in `QuestController.CreateFollowUp`, lines ~849–891) into a single `QuestService.CreateFollowUpQuestWithDetailsAsync` method, per D-01 (extract real business logic into the Domain service, no controller split) and D-04 (this is CONCERNS.md's specified fix and the concrete first case of D-01). Also fix the trivial `DateTime.Now` → `DateTime.UtcNow` tech-debt item in `ShopSeedService`.

This is the highest-blast-radius extraction in the phase; it is isolated to its own plan so the executor has full context budget for it. The controller keeps its legitimate MVC concerns (auth guard, existing-follow-up check, not-finalized check, ModelState/ProposedDates validation) — only the create-then-update-with-rollback sequence moves to the service.

Purpose: Move fragile transactional logic out of the controller into the Domain layer where it belongs; make the rollback path a single testable unit (Plan 05 adds the failure-path test).
Output: A new `QuestService` method + interface contract, a thinned controller action, and a two-line UTC fix.
</objective>

    `Task<int> CreateFollowUpQuestWithDetailsAsync(int originalQuestId, string title, string description, int challengeRating, int totalPlayerCount, bool dungeonMasterSession, IList<DateTime> proposedDates, CancellationToken token = default)`.

    Implement it by composing the two existing methods (do NOT duplicate their internals): first call `await CreateFollowUpQuestAsync(originalQuestId, token)` to create the shell + import selected players and obtain `newQuestId`. Then, inside a `try`, call `await UpdateQuestPropertiesWithNotificationsAsync(newQuestId, title, description, challengeRating, totalPlayerCount, dungeonMasterSession, updateProposedDates: true, proposedDates, token)`. In the `catch` (catch-all, `catch`), re-fetch the shell via `await GetQuestWithDetailsAsync(newQuestId, token)` and, if non-null, `await RemoveAsync(orphan, token)`; then `throw;` (re-raise the original exception).     Per CONCERNS.md's "add explicit logging on rollback failure so errors don't silently disappear": wrap the rollback `RemoveAsync` call so a failure during cleanup does not mask the original exception — attempt the removal, and if removal itself throws, swallow that secondary exception (it must not replace the primary one) but ensure the primary exception still propagates via `throw;`. QuestService has no injected `ILogger`; do NOT add a new constructor dependency just for logging — instead structure the catch so the primary exception always re-throws even if rollback fails. (A logger seam is out of scope for this method; the controller no longer owns this path.)

    Add the matching signature to `IQuestService` with a plain XML `/// <summary>` doc comment describing: creates a follow-up from a finalized original quest, applies the given dates/title/description, imports selected players, and rolls back the shell if the update fails; returns the new quest Id. Do NOT include any `(D-xx)`, phase number, or requirement-ID text in the doc comment (D-06 hygiene — see PATTERNS.md research correction #8).

    Do NOT change `CreateFollowUpQuestAsync`, `UpdateQuestPropertiesWithNotificationsAsync`, or `RemoveAsync` — only add the new composing method.
