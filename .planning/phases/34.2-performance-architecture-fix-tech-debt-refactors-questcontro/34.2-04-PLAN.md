## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Hangfire job → scoped services / group context | Every job must set the correct group context before any repository call; the helper centralizes this so it cannot be forgotten |
| controller → group-scoped entity load | The (deferred) Forbid() boundary; today the EF query filter is the enforced boundary |
| app → Resend SMTP relay | Single external dependency for all mail; failure handling is via bounded job retry |

| T-34.2-41 | Information Disclosure | Cross-tenant data at the controller/API boundary (ASVS V4 Access Control) | accept (documented) | The EF Global Query Filter is the working safety net today; per-controller `Forbid()` defense-in-depth is DEFERRED and documented (D-06), not implemented. Documented so the residual risk is explicit and tracked for a future phase. |
| T-34.2-42 | Tampering | Hangfire job group-context setup | mitigate | `HangfireJobHelper.RunInScopeAsync` guarantees `SetGroupId(groupId)` runs before any repository resolution (when a group is specified), removing the "forgot to set context" foot-gun; the null-group path is explicit for the cross-group sweep. |
| T-34.2-43 | Denial of Service / Availability | Resend SMTP single point of failure | accept (documented) + mitigate | Documented as a known SPOF; the practical mitigation is Hangfire's bounded retry (AutomaticRetryAttribute from Plan 03). A secondary relay is deferred. |
