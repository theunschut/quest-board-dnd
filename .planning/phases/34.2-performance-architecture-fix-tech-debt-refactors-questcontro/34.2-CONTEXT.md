<decisions>
## Implementation Decisions

### Controller Refactor Approach

- **D-01:** No physical split of `QuestController` or `AdminController` into multiple controller classes. Instead, extract business logic into the appropriate Domain service (`QuestService` for `QuestController`; the relevant user/admin service for `AdminController`) — but **only where an action currently mixes real business logic into the controller**. If an action is already thin (parses input, calls a service, returns a view/redirect), leave it as-is; don't force extraction where there's nothing to extract. This matches `CONCERNS.md`'s own stated fix approach ("reduce quest controller to <500 lines by delegating business logic entirely to services") more directly than the "consider sub-controllers" suggestion, and better fits this project's Service→Domain→Repository architecture (controllers are supposed to be thin already).
- **D-02:** Add extension methods / static helper methods to the Service project to reduce repeated MVC-layer boilerplate — specifically the ModelState-invalid early-return pattern and the TempData success/error + redirect pattern seen throughout `QuestController` and `AdminController`. Scope this to the two controllers this phase actually touches, not a codebase-wide helper-library initiative.
- **D-03:** `AdminController`'s Resend API / `EmailStats` logic (`GetResendStatsAsync` and related) is **excluded** from this phase's refactor scope — Phase 34.1 already extracts that into a dedicated `ResendStatsClient` class. This phase's `AdminController` work covers only user-management actions (`Users`, `PromoteToAdmin`, `DemoteFromAdmin`, `PromoteToDM`, `DemoteToPlayer`, `CreateUser`, `EditUser`, `ResetPassword`, `DeleteUser`, `SendConfirmationEmail`) and quest-admin actions (`Quests`, `DeleteQuest`).
- **D-04:** The fragile two-phase follow-up-quest create+rollback logic (`QuestController.CreateFollowUp`, lines 854–891) moves into a single `QuestService` method (e.g. `CreateFollowUpQuestWithDetailsAsync`) that handles creation and update in one transaction. This is `CONCERNS.md`'s specified fix for this item and the concrete first case of D-01's extraction principle.

### Cross-Phase Execution Order

- **D-05:** Execution order MUST be **Phase 34 → Phase 34.1 → Phase 34.2**. Phase 34's comment-cleanup already touches `Program.cs`, `AdminController.cs`, and `QuestController.cs`; Phase 34.1 further restructures `AdminController.cs` (Resend extraction) and `Program.cs` (startup validation) before this phase's broader controller/service work should begin. `ROADMAP.md`'s Phase 34.2 entry's "Depends on" field must be updated to list **both** Phase 34 and Phase 34.1 (it currently lists only Phase 34) so the executor doesn't run this out of order.

### Tenant-Isolation Defense-in-Depth (Forbid() checks)

- **D-06:** The "No Tenant Isolation Enforcement at API Boundary" item (`CONCERNS.md` Scaling Limits) is **documentation-only** in this phase — no `Forbid()` code checks are implemented. Add a note (in `ARCHITECTURE.md` or `CONCERNS.md`, planner's discretion on location) explaining that the EF Core Global Query Filter already provides working cross-tenant protection today, and that per-controller `Forbid()` defense-in-depth checks are a deferred hardening idea for a future phase, given the footprint rivals the `QuestController` refactor itself (per Phase 34's RESEARCH.md finding).

### Performance Fixes

- **D-07:** Composite index on `Quests(IsFinalized, FinalizedDate)` — implement unconditionally via an EF Core migration, regardless of current measured query time. This is the actual scale-readiness lever for future SaaS growth (the query-side bottleneck, not job count — see D-09).
- **D-08:** Shop Item Queries projection fix — exclude image BLOB loading from list views (only load images for detail views), per `CONCERNS.md`'s suggested `.ThenInclude()` scoping approach. Implement unconditionally.
- **D-09:** The "Hangfire Job Queue Not Filtered by Group" batching item (`CONCERNS.md` Performance Bottlenecks) is **skipped** — no `SessionReminderBatchJob` or job-consolidation code is implemented. The current fetch-once + N-small-jobs fan-out pattern (`DailyReminderJob` queries all groups once via `GetQuestsForTomorrowAllGroupsAsync`, then enqueues one small `SessionReminderJob` per quest) is the correct, scalable Hangfire pattern — parallel worker processing, isolated per-job retry, and horizontal scaling via additional Hangfire server instances, none of which a monolithic batch job would get for free. Add a documentation note (near D-06's note, or in `CONCERNS.md`) that this is a monitored future scaling path per `CONCERNS.md`'s own "if queue >1000 items" language — not a current problem.
- **D-10:** Application-level data caching (in-process caching of frequently-read data like shop items or quest lists to skip DB round-trips) was discussed and **explicitly declined** — not implemented, and not carried forward as a deferred/backlog idea (user's explicit request: "leave it"). Rationale discussed: (1) staleness would undermine the quest board's core value — a DM posts a quest and players must see it immediately, not after a cache TTL; (2) multi-tenant cache-key risk — an improperly `GroupId`-scoped cache key could leak one group's data to another group's request, a new class of cross-tenant leak this app has been specifically engineered against since Phases 27–30; (3) the actual measured perf pain point (Shop Item BLOB loading) is already fixed by D-08's projection change, not a caching problem.
- **D-11:** Digest batching (EMAIL-04/REMIND-02) reconfirmed deferred per Phase 34's D-02, with an important clarification from this discussion: it's **structurally impossible** under current quest-signup rules (a player cannot be signed up for two quests scheduled the same day, so the multi-quest-same-day scenario digest batching would address cannot occur), not merely rare/low-priority. Future phases should not treat this as "revisit when it becomes common" — it can only become relevant if the one-quest-per-day signup constraint itself changes.

### Reusable Assets
- `EmailPreviewController` is already separated from `AdminController` per `CONCERNS.md`'s own note — precedent that partial extraction from these large controllers is an established, low-risk pattern here.
