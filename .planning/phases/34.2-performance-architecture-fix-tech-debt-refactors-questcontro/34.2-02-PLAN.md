<objective>
Introduce the NET-NEW MVC-boilerplate helper pattern (D-02) — a `ControllerExtensions` static class in `QuestBoard.Service/Extensions/` — to reduce the two repeated boilerplate shapes seen throughout `QuestController` and `AdminController`: (1) the ModelState-invalid early-return, and (2) the TempData success/error message + `RedirectToAction`. Apply the helper to `AdminController`'s user-management and quest-admin actions (D-01 thin-controller principle).

Per 34.2-CONTEXT.md D-02 and 34.2-PATTERNS.md: this is a genuinely net-new pattern — no existing extension-method convention for `Controller`/`ModelState`/`TempData` exists in the codebase (only `ConfigurationDebugExtensions`, an `IConfiguration` extension). The plan introduces it, following `ConfigurationDebugExtensions`'s file-organization convention only. Scope is strictly the two in-scope controllers, not a codebase-wide helper library.

**Scope boundary (D-03 + D-05, MANDATORY):** Per D-05 the execution order is Phase 34 → 34.1 → 34.2, so this plan assumes Phase 34.1 has already extracted the Resend fetch loop from `AdminController` into `ResendStatsClient`. Per D-03 this plan does NOT touch `AdminController.GetResendStatsAsync` / `EmailStats`. Do not refactor, move, or wrap any Resend/EmailStats code. `QuestController` is owned by Plan 01 this wave — this plan does NOT edit `QuestController.cs` (only `AdminController.cs` + the new extensions file) to keep the wave's file sets disjoint. The helper is written to serve both controllers; a later touch to QuestController call sites is deferred (the helper existing is the D-02 deliverable).

Purpose: Reduce MVC-layer boilerplate duplication and move AdminController closer to the thin-controller ideal without a physical controller split (D-01).
Output: A new `ControllerExtensions` class and its application to AdminController's user/quest-admin actions.
</objective>
