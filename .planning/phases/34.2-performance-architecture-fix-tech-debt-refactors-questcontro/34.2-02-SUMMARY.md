## Files Created/Modified
- `QuestBoard.Service/Extensions/ControllerExtensions.cs` - New static class: `RedirectWithMessage(action, tempDataKey, message)`, `RedirectWithSuccess`/`RedirectWithError` convenience wrappers, `TryReturnInvalidModel(model, out result)`
- `QuestBoard.Service/Controllers/Admin/AdminController.cs` - `CreateUser`, `EditUser`, `ResetPassword`, `SendConfirmationEmail` now call the shared helper for the TempData+redirect and ModelState-guard shapes

## Decisions Made
- `RedirectWithMessage` keeps `tempDataKey` as an explicit parameter rather than only offering `Success`/`Error` convenience wrappers, because `ResetPassword` uses a third key (`"SuccessMessage"`) that predates this refactor ŌĆö preserving exact existing behavior (no unintended TempData key rename) took priority over forcing all call sites onto the two-key convention.
- `TryReturnInvalidModel` was applied only to `CreateUser`, where the ModelState check is a clean early-return at the top of the action. `EditUser` and `ResetPassword` wrap their entire success path inside `if (ModelState.IsValid) { ... }` (inverted condition, not an early return) ŌĆö converting those would require restructuring control flow beyond a boilerplate swap, so they were left as-is per D-01's "don't force extraction where there's nothing to extract" principle.
- The `activeGroupContext.ActiveGroupId == null` guards (`Users`, `PromoteToAdmin`, `DemoteFromAdmin`, `PromoteToDM`, `DemoteToPlayer`, `CreateUser`) were left untouched ŌĆö they redirect to `GroupPicker` (a different controller) and set no TempData message, which doesn't match either of the two shapes this helper targets.
