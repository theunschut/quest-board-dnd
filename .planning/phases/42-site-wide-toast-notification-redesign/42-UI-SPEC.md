# Phase 42 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

This phase is **not** open visual design exploration. It codifies an already-implemented, already-shipped Bootstrap toast pattern (`QuestBoard.Service/Views/Shop/Index.cshtml:450-489`) into a single shared partial (`_Toasts.cshtml`) used site-wide, plus one net-new toast type (Info). Every value below either matches the Shop reference exactly (byte-for-byte markup/classes) or was explicitly locked in `42-CONTEXT.md` (D-01 through D-13). There is no discretionary visual design in this phase beyond the two items CONTEXT.md already flagged and leaned on.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none — no `components.json`; project is server-rendered ASP.NET Core 10 MVC + Razor views, not React/Next.js/Vite. shadcn gate not applicable. |
| Preset | not applicable |

## Spacing Scale

Declared values (must be multiples of 4 — Bootstrap's default rem-based spacing is used as-is, matching the existing reference):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px (`me-2`≈0.5rem icon gap) | Icon-to-label gap inside toast header (`me-2`, FontAwesome icons) |
| sm | 8px | `toast-body` internal text spacing (default Bootstrap `.toast-body` padding: `0.75rem` ≈ 12px — see exception below) |
| md | 16px | Toast container edge padding (`p-3` = 1rem = 16px) — matches Shop exactly |
| lg | 24px | Not used within toast component itself |
| xl | 32px | Not used within toast component itself |

Exceptions: Bootstrap's default `.toast-body` padding is `0.75rem` (12px, not a multiple of 4) and `.toast-header` padding is `0.5rem 0.75rem` — these are Bootstrap framework defaults, unchanged from the existing Shop reference, and are NOT to be overridden. Do not introduce custom padding on `.toast`, `.toast-header`, or `.toast-body` — inherit Bootstrap's defaults exactly as Shop does today.

## Typography

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Toast header label | Bootstrap default (~1rem/16px, via `<strong>`) | Bold (`<strong>` = 700) | Bootstrap default (~1.5) |
| Toast body message | Bootstrap default (~1rem/16px) | Regular (400) | Bootstrap default (~1.5) |
| GoldReceived amount (`+X gp`) | `fs-5` (~1.25rem/20px) | Bold (`<strong>` = 700) | Bootstrap default |

No new font sizes or weights are introduced. This is an exact port of Shop's existing toast typography — do not restyle text within `.toast-header` or `.toast-body`.

## Color

Toast header bar uses **solid** Bootstrap semantic background classes per D-11 — not a subtle left-border-accent style. Each type maps 1:1 to a Bootstrap utility class already in use elsewhere in the app (buttons, alerts), so no new hex values are introduced.

| Role | Value | Usage |
|------|-------|-------|
| Success | `bg-success text-white` (Bootstrap `#198754`) | Toast header bar — Success type only |
| Error | `bg-danger text-white` (Bootstrap `#dc3545`) | Toast header bar — Error type only |
| Warning | `bg-warning text-dark` (Bootstrap `#ffc107`, dark text for contrast) | Toast header bar — Warning type only. **Not** the same visual instance as Shop's yellow "Mystical Merchant" novelty toast (D-06) — that stays untouched and out of scope, styled via its own `.merchant-toast` class. |
| Info (new) | `bg-info text-white` (Bootstrap `#0dcaf0`/`#0dcaf0` — verify sufficient contrast; use `text-white` to match the other 3 types' white-on-solid pattern) | Toast header bar — Info type only (D-05) |
| GoldReceived (bespoke, not a generic type) | `bg-warning text-dark` (unchanged from Shop's existing markup) | Shop-specific "+X gp" celebratory toast header only — kept separate from the generic Info/Success/Warning system per Claude's Discretion lean |

Accent reserved for: N/A — this is not a 60/30/10 dominant-surface color contract. Toast headers are the only colored surface in scope, and color choice is dictated entirely by semantic type (Success/Error/Warning/Info), not by a dominant/accent hierarchy. The rest of the app's 60/30/10 palette (fantasy parchment background, `modern-card` translucent surfaces, `#8B4513` brown accents) is unaffected by this phase and out of scope.

Destructive: N/A — no new destructive-action UI in this phase; `bg-danger` above is for Error flash messages, not a confirm-delete affordance.

## Copywriting Contract

This phase does not introduce new user-facing copy — it relocates existing TempData message strings (already written by controllers) into a shared rendering partial. The "copy" contract here concerns **toast chrome labels only** (the fixed strings surrounding the dynamic message), not the dynamic message text itself, which is unchanged and out of scope.

| Element | Copy |
|---------|------|
| Toast header label — Success | `Success` (unchanged, matches Shop exactly) |
| Toast header label — Error | `Error` (unchanged, matches Shop exactly) |
| Toast header label — Warning | `Warning` (new fixed header label — no prior toast precedent for Warning; Admin/Users.cshtml's current Warning alert has no header label at all since it's a flat alert box, so this is a net-new but obvious/low-risk string) |
| Toast header label — Info (new) | `Info` (new fixed header label, D-05) |
| Toast header label — GoldReceived (bespoke) | `Gold Received!` (unchanged, matches Shop `Details.cshtml:84` exactly) |
| Dynamic message body | Sourced verbatim from each controller's existing `TempData["Success"/"Error"/"Warning"/"Info"]` string — no rewriting of any existing flash message text in this phase |
| Empty state | Not applicable — toasts render conditionally only `@if (TempData[key] != null)`; there is no "empty toast" state to design |
| Destructive confirmation | Not applicable — no destructive-action UI introduced by this phase; existing confirm-dialogs (e.g. Delete/Remove-from-group flows from Phase 41) are unchanged and out of scope |

**TempData key standardization (Claude's Discretion, D-lean confirmed):** Standardize on `Success` / `Error` / `Warning` / `Info` everywhere. `AccountController.cs`'s three Profile-page call sites currently use `SuccessMessage` / `ErrorMessage` / `InfoMessage` — these must be renamed to the standard 4 keys as part of this phase so `_Toasts.cshtml` reads one consistent set of keys with no dual-scheme branching logic.

**GoldReceived interaction (Claude's Discretion, D-lean confirmed):** Keep `GoldReceived` as a bespoke Shop-only toast block inside (or alongside) `_Toasts.cshtml`, rendered in addition to — not merged into — the generic Success toast. Shop's `Details.cshtml:79-98` pattern (icon + "+X gp" badge + small-text success message underneath) is the exact markup to preserve. Note the existing coupling: the GoldReceived block currently also echoes `@TempData["Success"]` as a `<small>` sub-line (line 93) — preserve this dual-render behavior (Success toast fires separately AND its text is echoed inside the GoldReceived toast), since that is the current shipped behavior and no CONTEXT.md decision calls for changing it.

## Interaction / Dismiss Behavior Contract

(Non-template addition — required for this phase since dismiss timing is itself a locked decision, D-04, and is safety/UX-critical.)

| Toast Type | Autohide | Delay | Dismiss |
|------------|----------|-------|---------|
| Success | `data-bs-autohide="true"` | `data-bs-delay="5000"` (5000ms, matches Shop exactly) | Auto + manual `btn-close` |
| Error | **No** `data-bs-autohide` attribute (sticky) | N/A | Manual `btn-close` only |
| Warning | **No** `data-bs-autohide` attribute (sticky) | N/A | Manual `btn-close` only |
| Info (new) | `data-bs-autohide="true"` | `data-bs-delay="5000"` | Auto + manual `btn-close` |
| GoldReceived (bespoke) | `data-bs-autohide="true"` | `data-bs-delay="6000"` (unchanged from Shop's existing 6000ms — do not change to 5000ms) | Auto + manual `btn-close` |

Container/position: fixed top-right, `<div class="toast-container position-fixed top-0 end-0 p-3" style="z-index: 1055;">` — exact class string and z-index from Shop's reference (D-10). Multiple simultaneous toasts stack vertically via Bootstrap's default toast-container flow (D-13) — no custom stacking/offset CSS needed.

Icons per type (D-12):

| Type | Icon class |
|------|-----------|
| Success | `fa-check-circle` (existing) |
| Error | `fa-exclamation-circle` (existing) |
| Warning | `fa-exclamation-triangle` (new — deliberately distinct from Error's circle icon) |
| Info | `fa-info-circle` (new) |
| GoldReceived (bespoke) | `fa-coins` (existing, plus a larger `fa-2x` coin icon inside the body per Shop's current markup) |

JS init (unchanged from Shop, applies globally now): on `DOMContentLoaded`, `document.querySelectorAll('.toast')` → `new bootstrap.Toast(el)` → `.show()`. This script moves from being duplicated per-view into a single site-wide script include (layout-level or a shared `.js` file) so it fires once regardless of which layout rendered `_Toasts.cshtml`.

## Migration Scope Reference (for planner/executor — not a design token, but binding scope)

- **Shared partial:** `QuestBoard.Service/Views/Shared/_Toasts.cshtml` (new file), rendered via `<partial name="_Toasts" />` in all 3 layouts: `_Layout.cshtml`, `_Layout.Mobile.cshtml`, `_Layout.GroupPicker.cshtml` (D-01, D-02, D-03).
- **~20 alert-banner views convert** to rely on the shared partial instead of their own local `alert alert-dismissible` markup (D-08) — full list in `42-CONTEXT.md` Canonical References.
- **4 Shop toast files migrate** their local toast markup onto the shared partial too — `Views/Shop/Index.cshtml` + `.Mobile.cshtml`, `Views/Shop/Details.cshtml` + `.Mobile.cshtml` (D-07). The Shop "Mystical Merchant" novelty toast (`showMerchantDialog`/`showMerchantToast`, `Index.cshtml:514-572`) is explicitly OUT of scope and untouched (D-06) — its yellow `.merchant-toast` styling stays exactly as-is, and it is not part of the shared partial.
- **Form validation** (`asp-validation-summary`, `asp-validation-for`) is out of scope (D-09) — do not touch.
- **`RedirectWithInfo` extension method** should be added to `ControllerExtensions.cs` alongside the existing `RedirectWithSuccess`/`RedirectWithError`/`RedirectWithWarning`, following the exact same one-line pattern (`RedirectWithMessage(action, "Info", message)`).
