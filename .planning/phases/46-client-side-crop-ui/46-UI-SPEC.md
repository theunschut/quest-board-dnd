## Spacing Scale

This app has no declared spacing token system — it uses Bootstrap's default spacing utility scale (`0.25rem` increments: `p-1`…`p-5`, `mb-1`…`mb-5`, `gap-1`…`gap-5`) directly in markup, with occasional literal `rem`/`px` values in CSS files. This phase follows that existing convention rather than introducing a new 4px-multiple token system.

| Token | Value | Usage |
|-------|-------|-------|
| Bootstrap `mb-2`/`gap-2` | 0.5rem (8px) | Compact spacing inside modal body (label-to-input gaps) |
| Bootstrap `mb-3`/`p-3` | 1rem (16px) | Default form-group spacing — matches every existing Create/Edit view |
| Bootstrap `p-4`/`mb-4` | 1.5rem (24px) | Modal body padding (Bootstrap's own `.modal-body` default), section breaks |
| Crop stage height | 400px fixed | `<cropper-canvas>` height inside the modal (per RESEARCH.md Pattern 3) — fixed, not scaled, so the crop stage never overflows a mobile viewport unpredictably |

Exceptions: The crop-stage `400px` fixed height is a deliberate exception to rem-based sizing, needed because Cropper.js v2's `cropper-canvas` requires an explicit pixel/percentage height to lay out correctly; on mobile (`.Mobile.cshtml` views, viewport ≤ ~430px wide) reduce to **320px** so the modal fits above the on-screen keyboard/safe-area on a typical phone in portrait.

## Typography

This app declares no separate type scale for utility surfaces (modals, forms) — everything uses Bootstrap defaults (`1rem`/16px body, browser default line-height) except page-level headings, which use `Cinzel` per existing CSS. The crop modal introduces no new typography, only reuses these two existing tiers:

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Modal title (`.modal-title`) | 1.25rem (20px, Bootstrap default `h5`) | 500 (Bootstrap default) | 1.2 |
| Modal body text / helper copy | 1rem (16px) body, 0.875rem (14px) `.form-text`/`small` | 400 regular | 1.5 |
| Body (forms, labels) | 16px | 400 regular / 700 for `.form-label` (existing site.css rule) | 1.5 |
| Heading (page-level only, NOT this modal) | 28px+ (`Cinzel`) | 600–700 | 1.2 |

No new font sizes or weights are introduced by this phase — the crop modal strictly reuses Bootstrap's existing `.modal-title`/`.modal-body`/`.form-text` typography, consistent with every other modal already in this codebase (e.g. `#addCharacterModal` in `Quest/Details.cshtml`).

## Color

This app has no formal 60/30/10 token declaration — it is a fantasy-themed Bootstrap skin over a fixed background image (`body::before`, Notice Board parchment art). The dominant/secondary/accent roles below describe the *existing* visual system this phase must match, not a new palette.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | Fixed parchment/notice-board background image (`body::before`) + `rgba(255,255,255,0.15)` glass-card fill (`modern-card`) | Page background, card surfaces |
| Secondary (30%) | `rgba(20,20,30,0.95)` dark card gradient (character/contact cards), `#1a1a2e`→`#0f3460` dark gradient (shop), `bg-dark`/`text-light` (existing Bootstrap modal convention) | Card backgrounds, existing modal chrome, list-card containers |
| Accent (10%) | `#ffc107` (Bootstrap `warning` amber/gold) | **Reserved for:** the crop modal's primary "Use Photo" submit-adjacent button (`btn-warning`, matching every existing form's Save/Submit button), the Cropper.js `cropper-handle`/`cropper-grid` `theme-color` (amber to match the app's existing selected/active-state color used in `.custom-radio-label` checked states and `.main-badge`), and the file-input `:focus` ring (already globally amber via existing `.form-control:focus` rule in `site.css`) |
| Destructive | `#dc3545` (Bootstrap `danger` red) | Destructive actions only — not used inside the crop modal itself (there is no destructive action in the crop flow); reserved for the pre-existing "Remove class"/"Delete" buttons elsewhere on the same forms |

Accent reserved for: the modal's primary action button, the crop-selection handles/grid lines (so the interactive crop frame reads as "the same amber the rest of the app uses for active/selected state"), and the file-input focus ring. Never applied to body text, card borders in their default (non-hover) state, or secondary/cancel buttons.

## Copywriting Contract

| Element | Copy |
|---------|------|
| File input label (unchanged, all 3 forms) | "Profile Picture" (Character), "Profile Photo" (DM), "Photo" (Contact) — reuse each form's existing `asp-for` label text verbatim, no rename |
| Modal title | "Crop Your Photo" |
| Modal helper text (below the crop stage, above the footer buttons) | "Drag to reposition, use the corner handles to resize. We'll use a square crop — if you skip this, we'll center one automatically." |
| Primary CTA (modal footer, right-aligned per CLAUDE.md button-layout convention) | "Use This Crop" (confirms the current frame position/size and closes the modal) |
| Secondary/dismiss (modal footer, left-aligned) | "Discard Photo" (closes modal, clears the selected file from the input entirely — does not silently fall back to an uncropped submission, since there is nothing to submit without a file) |
| Empty state — no photo uploaded yet (Character/Contact/DM forms, "current photo" preview area) | No heading needed; existing placeholder icon + no copy (matches current behavior — the `@if (Model.ProfilePicture != null)` block simply doesn't render). No change from today's behavior. |
| Error state — file rejected (existing client-side file-size/type check, unchanged copy, now also covers the pre-processing step) | Existing: "File size ({X} MB) exceeds the maximum allowed size of 5 MB. Please choose a smaller image." / "Only image files (JPG, PNG, GIF) are allowed." New addition for a corrupt/undecodable file: "This image couldn't be read. Please choose a different photo." (shown in the same `#fileSizeError`-style div, triggered when `createImageBitmap()` rejects) |
| Destructive confirmation | None in this phase's scope — selecting a new file simply replaces the pending crop; there is no separate "delete photo" action inside the crop modal (that existing capability, if any, lives outside this phase's scope on the surrounding form) |

**D-04 interaction copy note:** Because the modal must never block submission, the primary button is framed as "confirm this is how you want it," not "you must click this to proceed" — a user can close the modal via the "×" close button or by clicking outside it (Bootstrap's default backdrop-click dismiss) and their **auto-centered default crop still applies** when they submit the surrounding form. Do not word the modal in a way that implies skipping it loses the upload (e.g. avoid "Crop your photo to continue" — use the helper text above instead, which explicitly states the fallback).

### 1. Crop Modal — Structure and Markup Pattern

Use a **standard Bootstrap 5 modal** (`modal fade`), matching the existing dark-modal convention from `Views/Quest/Details.cshtml`'s `#addCharacterModal` — do not build a custom overlay. One modal instance per view (Create/Edit have their own file input and their own modal ID; do not share a single global modal across multiple forms on the same page since only one page ever has one upload field active).

```html
<div class="modal fade" id="cropPhotoModal" tabindex="-1" aria-labelledby="cropPhotoModalLabel" aria-hidden="true"
     data-bs-backdrop="static" data-bs-keyboard="false">
  <div class="modal-dialog modal-dialog-centered modal-lg">
    <div class="modal-content bg-dark text-light">
      <div class="modal-header border-secondary">
        <h5 class="modal-title" id="cropPhotoModalLabel">
          <i class="fas fa-crop-alt me-2"></i>Crop Your Photo
        </h5>
        <button type="button" class="btn-close btn-close-white" data-bs-dismiss="modal" aria-label="Close"></button>
      </div>
      <div class="modal-body">
        <div class="crop-stage">
          <cropper-canvas background style="width:100%;height:400px;">
            <cropper-image id="cropperImageEl" alt="Photo to crop" rotatable scalable translatable></cropper-image>
            <cropper-shade hidden></cropper-shade>
            <cropper-selection id="cropperSelectionEl" initial-coverage="0.8" movable resizable aspect-ratio="1">
              <cropper-grid role="grid" bordered covered></cropper-grid>
              <cropper-crosshair centered></cropper-crosshair>
              <cropper-handle action="move" theme-color="rgba(255,193,7,0.35)"></cropper-handle>
              <cropper-handle action="n-resize"></cropper-handle>
              <cropper-handle action="e-resize"></cropper-handle>
              <cropper-handle action="s-resize"></cropper-handle>
              <cropper-handle action="w-resize"></cropper-handle>
              <cropper-handle action="ne-resize"></cropper-handle>
              <cropper-handle action="nw-resize"></cropper-handle>
              <cropper-handle action="se-resize"></cropper-handle>
              <cropper-handle action="sw-resize"></cropper-handle>
            </cropper-selection>
          </cropper-canvas>
        </div>
        <p class="text-muted small mt-2 mb-0">
          Drag to reposition, use the corner handles to resize. We'll use a square crop —
          if you skip this, we'll center one automatically.
        </p>
      </div>
      <div class="modal-footer border-secondary d-flex justify-content-between">
        <button type="button" class="btn btn-secondary" data-bs-dismiss="modal" id="cropCancelBtn">
          <i class="fas fa-times me-2"></i>Discard Photo
        </button>
        <button type="button" class="btn btn-warning" id="cropConfirmBtn">
          <i class="fas fa-check me-2"></i>Use This Crop
        </button>
      </div>
    </div>
  </div>
</div>
```

**Handle theme color:** `rgba(255,193,7,0.35)` (translucent amber) on the move handle — matches this app's `#ffc107` accent used everywhere else for active/selected state, applied at reduced opacity so it doesn't visually compete with the photo underneath.

**No canvas-level `cropper-handle[action="select"]`:** Cropper.js v2's own source shows this handle captures every canvas drag and always moves/resizes the existing selection (`$selection.$change(...)`), which prevents `cropper-image[translatable]`'s pan-drag from ever firing — after zooming with the wheel (unbounded; the library enforces no min/max scale or bounds-checking), the image could go out of view with no way to drag it back. Omitting this handle lets a drag that starts outside the selection fall through to `CropperImage.$handleAction`'s pan branch instead, restoring image panning. Found live during Phase 46's Task 2 desktop verification (46-07).

**Mobile variant:** Same markup, `cropper-canvas` height `320px` instead of `400px` (fits above the keyboard/safe-area on a phone in portrait), and drop `modal-lg` (use the modal's default width so it doesn't overflow a narrow viewport). Button labels/icons unchanged — do not shrink to icon-only, since D-04's "never blocks submission" framing depends on the user clearly reading which button does what.

### 2. Trigger Interaction (D-04)

On the file input's `change` event:
1. Run the file through `prepareImageForCropper()` (EXIF-correct + downscale, per RESEARCH.md Pattern 2).
2. Set the corrected Blob's object URL as `cropper-image`'s `src`.
3. Open the modal (`bootstrap.Modal.getOrCreateInstance(...).show()`).
4. Cropper.js auto-applies its `initial-coverage="0.8"` centered selection at `aspect-ratio="1"` — this **is** the D-04 default crop; no additional JS needed to "center" it, since Cropper.js v2's own default selection behavior already satisfies the requirement.
5. Populate the hidden `CroppedPictureFile` input immediately from that default selection (do not wait for the user to touch anything) — this guarantees a value exists even if the user closes the modal via "×", backdrop click is disabled (`data-bs-backdrop="static"` per the markup above — backdrop click and Esc are both disabled so the only way out is an explicit button, preventing an accidental miss-click from silently discarding the pending crop), or clicks "Use This Crop" without dragging anything.
6. If the user drags/resizes, re-run step 5's extraction only on "Use This Crop" click (avoid re-extracting the canvas on every pointermove — expensive and unnecessary).
7. "Discard Photo" clears the original file input's `.value` entirely (per the Copywriting Contract above) and destroys the Cropper.js instance so the next file selection starts fresh.

**Why static backdrop instead of the default dismissible backdrop:** D-04 says the modal "never blocks form submission," which is about *not requiring an explicit confirm click before the surrounding form can be submitted* — it does not mean the modal itself should be dismissible by accidental outside-click while mid-drag on a touch device (a stray tap outside the crop stage on mobile is a likely accidental gesture, not intentional dismissal). Disabling backdrop/Esc dismissal removes that accidental-loss risk while still fully satisfying D-04, since the default crop is already populated into the hidden input regardless of how the modal closes.

### 3. Current-Photo Preview Placement (all 3 forms)

No change from today's existing placement — this phase does not restructure the form layout, only repoints the `src` and adds the modal:

- **Character/Contact Create+Edit:** existing `<img class="img-thumbnail" style="max-width:200px;">` stays directly above the file input, inside the same `col-md-4` column. Repoint `src` to the new cropped-read endpoint (per D-03) — this preview must show the **cropped** version (it's the Create/Edit form preview, explicitly listed under D-03's "cropped displays everywhere except Details" list).
- **DM EditProfile:** existing `rounded-circle dm-profile-photo` `<img>` stays above the file input in the `col-md-4` photo column. Repoint `src` to the cropped-or-fallback DM read endpoint per D-03.
- The crop modal is a **separate, transient surface** triggered by file selection — it does not replace or sit inline with this existing "current photo" preview. The preview only updates after the surrounding form POSTs and the page reloads (standard full-page-reload MVC pattern already in place); this phase does not add a live in-page preview-swap after cropping, since D-04 explicitly frames this as an ordinary form submission with no separate upload step or reload.

### 4. List-Card Square Box (D-01/D-02)

`.character-image` (`characters.css:112-142`) and `.contact-image` (`contacts.css:91-121`) change from:

```css
width: 100%;
height: 200px;
```

to an aspect-ratio-based square:

```css
width: 100%;
aspect-ratio: 1 / 1;
height: auto; /* remove the fixed 200px */
```

Keep every other existing property on `.character-image`/`.contact-image` unchanged (`border-radius: 8px`, `overflow: hidden`, `background: linear-gradient(...)`, `box-shadow`, hover-state `transform: scale(1.02)`) — this is a box-shape change only, not a redesign. The inner `<img>` keeps `object-fit: cover` (already present) so a non-square source image (there shouldn't be one post-crop, but defensively) still fills the square without distortion.

Update the two `@media (max-width: 768px)` overrides that currently hardcode `height: 180px` for both files — remove those two lines entirely (the `aspect-ratio` rule already handles responsive square sizing at any grid column width; a fixed-height override would reintroduce the landscape-crop problem D-02 exists to fix).

DM's `.dm-profile-photo`/`.dm-profile-photo-mobile` (128×128 / 80×80 circle) needs **no change** — already square-boxed with `rounded-circle` applied on top, confirmed by D-01/D-02.

### 5. Dual-File Submission Wiring (all 6 upload views)

Add one hidden input per form, alongside the existing visible file input:

```html
<input type="file" name="CroppedPictureFile" id="croppedPictureFileInput" style="display:none" />
```

ViewModel gains a matching `IFormFile? CroppedPictureFile` property on `CharacterViewModel`, `ContactViewModel`/equivalent, and `EditDMProfileViewModel` — this is a planning/implementation task, not a visual-contract concern, but is called out here since the modal's "Use This Crop"/auto-default flow has nothing to populate without it existing.
