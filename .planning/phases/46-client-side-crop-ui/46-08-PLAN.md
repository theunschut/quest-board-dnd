<objective>
Close a gap discovered during Plan 03's execution: Character and Contact Create POST actions validate a submitted cropped photo but have nowhere to persist it — the generic `AddAsync(model, token)` inherited from `BaseService<T>`/`BaseRepository<T,TEntity>` maps only `model.ProfilePicture`/`model.ContactImageData` (the original) into a new `CharacterImageEntity`/`ContactImageEntity` row via AutoMapper; it never touches `CroppedImageData`. Today, a user who crops their photo while creating a brand-new character sees a fully working crop UI, submits successfully, and the crop is silently discarded — the new character shows the uncropped original everywhere until its first Edit. This runs against the phase's own locked success criterion #1 ("every image-upload field... create and edit").

The fix does NOT require a new domain-model field or a new repository method. `CharacterRepository.UpdateWithProfileImageAsync(model, originalImageData, croppedImageData, token)` and its Contact equivalent already exist from Phase 45 and already handle "create the image row if the character has none yet" (see the `entity.ProfileImage ??=` branch). Calling it once, right after the base `AddAsync` populates `model.Id`, is sufficient — no Rule 4 architectural change needed, contrary to Plan 03's initial assessment (that assessment was made without checking whether an existing repository primitive could be reused).

Purpose: Make Create genuinely honor the phase's "create and edit" success criterion before Wave 3's Plan 46-06 wires the crop modal into the Character/Contact Create forms — otherwise Wave 3 would ship a crop UI on Create that silently does nothing.
Output: A new 3-arg `AddAsync` overload on `ICharacterService`/`IContactService` (additive, existing 2-arg generic signature untouched), wired into both Create POST actions, with unit and integration test coverage.
</objective>

ICharacterService.cs — no explicit AddAsync declared today; inherits the generic
  Task AddAsync(Character model, CancellationToken token = default);
from IBaseService<Character>. Add a NEW additive overload:
  Task AddAsync(Character model, byte[]? newCroppedImageData, CancellationToken token = default);

CharacterService.cs (BaseService<Character>) — base AddAsync is:
  public virtual async Task AddAsync(TModel model, CancellationToken token = default) { await repository.AddAsync(model, token); }
New 3-arg override implementation:
  public async Task AddAsync(Character model, byte[]? newCroppedImageData, CancellationToken token = default)
  {
      await repository.AddAsync(model, token); // populates model.Id and creates the ProfileImage row with OriginalImageData via existing AutoMapper CreateMap<Character, CharacterEntity>()
      if (newCroppedImageData != null)
      {
          await repository.UpdateWithProfileImageAsync(model, model.ProfilePicture, newCroppedImageData, token);
      }
  }

Identical shape for IContactService/ContactService, substituting `model.ContactImageData` for `model.ProfilePicture` (Contact's original-image field has a different name — confirmed in QuestBoard.Domain/Models/Contact.cs).

Why this works without touching AutoMapper or the domain model: `repository.AddAsync` already creates the `CharacterImageEntity`/`ContactImageEntity` row with `OriginalImageData` set (via the existing `CreateMap<Character, CharacterEntity>().ForMember(dest => dest.ProfileImage, ...)` in QuestBoard.Repository/Automapper/EntityProfile.cs). `UpdateWithProfileImageAsync` re-fetches that same freshly-created entity (by `model.Id`, now populated) and sets `CroppedImageData` on it — see `CharacterRepository.cs` lines ~154-224 and `ContactRepository.cs` lines ~106-145 for the exact `entity.ProfileImage.CroppedImageData = croppedImageData;` assignment. Two SaveChanges calls (one from AddAsync, one from UpdateWithProfileImageAsync) is an accepted, momentary non-atomicity — consistent with this codebase's existing patterns elsewhere.

CharactersController.Create POST (QuestBoard.Service/Controllers/Characters/CharactersController.cs, current Create action) — today:
  - Builds `original` and `cropped` ImageFileInput, calls `imageValidationService.ValidateImagePair(original, cropped)`.
  - Copies ONLY the original file's bytes: `viewModel.ProfilePicture = memoryStream.ToArray();` — the cropped file's bytes are never extracted into a variable.
  - Calls `await characterService.AddAsync(character, token);` (the plain 2-arg overload).
Needed change: after the existing original-bytes copy, if `viewModel.CroppedPictureFile is { Length: > 0 }`, copy its bytes into a local `byte[]? croppedImageData` (mirroring the exact `CopyToAsync`-to-`MemoryStream` pattern already used for the original). Change the final call to `await characterService.AddAsync(character, croppedImageData, token);`.

ContactsController.Create POST (QuestBoard.Service/Controllers/Contacts/ContactsController.cs, current Create action) — identical shape: builds `original`/`cropped` ImageFileInput at lines ~104-111, copies only `newContactImageFile` bytes into `viewModel.ContactImage` (the ViewModel's original-photo byte[] property — distinct from the domain model's `Contact.ContactImageData`), calls `await contactService.AddAsync(contact, token);` at line ~137. Apply the identical local-variable-extraction + 3-arg-call-site change.
