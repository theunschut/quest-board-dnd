<objective>
Create the shared client-side crop pipeline as two feature-scoped assets: `wwwroot/js/image-crop.js` (one `initImageCrop({...})` initializer that each upload view calls with its own element IDs) and `wwwroot/css/image-crop.css` (crop-modal layout). The JS implements the full verified pipeline from RESEARCH.md Patterns 1-4: intercept the file `change` event, EXIF-correct and downscale via `createImageBitmap({imageOrientation:'from-image'})` → scratch canvas → `toBlob()`, feed the corrected Blob URL to `cropper-image`, open the Bootstrap modal with a locked 1:1 selection, immediately populate the hidden `CroppedPictureFile` input from the default centered crop (so a value exists even if the user never touches the frame, per D-04), re-extract on "Use This Crop" via `$toCanvas()`, and clear/destroy on "Discard Photo".

This plan produces the assets ONLY — wiring them into the 10 upload views (script includes, modal markup, hidden input, preview repoint) is Plan 06. This separation keeps the novel canvas/Cropper.js work isolated in one reviewable file.

Purpose: Deliver the interactive drag/resize/zoom crop capability (IMAGE-01) as reusable, EXIF-safe, canvas-memory-safe client code.
Output: `image-crop.js` and `image-crop.css`.
</objective>

| T-46-05-01 | Denial of Service (client) | iOS Safari 16.7M-pixel canvas ceiling | mitigate | The prepareImageForCropper step downscales every file to ≤2400px long-edge BEFORE Cropper.js initializes, and explicitly frees bitmap/canvas memory (bitmap.close(), canvas dims = 0) — so no raw 12MP image ever reaches a canvas.
