<objective>
Wire the UI and the HTTP endpoint. Replace the narrow `ChangeVoteToYes` controller action with a single `ChangeVote(int id, VoteType vote)` action that validates the vote enum, resolves the caller's own signup (IDOR guard), and delegates to `questService.ChangeVoteAsync` (never rejecting on capacity). Wire `RevokeSignup` to `questService.RevokeSignupAsync` so a selected player's revoke triggers promotion. Rewrite the desktop `Details.cshtml` waitlist to order via the shared `OrderWaitlist` extension and render Vote Yes/Maybe/No buttons in the Revoke row (Revoke left, votes right), and build the same section from scratch on `Details.Mobile.cshtml` for full parity (D-05).

Purpose: VOTE-01 (waitlist instead of reject at the HTTP boundary), VOTE-02 (both views render the centralized order), VOTE-04/05/06 (the three-button UI drives the correct behaviors), VOTE-07 (the promotion email fires from the wired revoke/vote-No paths). Per D-04, Revoke Signup stays available alongside the three vote buttons (Revoke left, votes right per D-02). Per D-03, the brand-new-player `JoinFinalizedQuest` action stays UNCHANGED — the three-button vote UI applies only to players who already have a `PlayerSignup` row; do not add Maybe/No to initial join and do not touch `JoinFinalizedQuest`. Security V4/V5: ownership resolution and `Enum.IsDefined` validation at the controller.
Output: A rewritten `ChangeVote` action, a promotion-wired `RevokeSignup`, and the vote UI + centralized waitlist ordering on both desktop and mobile.
</objective>
