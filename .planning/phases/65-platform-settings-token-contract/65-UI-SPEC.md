# Phase 65 — UI Design Contract

> Visual and interaction contract for the SuperAdmin-only "Integrations" settings page (`/platform/Integrations`). Generated by gsd-ui-researcher, verified by gsd-ui-checker.

**Note on scope:** Phase 65 is 100% server-rendered Razor (ASP.NET Core MVC + Bootstrap, no SPA/component framework), so the shadcn initialization gate in the base template does not apply. Every visual decision below is either pulled directly from CONTEXT.md's locked decisions (D-01–D-06) or matched against a verified, currently-in-use precedent elsewhere in this codebase — no new visual language is introduced. No user prompts were needed; all design-contract questions were answerable from upstream artifacts + direct codebase inspection.

## Spacing Scale

Declared values (must be multiples of 4) — mapped onto the Bootstrap utility classes already used by every other Platform-area form (`EditUser.cshtml`, `Group/Edit.cshtml`), not a new scale:

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon-to-label gap (`me-2` on all `<i class="fas ...">` per CLAUDE.md) |
| sm | 8px | `gap-2` on the header button row (`Group/Index.cshtml`'s `d-flex gap-2`) |
| md | 16px | `mb-3` between each form field group (URL field, secret field, enabled toggle) — matches `EditUser.cshtml`/`Group/Edit.cshtml` exactly |
| lg | 24px | `modern-card-body` internal padding (`site.css:1083`, `padding: 1.5rem` = 24px) — inherited automatically, no override needed |
| xl | 32px | Not used on this page (single-card layout, no multi-section vertical rhythm needed) |
| 2xl | 48px | `py-5` on the "not configured" first-run indicator area, if a centered empty-style block is used (matches `Group/Index.cshtml`'s `py-5 text-center` empty-state pattern) |
| 3xl | 64px | Not used — this is a single-card settings form, not a full page-level layout |

Exceptions: **`<hr>` before the button row is mandatory** (root `CLAUDE.md` UI guideline) — not a spacing token, but a required structural element between the form fields and the Back/Save button row, confirmed present in every existing Platform-area edit form.

## Typography

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 16px | 400 | 1.5 |
| Label | 16px | 600 | 1.5 |
| Heading | 20px | 600 | 1.2 |
| Display | 32px | 600 | 1.2 |

- **Body (16px/400/1.5):** paragraph copy, table/badge-adjacent text, `.modern-card p`/`.form-text` helper copy (`site.css` root `body` uses Bootstrap's default `1rem`/`1.5` — confirmed, no override).
- **Label (16px/600/1.5):** `.form-label` — `modern-card .form-label` explicitly forces `font-weight: 600` (`site.css:1125-1131`); applies to "Omphalos URL", "Shared Secret", "Integration Enabled" labels.
- **Heading (20px/600/1.2):** Bootstrap default `<h5>` size — used for any sub-section heading within the card if one is needed (none required by this phase's single-form layout, declared for consistency with the rest of the design system, e.g. `EditUser.cshtml`'s "Password Management" `<h5>`).
- **Display (32px/600/1.2):** Bootstrap default `<h2>` size — the card title itself (`<h2 class="mb-0">Integrations</h2>`), matching `Group/Index.cshtml`/`Users/Index.cshtml`/`EditUser.cshtml` exactly. Color/text-shadow (parchment `#F4E4BC` + dark shadow) is applied automatically by `.modern-card h1-h6` (`site.css:1094-1105`) — no additional styling needed.
- **Exception — small/meta text (14px, 400–500, 1.5):** badges (`.badge`), `.form-text` helper lines, and any secret-status indicator text ("•••• Configured" / "Not configured yet") use Bootstrap's `0.875rem` small-text convention, matching `vote-summary`/badge sizing used site-wide (`site.css:34-37`).

## Color

This app does not use a flat single-hex 60/30/10 palette — it layers Bootstrap semantic colors under a photographic-background + glassmorphism system. The table below maps the standard 60/30/10 contract onto that reality, using only colors already established and in production use elsewhere in the Platform area (no new hex values introduced).

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#8B4513` (saddle brown, representative of the fixed background image + `.navbar` gradient `rgba(139,69,19,.5)→rgba(205,133,63,.5)`, `site.css:501-511`) | Page backdrop, navbar chrome — pre-existing site-wide, unmodified by this phase |
| Secondary (30%) | `#F4E4BC` (parchment) on `rgba(255,255,255,0.15)` blurred glass (`.modern-card`, `site.css:1045-1052`) + `rgba(255,255,255,0.95)` opaque fields (`.modern-card .form-control`, `site.css:1142-1150`) | The Integrations card surface itself: card body background, `Omphalos URL`/`Shared Secret` input backgrounds |
| Accent (10%) | `#ffc107` (Bootstrap `btn-warning`) | Reserved for exactly one element: the **"Save Settings"** submit button. Matches `Group/Edit.cshtml`'s `btn-warning` "Save Changes" — the closest architectural analog (same Platform area, same "modify an existing config entity" semantics), and the majority pattern across the codebase's "Edit existing entity" forms (`Group/Edit`, `Contacts/Edit`, `Quest/Edit`, `Characters/Edit`, `DungeonMaster/EditProfile` all use `btn-warning`, vs. only `EditUser`/`Account/Edit` using `btn-success`). The existing global gold focus-ring glow (`.form-control:focus`, `site.css:40-43`) applies automatically to this page's inputs — no additional work, not counted against the reserved-for list. |
| Destructive | `#dc3545` (Bootstrap `btn-outline-danger` / `.modern-card .alert-danger`) | Reserved for exactly one element: the **"Generate Secret"** button — `btn-outline-danger` + `fa-key`, directly matching `EditUser.cshtml`'s "Reset Password" precedent (`btn-outline-danger`, same "sensitive credential action" semantics) — plus its `confirm()` dialog copy (see Copywriting Contract). This is a credential-rotation action, not a data-delete action, but it immediately invalidates the previously-working secret, which is the same risk class the codebase already treats as `btn-outline-danger`. |

**Accent reserved for:** the "Save Settings" submit button only. Not applied decoratively to the header icon, badges, or the toggle checkbox — those use their own established semantic colors below.

**Status badges (semantic, not part of the accent/destructive budget):** `bg-success` = "Configured" (secret) / green toggle icon for "Enabled"; `bg-secondary` = "Not configured" — directly matching `Users/Index.cshtml`'s existing `Confirmed`/`Unconfirmed`, `Active`/`Disabled` badge convention (`bg-success`/`bg-secondary`, never a new color pair).

**Explicitly not used on this page:** gold/`#ffc107` beyond the one reserved Save-button use — the app's other gold accents (navbar brand glow, shop UI) belong to a different visual register (Cinzel hero surfaces) and would be inconsistent with the `modern-card` form pattern this page follows.

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA | **"Save Settings"** — `btn-warning`, `fa-save`, right-aligned (per CLAUDE.md: secondary/cancel left, primary/submit right) |
| Empty state heading | Not applicable in the list-empty sense — `IntegrationSettingEntity` is a singleton row, auto-created (disabled, blank) on first `GetAsync()` call; the page always renders the settings form, never a "no items" list state. The first-run/unconfigured condition is instead communicated per-field (see Empty state body). |
| Empty state body | Secret-status indicator shown on GET in place of the real value (Pattern 3 — GET never pre-fills `SharedSecret`): **configured** → `"•••• Configured — leave blank to keep the current secret."` (`text-muted`, badge `bg-success` "Configured"); **not yet configured** → `"Not configured yet — click Generate Secret or paste a value below."` (`text-muted fst-italic`, badge `bg-secondary` "Not configured"), matching `Users/Index.cshtml`'s existing badge-plus-caption pattern. |
| Error state | Invalid URL (`[Url]` DataAnnotation on `OmphalosUrl`, precedented by `EditShopItemViewModel.ReferenceUrl`): **`"Enter a valid URL, e.g. https://your-omphalos-instance.example.com."`** rendered via `asp-validation-for`. Generic save failure (`ModelState` summary, `text-danger`, matching `asp-validation-summary="ModelOnly"` used in `Group/Edit.cshtml`/`EditUser.cshtml`): **`"Couldn't save integration settings. Check the fields below and try again."`** |
| Destructive confirmation | **"Generate Secret"** (`btn-outline-danger`) requires a JS `confirm()` before the POST fires, mirroring `UsersController.Disable`'s consequence-explaining pattern (`Users/Index.cshtml:86`): **`"Generate a new shared secret? The current secret stops working immediately. Update Omphalos's configured secret to match, or SSO will fail until you do."`** The action persists immediately on confirm (RESEARCH.md Pitfall 3 — no separate deferred "Save" click), then redirects to `Index` GET with the plaintext value carried in `TempData` for exactly one render, shown once in the masked field per D-03. |

**Supporting copy (not itemized above, but locked for consistency):**
- Nav entry button on `Group/Index.cshtml`/`Index.Mobile.cshtml`: **"Integrations"** — `btn-secondary`, `fa-plug`, added as the third header button alongside "Create Group"/"Manage Users" (RESEARCH.md Pattern 4).
- Card header: **"Integrations"** — `<h2>`, `fa-plug text-danger` (matches the majority Platform-header icon color: `Group Management`/`User Accounts` both use `text-danger`).
- Intro line under the header (frames the generic "Integrations" naming from D-05 while only one integration exists today): **"Configure the connection Quest Board uses to hand DMs and players off into Omphalos session notes."** (`text-muted`, plain paragraph, no card treatment).
- Field labels: **"Omphalos URL"** (placeholder `https://your-omphalos-instance.example.com`, helper `form-text`: *"Base URL of your Omphalos instance. Used to build the signed redirect link."*); **"Shared Secret"** (`type="password"`).
- Toggle label: **"Integration Enabled"** — plain `form-check` checkbox (never `form-switch` — RESEARCH.md Pattern 5, no toggle-switch precedent exists anywhere in this app), `fa-toggle-on text-success` icon, matching `EditUser.cshtml`'s `HasKey` checkbox markup exactly.
- Save success: **`TempData["Success"] = "Integration settings saved."`** (matches `GroupController`'s `"Group name updated."` tone — short, past-tense, no exclamation).
- Generate success: **`TempData["Success"] = "New shared secret generated. Copy it now and update Omphalos's configuration — it won't be shown again."`**
- Back link: **"Back to Group Management"** — `btn-secondary`, `fa-arrow-left`, left-aligned, returns to `Group/Index` (the page this feature is entered from).
