<objective>
Widen the authorization boundary on `GuildMembersController` so an Admin (per-group `GroupRole.Admin`) or SuperAdmin (global role, resolved to Admin-equivalent by `GetEffectiveGroupRoleAsync`) can Edit, Delete, and Retire/Reactivate a character owned by another player in their active group — while Players remain restricted to their own characters and cross-tenant access stays blocked.

Purpose: Admins/DMs frequently need to correct another player's character sheet or manage a stranded/broken character without the player's own login. Today all three actions (`Edit`, `Delete`, `ToggleRetirement`) hard-code an owner-only guard with no admin escape hatch.

Output: Updated controller guards (all three actions + Details flag), a new `CanEdit` ViewModel flag, both Details views gated on `CanEdit`, and 11 new authorization integration tests. No schema change, no migration, no new packages — this is a pure pattern-mirror of the already-shipped `DungeonMasterController.EditProfile` / `DMProfileViewModel.CanEdit` analog.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser → GuildMembersController (Edit/Delete/ToggleRetirement) | Authenticated but potentially non-owner / non-admin user submits an action against a character id they do not own; the controller guard is the enforcement point |
| Controller → EF Core (DbContext.Characters) | Character id from the request is resolved through the global query filter; cross-group ids must never surface |
| Controller-computed CanEdit flag → Razor view | UI-only convenience flag; must not be mistaken for the security boundary |

Verified by `Edit_Player…ShouldBeForbidden`, `Delete_Player…ShouldBeForbidden`, `ToggleRetirement_Player…ShouldBeForbidden`. |
| T-56-02 | Information Disclosure / EoP (IDOR) | Admin in Group A guessing a Group B character's numeric id | accept (existing control relied upon) | No new code — `CharacterEntity`'s fail-closed EF Core `HasQueryFilter` (Phase 49, Phase 55 shipped) returns null → the controller's existing `NotFound()` fires. D-03 forbids a parallel hand-rolled guard. Verified by `Edit_Admin…DifferentGroup…ShouldReturnNotFound` and `Delete_Admin…DifferentGroup…ShouldReturnNotFound` (expect 404, not 403). |
