<objective>
Add a nullable `Rewards` string field to Quest end-to-end through the backend: the EF entity, the domain model, both quest ViewModels, a dedicated EF Core migration, and the two explicit-parameter service/repository methods (plus their interfaces) that the Edit and Follow-Up flows route through — then update the existing unit tests whose mock call-sites break when the new parameter is added.

Purpose: Establish the data + service plumbing so the Rewards value can be created, edited, persisted, and read. The view layer (Plan 59-02) depends on the ViewModel/domain properties added here.
Output: A compiling, migration-backed `Rewards` column that survives round-trip through Create (AutoMapper), Edit (explicit params), and Follow-Up (explicit params, blank by default), with green unit tests.
</objective>
