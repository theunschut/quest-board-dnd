<objective>
Add the `BoardType` bindable field to the Create ViewModel (nullable + `[Required]` so it forces an explicit choice), add a display-only `BoardType` to the Edit ViewModel, and wire the controller: Create POST persists the selection, Edit GET populates the display value, and Edit POST deliberately never reads `BoardType` (the mass-assignment mitigation, D-06).

Purpose: This is the request/response contract between the browser and the data layer built in Plan 01. Making the Create ViewModel's `BoardType` a nullable `BoardType?` with `[Required]` is what satisfies D-03 ("no pre-selected default, forces an active choice") — a plain non-nullable enum would silently bind an empty submission to `OneShot`. Leaving `model.BoardType` unread in the Edit POST body is what implements D-06's silent-tamper-rejection with zero extra code.

Output: Two updated ViewModels and an updated `GroupController` whose Create/Edit actions correctly persist, display, and protect `BoardType`. After this plan, the Create-persist and Create-validation integration tests from Plan 01 should go green; the Edit-tamper test should also go green (POST ignores BoardType). The views still need the actual form controls (Plan 03) for a human to exercise the flow, but the server-side contract is complete and test-verified here.
</objective>
