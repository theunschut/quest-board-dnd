## Component-Level Implementation Notes

(Supplementary detail for planner/executor — not a template section, but load-bearing for this phase's specific views.)

### Create.cshtml / Create.Mobile.cshtml
- Add a new `<div class="mb-3">` block immediately after the existing `Name` field block, before the `<hr>`.
- Structure: `<label asp-for="BoardType" class="form-label">Board Type</label>` → `<select asp-for="BoardType" class="form-select">` containing a hardcoded `<option value="">-- Select Board Type --</option>` followed by `Html.GetEnumSelectList<BoardType>()` items (per RESEARCH.md Pattern 2) → `<span asp-validation-for="BoardType" class="text-danger"></span>` → `<small class="form-text text-muted">This cannot be changed after creation.</small>`.
- No icon on the field itself (D-02 — no explanatory text means no decorative icon either; keep parity with the plain `Name` field above it).

### Edit.cshtml / Edit.Mobile.cshtml
- Add the same field block shape as Create, but the `<select>` carries the `disabled` attribute and is populated with the single current value only (or the full enum list with `disabled` — either renders non-interactive; using `Html.GetEnumSelectList<BoardType>()` with `disabled` on the `<select>` itself is simplest and matches D-05's "visually consistent with the Create form's dropdown, but non-interactive").
- Because `disabled` selects do not postback their value, no hidden `<input type="hidden" asp-for="BoardType">` should be added — per D-06/RESEARCH.md Anti-Patterns, `BoardType` must never flow back into the POST body at all, disabled or otherwise.
- No permanence-warning `<small>` needed on Edit (D-04 specifies it for the Create form; the Edit form's `disabled` state is self-evidently permanent).

### Index.cshtml / Index.Mobile.cshtml
- Desktop: new `<th scope="col">Board Type</th>` between `Name` and `Members` headers; corresponding `<td>` renders the badge via `@if (item.BoardType == BoardType.OneShot) { <span class="badge bg-primary"><i class="fas fa-dice-d20 me-1"></i>One-Shot</span> } else { <span class="badge bg-info text-dark"><i class="fas fa-book-open me-1"></i>Campaign</span> }` (two-value if/else, no else-if chain needed per RESEARCH.md Pattern 3).
- Mobile: same badge markup inserted into the existing `group-card-mobile` block, directly under the `<strong class="parchment-text">@item.Name</strong>` line and above the "Members: ... · Created: ..." `<small>` line.
- Badge text uses the display-friendly hyphenated form "One-Shot" / "Campaign" (not the raw enum token `OneShot`) — consistent with how `GroupRole.DungeonMaster` is still shown as literal "DungeonMaster" in `Members.cshtml:64` today; since this phase has no existing precedent forcing raw-enum-text, prefer the more readable "One-Shot" for the new badge copy.
