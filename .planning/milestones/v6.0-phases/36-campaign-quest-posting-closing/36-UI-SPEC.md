## Spacing Scale

Declared values (must be multiples of 4): the project does not use a systematic 4px-based spacing scale — it uses Bootstrap's default rem-based utility spacing (`0.25rem`/`0.5rem`/`0.75rem`/`1rem`/`1.5rem`/`2rem`, i.e. 4/8/12/16/24/32px at 16px root) plus fixed-px values in `quests.css` for the fantasy card layout. This phase does not introduce a new scale — it reuses existing spacing exactly as it appears in the sections being touched.

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px (`0.25rem`) | Icon-to-text gaps (`me-1`, `me-2` classes) |
| sm | 8px (`0.5rem`) | Compact element spacing (`mb-2`, gap in `.d-flex.gap-2`) |
| md | 16px (`1rem`) | Default card padding, `.quest-content` padding, button spacing |
| lg | 24px | Not used at this granularity in touched views |
| xl | 32px (`2rem`) | `.quest-log-header` padding |

Exceptions: `.quest-content` uses percentage-based inset (`top:10%; left:12%; right:12%; bottom:12%`) rather than px/rem because it's absolutely positioned over a variable-aspect-ratio poster background image — this is existing, out-of-scope-to-change layout mechanics, not a new exception introduced by this phase.

## Color

This phase does not introduce new brand colors. It must reuse the existing state-color convention already locked in Phase 35 for `BoardType` badges, extended consistently to the Close/Reopen control and the campaign seal state:

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | Dark parchment/notice-board backdrop (`body` background image, near-black overlay panels `rgba(0,0,0,0.85–0.98)`) | Page background, Quest Log card surfaces |
| Secondary (30%) | Parchment poster art (`.fantasy-quest-card` background images) + `#F4E4BC` (warm parchment text) | Quest board cards, Quest Log card text |
| Accent (10%) | `#ffc107` (Bootstrap `warning`/amber-gold) | Quest Log borders/glow, recap badges, headings — **existing** accent; this phase adds one new accent use: the **Reopen** button (`btn-warning`), matching the existing `Open Quest` button exactly |
| Destructive/Close | `#0d6efd`-adjacent is NOT used for Close — reuse `btn-secondary` (grey, Bootstrap `#6c757d`) for the **Close** action, since closing a campaign quest is a reversible, non-destructive status toggle (per D-02/D-03: matches the "quiet" tone of `Open Quest`'s sibling actions, not a `btn-danger` action) | Close Quest button only |

Accent (`#ffc107`/amber) reserved for: Quest Log ambient theming (existing, untouched) and the new **Reopen Quest** button, matching `Open Quest`'s existing `btn-warning` treatment exactly (same semantic: "bring this back to active"). Do not introduce `btn-danger` (red) for either Close or Reopen — neither is destructive/irreversible, consistent with D-02/D-03's framing of Close/Reopen as a reversible toggle.

**Close button color: `btn-secondary`** (grey) — deliberately quieter than `btn-warning`, signaling "step back/pause" rather than "act now." This mirrors no exact existing precedent (Finalize=`btn-success` green because it's a commit; Open=`btn-warning` amber because it's undoing a commit) — Close has no prior analog since one-shot has no reversible "pause" action. Recommendation: `btn-secondary` for Close, `btn-warning` for Reopen. If the planner/executor prefers stronger visual parity with Finalize/Open's success/warning pairing, `btn-success` (Close) / `btn-warning` (Reopen) is an acceptable alternate — but **grey Close + amber Reopen is the primary recommendation** because Close is not an achievement (unlike Finalize) and using green for it would visually overstate the action.

## Key Design Decision: Campaign Wax-Seal Treatment (resolves D-06)

**Decision: Reuse the exact same wax-seal mechanic, relabeled Open/Closed. No new visual, no new CSS.**

### Rationale
The existing bottom-left seal already encodes a **binary lifecycle state** via presence/brightness (`.open-seal{display:none}` vs `.finalized-seal{brightness+saturate}`), driven by one boolean-ish condition (`IsFinalized`). Campaign's Close/Reopen is the same shape of state — an open quest vs. a closed one — and the user explicitly asked to stay consistent with the wax-seal aesthetic "unless there's a strong reason to diverge." There is no such reason: the interaction is semantically identical (quest is either active on the board, or archived), so introducing a different visual (e.g. a padlock icon, a stamp, a ribbon) would create an inconsistent second visual language for the same underlying concept without adding information value. Reusing the mechanic also means **zero new CSS/assets** — this phase only needs a text/condition change in the Razor view, not new image assets or stylesheet rules.

### Implementation contract
1. **CSS: no changes needed.** `.open-seal` and `.finalized-seal` classes in `quests.css` (lines 237–245) are reused verbatim — they already key off presence/absence of a "hidden" class, not off the literal word "finalized."
2. **Razor condition:** the board card (`Quest/Index.cshtml` line ~84-85) currently branches on `quest.IsFinalized`. For campaign quests, branch on whatever closed-state field the planner/researcher chooses at the entity level (context flags this as an open implementation decision — likely a new `IsClosed` bool per the "Claude's Discretion" note in 36-CONTEXT.md). The CSS class assignment stays the same shape:
   ```
   class="seal-image @((quest.BoardType == BoardType.Campaign ? quest.IsClosed : quest.IsFinalized) ? "finalized-seal" : "open-seal")"
   ```
   (exact field name TBD by planner; the important contract is: **one class swap, same two CSS classes, no third state**).
3. **Alt text only changes for campaign quests** — swap the word "Finalized" for "Closed" when `BoardType == Campaign`, so screen readers announce accurately:
   ```
   alt="@((quest.BoardType == BoardType.Campaign ? (quest.IsClosed ? "Closed" : "Open") : (quest.IsFinalized ? "Finalized" : "Open"))) Quest"
   ```
4. **Random seal image selection** (`Crown Seal.png`, `Eagle Seal.png`, `Flower Seal.png`, `Sigil Seal.png`, chosen by `quest.Id.GetHashCode() % 4`) is **unchanged** for campaign quests — same pool, same per-quest-consistent selection, no new seal artwork needed.
5. **Top-right user-signup seal:** confirmed no work needed — `isUserSignedUp` will always evaluate false for campaign quests (no `PlayerSignups` ever populated), so the `@if (isUserSignedUp)` block naturally never renders. No explicit `BoardType` guard needed here, but the planner may add one defensively for clarity/future-proofing — not required for correctness.
6. **Manage.cshtml / Details.cshtml:** these pages don't currently render the wax-seal image at all (only the board card does) — no changes needed there beyond the existing Close/Reopen button work (D-01/D-02/D-03, covered in Copywriting Contract above).

### Alternatives considered and rejected
- **Different visual entirely (e.g. lock/unlock icon):** rejected — breaks visual consistency with the one-shot board for no functional gain; the DM/party would now need to learn two different "is this active?" visual languages depending on group type.
- **No seal at all for campaign quests (just omit it):** rejected — the seal is the *only* at-a-glance lifecycle indicator on the board card (there's no status text elsewhere on the card face); removing it would regress scannability of the campaign board, which already loses the CR badge and signup-count line per D-04/D-05. The board card needs at least one clear "is this live?" signal remaining.
