<objective>
Make `GroupController.AddMember` (Platform area, SuperAdmin-only) send a notification email when a SuperAdmin adds an *existing* user to a group via the Members page's available-users panel — closing the silent gap that triggered this phase. The fix reroutes `AddMember` through the existing shared `UserService.CreateOrAddToGroupAsync` method (already used by `CreateMember` and `AdminController.CreateUser`) and copies `CreateMember`'s outcome-switch email-dispatch logic, so all three "add user to group" entry points behave identically. A regression test locks the new enqueue behavior in place.

Purpose: A confirmed user added to a new group currently receives no "you've been added to {group}" email. This is inconsistent with the two other add-to-group entry points and was reported as a real incident.
Output: Modified `AddMember` action; a capturing job-client spy in the test harness; regression tests asserting the correct email job is enqueued per outcome.
</objective>
