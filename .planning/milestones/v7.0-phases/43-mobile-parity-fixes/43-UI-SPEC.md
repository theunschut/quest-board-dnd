## Fix 1: iOS Safari Fixed-Background (MOBILE-01)

Purely structural CSS fix — **no new visual design**. The rendered result must look pixel-identical to today's (intended) appearance; only the technique changes.

### Contract

| Property | Value |
|----------|-------|
| Files touched | `QuestBoard.Service/wwwroot/css/site.css` (body rule at lines 377–383, `background-attachment: fixed` on line 382) and `QuestBoard.Service/wwwroot/css/mobile.css` (body rule at lines 13–22, `background-attachment: fixed` on line 21) |
| Technique | Replace `background-attachment: fixed` with a `position: fixed` layered element (`::before` pseudo-element or dedicated `<div>`), sized `100vh`/`100dvh` (prefer `100dvh` with `100vh` fallback for older iOS Safari versions that don't support dynamic viewport units), `z-index: -1`, carrying the same `background-image` / `background-size: cover` / `background-position: center` / `background-repeat: no-repeat` declarations currently on `body` |
| Explicitly rejected approach | `-webkit-` prefix hacks (no such prefix exists for this property) and JS scroll-listener workarounds — do not use either |
| Stacking safety | Confirmed no existing z-index in `site.css` uses a negative value today (existing values: 10, 297, 499, 1025, 1030, 1493, 1645 — all positive) — a `z-index: -1` background layer will not collide with any existing stacking context |
| Background image asset | `/images/Notice Board-blank2.jpg` (identical in both files today — do not change the asset) |
| Scope | Fix both `site.css` and `mobile.css`, even though `MobileDetectionMiddleware` routes all iPhone/iPad user agents to `mobile.css` today (per D-02) — `site.css`'s copy is currently unreachable by iOS Safari but is fixed anyway for consistency, at no extra cost |
| Visual acceptance bar | No visible regression on desktop or existing mobile breakpoints — background must still appear to "cover," stay centered, and not repeat, exactly as it does today on non-iOS browsers |

### Interaction Contract

| State | Behavior |
|-------|----------|
| Scroll (any browser) | Background image visually stays in place (does not move/scroll with content) — this is the entire fix; verify by scrolling past one full viewport height |
| Page resize / orientation change | Background layer must still cover full viewport (this is why `100dvh`/`100vh` sizing matters, particularly for iOS Safari's dynamic toolbar-driven viewport height changes) |

### Verification Requirement (non-visual, but load-bearing for this contract)

Per D-03/D-04 and PITFALLS.md Pitfall 5: this fix must be verified on a real iOS Safari session (physical iPhone over LAN), not devtools "iPhone" emulation — desktop and emulated environments cannot reproduce the underlying WebKit compositing bug and will falsely appear to pass either way. Record exact device/iOS version/method in verification evidence.

## Fix 2: Session Recap Badge on Mobile Quest Log (MOBILE-02)

Pixel-for-pixel port of an existing, shipped desktop component. No new visual design — the values below are copied directly from desktop's implementation.

### Markup Contract

Source (desktop, unchanged): `QuestBoard.Service/Views/QuestLog/Index.cshtml:72-77`

```html
@if (!string.IsNullOrWhiteSpace(quest.Recap))
{
    <span class="recap-badge">
        <i class="fas fa-scroll me-1"></i>Session Recap Available
    </span>
}
```

Target: `QuestBoard.Service/Views/QuestLog/Index.Mobile.cshtml` — insert this exact block as a new row inside the `.quest-log-item` card (currently lines 39–56), placed immediately after the existing DM-name `<small>` element (after line 55, before the closing `</div>` on line 56).

Resulting mobile card row order (top to bottom): title + CR badge row → completed date row → DM name row → **[new] recap badge row**.

Label text: keep the full phrase **"Session Recap Available"** — no shortened/truncated variant (D-06). The mobile `.quest-log-item` card is full-viewport-width minus small padding, with room for the full phrase + icon on its own row.

Icon: `fa-scroll` with `me-1` spacing — identical to desktop, no substitution.

Guard condition: identical to desktop — `!string.IsNullOrWhiteSpace(quest.Recap)`. No new guard logic; do not add any additional condition (e.g. board-type check) not present in the desktop source.

### Visual/Style Contract

Source (desktop, unchanged): `QuestBoard.Service/wwwroot/css/quests.css:891-899`

```css
.quest-log-card-footer .recap-badge {
    background: rgba(255, 193, 7, 0.2);
    padding: 0.25rem 0.75rem;
    border-radius: 12px;
    border: 1px solid rgba(255, 193, 7, 0.4);
    font-size: 0.85rem;
    display: inline-block;
    margin-bottom: 0.5rem;
}
```

Plus the shared text-color rule that also applies to `.recap-badge` (desktop, `quests.css:884-889`):

```css
.quest-log-card-footer .recap-badge,
.quest-log-card-footer .view-details-text,
.quest-log-card-footer span {
    color: #F4E4BC !important;
    text-shadow: 1px 1px 2px rgba(0, 0, 0, 0.9);
}
```

Target: add an equivalent `.recap-badge` rule to `QuestBoard.Service/wwwroot/css/quest-log.mobile.css` (no equivalent class exists there today — this is net-new CSS in that file only). Port the exact values below; do not adapt, lighten, or otherwise "mobile-ize" them (D-05).

| Property | Value | Source |
|----------|-------|--------|
| Background | `rgba(255, 193, 7, 0.2)` (amber/gold, 20% opacity) | `quests.css:892` |
| Border | `1px solid rgba(255, 193, 7, 0.4)` | `quests.css:895` |
| Border radius | `12px` (pill shape) | `quests.css:894` |
| Padding | `0.25rem 0.75rem` | `quests.css:893` |
| Font size | `0.85rem` | `quests.css:896` |
| Display | `inline-block` | `quests.css:897` |
| Margin bottom | `0.5rem` | `quests.css:898` |
| Text color | `#F4E4BC` (parchment) with `text-shadow: 1px 1px 2px rgba(0, 0, 0, 0.9)` | `quests.css:887-888` |
| Icon | FontAwesome `fa-scroll`, `me-1` spacing, inherits badge text color | `Index.cshtml:75` |

Explicitly rejected style: mobile's plainer Bootstrap-badge look (the style currently used for the adjacent CR badge, `<span class="badge bg-secondary ...">`) — do NOT reuse that class or pattern for the recap badge (D-05).

Coexistence requirement: the new `.recap-badge` rule must sit alongside — not replace or conflict with — `quest-log.mobile.css`'s existing `.quest-log-item` glass-card conventions (`rgba(255,255,255,0.15)` background, `backdrop-filter: blur(15px)`, parchment text `#F4E4BC` with dark text-shadow on `small`/`.text-muted`). The badge's own background/border/color values take precedence locally (same specificity pattern desktop already uses via its own `.recap-badge` class scoping).

### Layout / Placement Contract

| Property | Value |
|----------|-------|
| Position in card | New standalone row, below the DM-name row, as the last element in the card (mirrors desktop's card-footer placement — recap badge is the last item shown before the "click to view details" affordance, per D-07) |
| Row wrapping | Block-level row (own line), not inline with the DM-name text — matches desktop's separate `<span class="recap-badge">` in its own footer flow |
| Conditional rendering | Only renders when `quest.Recap` is non-empty — absent from cards with no recap, no placeholder/empty-state shown in its place |
| Spacing to next element | `margin-bottom: 0.5rem` (inherited from the ported rule) provides the gap; no additional spacing needed since this is the last row in the card |

### Copywriting Contract

| Element | Copy |
|---------|------|
| Badge label | "Session Recap Available" (verbatim, full phrase, no truncation — D-06) |
| Icon | `fa-scroll` (no change) |

No new empty-state, error-state, CTA, or destructive-action copy is introduced by this phase — both fixes are visual/structural parity fixes with no new user-facing text beyond the ported badge label above.
