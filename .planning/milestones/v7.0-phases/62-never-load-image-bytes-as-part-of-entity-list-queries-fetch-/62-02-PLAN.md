<objective>
Fix the single highest-risk item in Phase 62 (RESEARCH.md Pitfall 1, HIGH-risk assumption A2): once Plan 01 removes `.Include(ProfileImage)` from `GetCharacterWithDetailsAsync`/`GetContactWithDetailsAsync`, the round-tripped Domain model's `ProfilePicture`/`ContactImageData` byte[] is permanently null on the write path. `CharacterService.UpdateAsync`/`ContactService.UpdateAsync` currently pass that (now-null) value straight to `UpdateWithProfileImageAsync`, whose `ApplyProfileImage` treats a null original as "delete the image" — so every "edit without changing the photo" would silently WIPE the stored image (real data loss). Fix both services' "no new upload" branch to re-fetch the current original bytes fresh via the already-existing `GetCharacterOriginalPictureAsync`/`GetContactOriginalImageAsync`, exactly mirroring how Phase 45-02 already fixed the analogous cropped-image branch in these same methods.

Purpose: Prevent data-loss regression introduced by Plan 01's read-side change. This task is MANDATORY even though CONTEXT.md's `<code_context>` states "services are no longer in scope" — that statement is confirmed incomplete (RESEARCH.md Pitfall 1, verified against current code).

Output: Both `UpdateAsync(model, hasNewOriginalUpload, newCroppedImageData, token)` overloads re-derive the original bytes from a fresh repository fetch on the no-upload branch; unit tests prove an unrelated-field edit preserves the original image.
</objective>
