<objective>
Fix the one independently-exploitable PlayerSignup leak and harden the incidentally-safe paths (D-12/D-13). `QuestController.RemovePlayerSignup` is `[Authorize(Policy = "AdminOnly")]`, which validates the caller is an Admin in their own active group — but it never checks the target signup's parent Quest belongs to that same group, so a group-A Admin can delete a signup on a group-Z quest by ID. This is the same "check caller, not target" root cause as the DungeonMasterController finding. Add a target-Quest group-membership check returning 404 for cross-group targets, and add regression coverage documenting the other three repository methods' caller-dependent safety.

Purpose: Close the real vulnerability (Admin cross-group deletion) and lock in the incidental safety of the remaining paths with a test so a future refactor that skips caller-side pre-validation fails loudly.
Output: A Quest-including lookup, the RemovePlayerSignup fix, and extended PlayerSignupRepositoryTests. (The QuestBoardContext.cs comment correction for PlayerSignup is handled by plan 49-01, which owns that file.)
</objective>
