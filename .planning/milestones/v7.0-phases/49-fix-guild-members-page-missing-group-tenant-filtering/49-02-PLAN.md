<objective>
Close the DungeonMasterController cross-group leak (D-06 through D-09): `Profile`, `EditProfile` (GET and POST), and `GetDMProfilePicture` never check that the target user `id` is a member of the viewer's active group. The POST path is the most severe — it lets an Admin in group A overwrite the bio/profile picture of a DM in unrelated group Z. Add a target-group-membership check using the existing `IUserService.GetGroupRoleByIdAsync` primitive, returning 404 (not 403) for cross-group targets and for SuperAdmin-with-no-active-group.

Purpose: The `AdminOnly`/`DungeonMasterOnly` policies and the existing same-tenant Forbid() only validate the CALLER's role in their own group — never the TARGET user's group. This is the same "check caller, not target" root cause as the PlayerSignup finding.
Output: A membership check on four controller actions plus integration-test coverage. No schema change (D-09a: DungeonMasterProfileEntity stays shared).
</objective>

- `GetDMProfilePicture(int id, ...)`: if `!await IsTargetInActiveGroupAsync(id)` return `NotFound()` before loading the picture bytes.

Return 404 (`NotFound()`), never 403 (`Forbid()`), for the cross-group case — hide cross-tenant existence, consistent with the Characters convention. Do NOT add a GroupId to DungeonMasterProfileEntity or change DungeonMasterProfileService — the profile data is intentionally shared across a DM's groups; this is access control only.
