## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| CDN → browser | Third-party Cropper.js v2.1.1 script loaded from jsDelivr into every upload page |
| Browser form → Controller | The hidden CroppedPictureFile travels in the ordinary multipart POST |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-46-06-01 | Tampering | Cropper.js CDN script integrity | mitigate | The `<script>` tag uses a pinned version (2.1.1) and a Subresource Integrity `integrity="sha384-..."` hash with `crossorigin="anonymous"` — the browser refuses to execute a tampered/substituted script. The real hash MUST be computed from the actual dist file at implementation time (the placeholder is not acceptable). Matches the app's existing CDN-with-SRI convention. |
| T-46-06-02 | Tampering | Client-submitted CroppedPictureFile | mitigate | Server-side ValidateImagePair (Plan 03) re-validates the submitted crop regardless of any client-side processing — the modal is UX, not a trust boundary. |
| T-46-06-03 | Denial of Service (client) | Accidental modal dismissal losing the pending crop on touch | mitigate | data-bs-backdrop="static" data-bs-keyboard="false" prevent a stray outside-tap/Esc from discarding the pending crop mid-drag; and the hidden input is populated immediately from the default crop (Plan 05) so a value survives any close path. |
| T-46-06-SC | Tampering | npm/pip/cargo installs | mitigate | No package-manager install — Cropper.js is loaded via pinned CDN + SRI (no `npm install`, no postinstall-script surface). Per RESEARCH.md Package Legitimacy Audit, cropperjs is Approved (real npm package, real GitHub source, active maintenance); the one [SLOP] slopcheck result was a documented cross-registry false positive (JS package checked against PyPI). No [ASSUMED]/[SUS] package requiring a blocking legitimacy checkpoint. |
</threat_model>
