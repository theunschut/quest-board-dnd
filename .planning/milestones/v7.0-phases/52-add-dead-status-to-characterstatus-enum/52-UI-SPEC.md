## Color

This phase adds exactly one new semantic color pairing (Dead badge) to an existing single-theme dark-fantasy palette. No 60/30/10 surface repaint applies — the phase does not touch page-level background/surface colors, only a status badge and a card/row dimming treatment.

| Role | Value | Usage |
|------|-------|-------|
| Existing dominant surface (unchanged) | Dark fantasy gradient background (`rgba(20,20,30,0.95)` → `rgba(30,30,40,0.95)` card gradient over page background) | Card surfaces — not touched by this phase |
| Existing accent (unchanged) | `#ffc107` / `#ff9800` (amber/gold) | Reserved for: hover glows, Main-character badge, headings, class/level text — this phase does not add or consume accent color |
| **New: Dead badge background** | Bootstrap `bg-dark` = `#212529` (Details/Details.Mobile inline badge); `rgba(33, 37, 41, 0.95)` (Index/Index.Mobile floating `.dead-badge`, matches `.retired-badge`'s alpha-blended style) | Dead status badge only — all 4 views |
| **New: Dead badge text/icon** | White (`#fff`, Bootstrap default badge foreground on `bg-dark`) | Dead badge label + `fa-skull` icon |
| Existing Retired badge (reference, unchanged) | `bg-secondary` = `#6c757d` / `rgba(108,117,125,0.95)` | Retired status only — confirms Dead must be visually distinct (dark charcoal vs. mid-gray) |
| Existing Active badge (reference, unchanged) | `bg-success` = `#198754` | Active status only |
| Destructive (unchanged) | `btn-danger` red (Bootstrap default) | Delete Character button only — not affected by this phase |

**Accent reserved for:** hover glow effects on character cards, the Main-character badge gradient, page headings, and character class/level text. The Dead badge intentionally does **not** use the accent color — it uses the neutral-dark palette to signal "removed from play," consistent with how Retired uses neutral gray rather than accent gold.

**Contrast verification (accessibility, not previously specified in CONTEXT.md/RESEARCH.md):**
- `bg-dark` (`#212529`) with white text (`#fff`): contrast ratio ≈ 15.8:1 — exceeds WCAG AAA (7:1) for normal text by a wide margin. No contrast risk.
- `.dead-badge` at `rgba(33, 37, 41, 0.95)` background with white text: effectively opaque (95% alpha) over any card background in this dark theme; contrast is equivalent to the solid `#212529` case above. No contrast risk.
- This exceeds the Retired badge's own contrast ratio (`#6c757d` + white ≈ 4.6:1, just above AA minimum), meaning Dead is measurably easier to read than the existing Retired badge — a net accessibility improvement, not a regression.

## Copywriting Contract

D-01 already rules out a dedicated "Mark as Dead" action or confirmation dialog — status is set via the existing Status `<select>` dropdown with no additional confirm step (consistent with how Retired is set today, which also has no confirmation). The items below are the only copy surfaces this phase touches.

| Element | Copy |
|---------|------|
| Status dropdown option (Create/Edit) | `Dead` — literal enum member name via `Html.GetEnumSelectList<CharacterStatus>()`; no `[Display(Name=...)]` attribute exists on the enum today (confirmed in RESEARCH.md) and this phase does not add one, for consistency with `Active`/`Retired` which also render as raw enum names |
| Details/Details.Mobile badge label | `Dead` (icon: `fa-skull`) — matches the existing terse one-word convention (`Active`, `Retired`) |
| Index/Index.Mobile floating badge label | `Dead` (icon: `fa-skull`, no additional text) — matches existing `Retired` / `Main` floating-badge convention of icon + single word |
| Toggle button (Retire/Reactivate) | No copy change — button is hidden entirely when `Status == Dead` per D-03. No "this character is dead" placeholder text or disabled-button label is introduced; the button simply does not render, identical in absence-behavior to how no button renders for other unrelated non-owner views |
| Empty states (My Characters / Guild Roster) | No change — Dead characters still count as existing characters and appear in the grid/list; the empty-state copy ("No Characters Yet", "No Other Characters") is unaffected since it only fires when the underlying collection is empty, not status-dependent |
| Error state | Not applicable — this phase introduces no new failure path, form validation message, or error page. `Dead` flows through the same validated `Status` field as `Active`/`Retired` with identical model-binding behavior |
| Destructive confirmation | Not applicable — no destructive action is added. The existing `Delete Character` confirm dialog (`onsubmit="return confirm('Are you sure you want to delete this character? This action cannot be undone.');")` is unrelated to status and unchanged by this phase |

**Accessibility note (screen readers):** Use icon + visible text label for the Dead badge in all 4 locations (not icon-only) — this matches the existing Retired/Active/Main badge pattern exactly, all of which pair a Font Awesome icon with a visible text label rather than relying on the icon alone. No new `aria-label` is required since the visible text ("Dead") is already accessible to screen readers as part of the badge's text content; Font Awesome icons in this codebase are decorative-only (no `aria-hidden` currently applied to any status-badge icon, and this phase should not introduce an inconsistent one-off `aria-hidden` on just the skull icon — match the zero-`aria-hidden` convention already in place for Retired/Active/Main icons).

## Implementation Reference (from RESEARCH.md — locked, not re-litigated)

The following are already-validated decisions carried forward from CONTEXT.md (D-01–D-05) and RESEARCH.md's Code Examples. This UI-SPEC does not change them; it confirms they satisfy the design-quality dimensions above.

- **Badge markup, Details.cshtml/Details.Mobile.cshtml:** 3-way `if (Dead) {...} else if (Retired) {...} else {...}` — `bg-dark fs-6` + `fa-skull me-1` + "Dead" text.
- **Badge markup, Index.cshtml/Index.Mobile.cshtml:** independent second `if (Status == Dead)` block (not else-if) rendering a `.dead-badge` div with `fa-skull` icon + "Dead" text — Active characters continue to render no badge, per the existing no-else convention for these two views.
- **Card/row class:** `character-dead` (desktop, `guild-members.css`) and `.guild-member-row.dead` (mobile, `guild-members.mobile.css`, following the existing `.retired` mobile-modifier naming precedent) — both applied via ternary/conditional class-string alongside the existing `character-retired` / `.retired` logic, checked for Dead first.
- **Toggle button visibility:** entire `<form asp-action="ToggleRetirement">` block wrapped in `@if (Model.Status != CharacterStatus.Dead)` on both Details.cshtml and Details.Mobile.cshtml — no relabeling, no tri-state cycle, button is simply absent for Dead characters.
- **CSS starting values** (`.character-dead`: `opacity: 0.5; border-color: rgba(33,37,41,0.6); filter: grayscale(60%);` and hover state) are visually distinct from `.character-retired` (`opacity: 0.7; border-color: rgba(108,117,125,0.5)`) — lower opacity + charcoal border + grayscale filter vs. Retired's lighter gray, satisfying D-04's "distinguishable at a glance" requirement. Exact intensity remains implementation discretion per CONTEXT.md; the values in RESEARCH.md's Code Examples section are pre-approved defaults, not requiring re-confirmation.
- **Sort order:** unchanged — Dead falls into the existing "not Active" bucket alongside Retired (D-05). No visual/UI consequence beyond the badge/card treatment above.
