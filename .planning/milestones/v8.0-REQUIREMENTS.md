# Requirements: D&D Quest Board — v8.0 Markdown Support

> **Archived** 2026-07-11 at v8.0 milestone close. All v1 requirements below shipped. See `.planning/v8.0-MILESTONE-AUDIT.md` for the full requirements-coverage audit and `.planning/milestones/v8.0-ROADMAP.md` for phase-level detail.

**Defined:** 2026-07-09
**Core Value:** The quest board must reliably let DMs post quests and players sign up — everything else enhances that loop.

## v1 Requirements — All Shipped

Requirements for the v8.0 milestone. Each maps to roadmap phases.

### Rendering Infrastructure

- [x] **RENDER-01**: Markdown text in any of the 9 target fields is converted to safe, sanitized HTML for display — no raw HTML or script execution is possible regardless of what a user types — *validated: unit-tested XSS payload coverage, Phase 65*
- [x] **RENDER-02**: Page views and the 3 HTML email templates that echo Quest Description use the exact same rendering pipeline — no separate or duplicated rendering logic — *validated: single call-site grep-confirmed, Phase 65*
- [x] **RENDER-03**: A blank line (not a single Enter) is required to start a new paragraph (strict CommonMark) — a deliberate change from today's line-break-preserving plain-text display — *validated, Phase 65*

### Editor UI

- [x] **EDITOR-01**: Editing a free-text field shows a formatting toolbar (Bold, Italic, Heading, List, Link, Blockquote) that inserts Markdown syntax around the current selection or at the cursor — *validated, Phase 66*
- [x] **EDITOR-02**: A Preview button toggles the same input area between the raw-text editor and a rendered-HTML preview, without leaving the page — *validated, Phase 66*
- [x] **EDITOR-03**: While in Preview mode, the rest of the toolbar is disabled (not clickable-but-nonfunctional) — *validated, Phase 66*
- [x] **EDITOR-04**: What a user sees in Preview mode exactly matches how the content actually displays once saved — no formatting surprises after saving — *validated by construction (shared render call), Phase 66*
- [x] **EDITOR-05**: An inline hint near the editor explains that a blank line starts a new paragraph — *validated, Phase 66*
- [x] **EDITOR-06**: The toolbar and editor work identically on desktop and mobile, with icon-only buttons sized for touch (44px+) on mobile, fitting one row with no overflow/scroll mechanism — *satisfied with operator-accepted override: buttons are 44px tall but ~30-34px wide at 320px (CSS cascade tie), no functional breakage — Phase 66*

### Quest Fields

- [x] **QUESTMD-01**: Quest Description supports the Markdown editor on Create/Edit/Follow-Up forms (desktop + mobile) and renders as formatted HTML wherever displayed (board card, Details, Manage) — *validated, Phase 66; QuestLog Details raw-render gap found and fixed at milestone close, commit `222b9a3d`*
- [x] **QUESTMD-02**: Quest Rewards supports the Markdown editor and renders as formatted HTML on Details/QuestLog — *validated, Phase 67*
- [x] **QUESTMD-03**: Quest Recap (via the EditRecap form) supports the Markdown editor and renders as formatted HTML on Details/QuestLog — *validated, Phase 67*

### Character Fields

- [x] **CHARMD-01**: Character Description supports the Markdown editor and renders as formatted HTML on Character Details — *validated, Phase 68*
- [x] **CHARMD-02**: Character Backstory supports the Markdown editor and renders as formatted HTML on Character Details — *validated, Phase 68*

### Contact Fields

- [x] **CONTACTMD-01**: Contact Description supports the Markdown editor and renders as formatted HTML on Contact Details/Index — *satisfied with operator-accepted override: Index intentionally has no Description preview (D-01), matching Character's Phase 68 Index precedent — Phase 69*
- [x] **CONTACTMD-02**: Each Contact Note supports the Markdown editor and renders independently as formatted HTML — one author's formatting never bleeds into another note — *validated, Phase 69*

### DM Profile & Shop

- [x] **PROFILEMD-01**: DM Profile Bio supports the Markdown editor and renders as formatted HTML on the DM Profile page — *validated, Phase 70*
- [x] **PROFILEMD-02**: Shop Item Description supports the Markdown editor and renders as formatted HTML on Shop Index/Details/Manage — *validated, Phase 70*

### Email Integration

- [x] **EMAILMD-01**: Quest Description renders as formatted HTML (not raw Markdown syntax) in the Quest Finalized, Session Reminder, and Waitlist Promoted emails — *validated, Phase 67*
- [x] **EMAILMD-02**: A recipient viewing these 3 emails in real Outlook desktop or Gmail webmail sees correctly formatted content (visible bullets, intact styling) — not broken or missing formatting — *satisfied with operator-accepted override: Gmail confirmed for Quest Finalized directly, Session Reminder/Waitlist Promoted accepted via shared-engine grounds; real Outlook desktop deferred to production access — see `71-VERIFICATION.md` overrides*
- [x] **EMAILMD-03**: A recipient can read the full quest description in these emails even when formatted with headings/lists/blockquotes — content is not silently clipped by a fixed-height card — *validated: server-side block-boundary truncation, unit/integration tested; a real card-overflow bug found and fixed during Phase 71*

## Out of Scope

Explicitly excluded. Documented to prevent scope creep.

| Feature | Reason |
|---------|--------|
| Inline code / code block toolbar button | No dice-notation or code use case anywhere in these 9 fields |
| Image embed button (`![alt](url)`) | Duplicates the existing Cropper.js structured photo-upload pipeline; would introduce unmoderated external-image loading |
| Table button / GFM table promotion | Painful to hand-edit as pipe-delimited text in a bare mobile textarea; none of the 9 fields are long-form documents |
| @Mention / #Ref toolbar buttons | No analog in this app's data model; would require new notification infrastructure |
| Task list / interactive checkboxes | Interactive checkboxes have no equivalent in static HTML email — a real architectural mismatch, not just added complexity |
| Side-by-side / split-pane preview | Contradicts the locked single-toggle-in-place design; confirmed wrong for mobile by both EasyMDE's own `no-mobile` class and Discourse's UX team |
| Fullscreen editing mode | None of the 9 fields are long-form documents |
| TradeItem Description/Notes | Entity exists but is not wired into any controller or view — dead code, out of scope entirely |
| UserTransaction.Notes | Property exists but is never populated anywhere in the codebase — dead code, out of scope entirely |
