# Milestone v8.0: Markdown Support

**Status:** ✅ SHIPPED 2026-07-11
**Phases:** 65-71
**Total Plans:** 26

## Overview

Let users write and view formatted text (bold, lists, headings, links, blockquotes) in all 9 free-text fields across the app — Quest Description/Rewards/Recap, Character Description/Backstory, Contact Description/Notes, DM Profile Bio, Shop Item Description — via a shared Markdown editor + toolbar + Preview toggle, replacing the old line-break-preserving plain-text display with strict CommonMark paragraph rules. A single, unit-tested sanitizing rendering pipeline (Markdig + HtmlSanitizer) was built once and reused identically by every page view and by the 3 HTML email templates that quote Quest Description.

## Milestone Summary

**Key Decisions:**

- Markdig extensions composed individually via explicit `Use*()` calls, never `.UseAdvancedExtensions()` — avoids the bundled `UseGenericAttributes()` attribute-injection XSS vector
- One shared Markdig parse feeds two immutable `HtmlSanitizer` profiles (web keeps `<img>`, email strips it) rather than two separate parses
- `POST /markdown/preview` calls the exact same `RenderToHtml(text, Web)` path saved-page display uses — guarantees Preview-matches-saved by construction
- `IMarkdownService.ExtractPlainText()` is the single mechanism for every plain-text card/teaser preview, preventing raw Markdown syntax from leaking into surfaces that predate the Markdown-authoring change
- `RenderEmailHtml` is a dedicated third rendering path (inline styles + MSO fallback + block-boundary truncation) since email clients can't load external CSS or reliably support `overflow`/scroll
- Real Outlook desktop / full 3-template Gmail confirmation deferred to production access via an explicit, structured `overrides`/`deferred` record rather than silently marked done

**Issues Resolved:**

- Critical: Quest Finalized email wrapped Markdown in `<p>`, silently dropping styling for multi-paragraph Description (Phase 66)
- Cross-plan integration bug: `EditRecap`'s cross-folder partial reference 404'd once merged (Phase 67)
- App-wide EasyMDE-to-textarea silent-submission-failure gotcha (Phase 69)
- Table-based email layout `overflow:hidden` doesn't reliably clip in real clients (Phase 71)
- Truncation logic bugs: single over-budget first block rendered an empty card; pathological-nesting fallback bypassed the character budget (Phase 71)
- QuestLog Description raw-rendering gap (deferred at Phase 67 as WR-02) — re-surfaced by the milestone-close audit's cross-phase integration check and fixed before ship (commit `222b9a3d`)

**Issues Deferred:**

- Real Outlook desktop verification for all 3 quest email templates (EMAILMD-02) — never tested; deferred to first production deployment
- Full independent Gmail confirmation for Session Reminder and Waitlist Promoted (only Quest Finalized was independently sent/opened) — accepted on shared-rendering-engine grounds

**Technical Debt Incurred:**

- Minor Info-level code-review findings left unactioned: `MarkdownRenderTarget` enum placement, `start`/`style` attributes not allowlisted (Phase 65); `.quest-description-mobile`/`.quest-description-box` class duplication (Phase 67); email footnote/`<dl>` styling gap, inconsistent null-forgiving operator usage (Phase 71)
- Pre-existing, unrelated findings surfaced by Phase 68's code review while touching Character views: `classIndex` empty-list guard missing from `Edit.cshtml`'s "Add Another Class" script; dead `isEdit` branch in `Create.cshtml`; `markdown-editor.css`'s mobile-only touch-target rules leaking to desktop
- Nyquist `VALIDATION.md` was never produced for Phases 67, 68, or 70 (discovery-only tooling gap, not a code defect)
