# Phase 34.3 Plan 04: AccountController Fail-Soft Role Badge Migration Summary

**Migrated AccountController's Profile/Edit GET role badges (4 inline `IsInRoleAsync` calls) to `GetEffectiveGroupRoleAsync`, using a fail-soft null-ActiveGroupId guard instead of the fail-hard `RequireActiveGroupId()` pattern used elsewhere in this phase, since `/Account` is exempt from `GroupSessionMiddleware`.**

## Decisions Made
- Followed the plan's exact fail-soft shape (`GroupRole? role = null; if (activeGroupContext.ActiveGroupId is { } groupId) { role = await userService.GetEffectiveGroupRoleAsync(User, groupId); }`) rather than the fail-hard `RequireActiveGroupId()` used in plans 02/03, per D-04 and the plan's explicit Pitfall 3 warning — `/Account` is exempt from `GroupSessionMiddleware` so a null `ActiveGroupId` is a real, reachable state here, not an invariant violation.
- Passed the controller's inherited `User` (`ClaimsPrincipal`) to the helper rather than the domain `user` model already fetched via `userService.GetUserAsync(User)`, matching the helper's declared `GetEffectiveGroupRoleAsync(ClaimsPrincipal, int)` signature exactly as the plan's interface note specified.
