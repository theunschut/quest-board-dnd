# Phase 34.3 Plan 03: QuestLogController and DungeonMasterController GetEffectiveGroupRoleAsync Migration Summary

**Migrated all 5 remaining inline `IsInRole`/`IsInRoleAsync("Admin")` call sites in `QuestLogController` and `DungeonMasterController` to the shared `GetEffectiveGroupRoleAsync` helper, fixing the recap-edit link and DM-profile edit link for non-owner Admins.**

## Accomplishments
- `QuestLogController` now injects `IActiveGroupContext`; `Details.ViewBag.CanEditRecap` and `UpdateRecap`'s admin check both resolve `GroupRole.Admin` via `GetEffectiveGroupRoleAsync`, fixing the "QuestLog recap-edit link disappears" symptom for Admins who are not the quest's DM
- `DungeonMasterController` now injects `IActiveGroupContext`; `Profile.CanEdit` and both `EditProfile` GET/POST `Forbid()` guards resolve `GroupRole.Admin` the same way, fixing the DM-profile edit link/access for Admins who are not the profile owner
- Anonymous-reachable sites (`QuestLogController.Details`, `DungeonMasterController.Profile`) preserve their `currentUser != null` short-circuits so `RequireActiveGroupId()` is never evaluated for unauthenticated visitors
- Ownership comparisons (`currentUser.Name == quest.DungeonMaster?.Name`, `currentUser.Equals(user)`, `currentUser.Equals(targetUser)`) were left untouched inline at every site — only the role term was replaced

## Files Created/Modified
- `QuestBoard.Service/Controllers/QuestBoard/QuestLogController.cs` - Added `IActiveGroupContext` constructor param; `Details.CanEditRecap` and `UpdateRecap`'s admin check now call `GetEffectiveGroupRoleAsync(User, activeGroupContext.RequireActiveGroupId()) == GroupRole.Admin`
- `QuestBoard.Service/Controllers/DungeonMaster/DungeonMasterController.cs` - Added `IActiveGroupContext` constructor param; `Profile` computes `GroupRole? role` inside a `currentUser != null` guard before the object initializer; both `EditProfile` guards hoist `var role = await userService.GetEffectiveGroupRoleAsync(...)` above the `Forbid()` check

## Decisions Made
- `DungeonMasterController.Profile`'s `CanEdit` could not call the async helper inline inside the `DMProfileViewModel` object initializer (C# does not allow `await` inside an object-initializer expression), so the role is resolved into a local `GroupRole? role` variable in a `currentUser != null`-guarded block immediately before the initializer, matching the plan's specified approach.
- Both `EditProfile` GET and POST guards had byte-for-byte identical surrounding text (`targetUser == null) return NotFound();` through the `Forbid()` block), so a single `replace_all` Edit applied the same transformation to both sites in one operation.
