## Performance Fixes: Implement vs. Benchmark-First

| Option | Description | Selected |
|--------|-------------|----------|
| Benchmark first, fix only if needed | Matches CONCERNS.md's conditional thresholds and current small-scale reality | |
| Implement composite index now, benchmark-gate the rest | Index is cheap/low-risk regardless of scale; gate shop-query projection and Hangfire batching | |
| **Implement everything now** | Front-load all performance work for future-proofing | ✓ (refined below) |

**User's choice:** "Implement everything now" for future-proofing — refined through follow-up discussion into a more nuanced split (see below).
**Notes:**
- Claude flagged a naming collision between "Hangfire job-queue batching" (in scope) and "email digest batching" (deferred per D-02) before proceeding, to avoid accidentally reopening the deferred item.
- User explained the digest-batching deferral is actually structural, not just low-priority: a player can't be signed up for two same-day quests under current rules, so the scenario digest batching addresses cannot occur.
- User then questioned whether Hangfire job-queue batching (SessionReminderBatchJob) is a real problem at all, given each job is small/fast (just an email send). Claude agreed and explained: Hangfire's fan-out pattern (fetch once, N small independently-retried jobs) is the textbook-correct pattern for this workload — not something to consolidate. Consolidating into one big job would lose parallel processing and per-job retry isolation, and would need custom idempotency/partial-failure handling to match what Hangfire already gives for free. The real future-scale lever is the *fetch* query (composite index, D-07), not job count.
- User asked about caching frequently-read data (app is the only DB reader/writer, so no "out of sync" risk). Claude advised against: staleness would undermine the quest board's core "DM posts, players see it immediately" value even for a single writer (in-app write/read timing), and — more seriously for this specific app — an improperly `GroupId`-scoped cache key risks a new cross-tenant data leak, the exact class of bug Phases 27-30 were built to prevent. User accepted and asked to drop the idea entirely (not even as a deferred backlog note).

**Final split (captured in CONTEXT.md D-07 through D-11):**
- Composite index (`Quests(IsFinalized, FinalizedDate)`) — implement now
- Shop Item Queries projection (skip BLOB loading on list views) — implement now
- Hangfire job-queue batching — skip; document as monitored future scaling path
- Digest batching — stays deferred (confirmed structural, not just rare)
- Application-level caching — declined outright, not carried forward
