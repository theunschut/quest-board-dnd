<objective>
Persist ASP.NET Core Session data (specifically `ActiveGroupId`) across app restarts by backing `AddSession` with a SQL Server distributed cache, so logged-in users no longer have to re-pick their group after every deploy.

Purpose: Today `AddSession` falls back to an in-memory store wiped on every restart. The Identity auth cookie survives (Data Protection), but the group selection stored in Session does not — forcing every user to re-pick their group after each deploy. Backing Session with `AddDistributedSqlServerCache` against the existing SQL Server connection fixes this with no new infrastructure.

Output:
- `Microsoft.Extensions.Caching.SqlServer` 10.0.9 added to `QuestBoard.Service.csproj`
- `AddDistributedSqlServerCache` registered (Testing-guarded) before `AddSession` in `Program.cs`
- A raw-SQL EF migration `AddSessionStateTable` provisioning `dbo.AspNetSessionState`

Implements SESSION-01, SESSION-02.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| ASP.NET Core app → SQL Server | Session data (ActiveGroupId) now persisted to the `dbo.AspNetSessionState` cache table over the existing DB connection |
| Testing environment ↔ real dev SQL Server | Integration tests must NOT reach the real `localhost` SQL Server for session storage |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-33-03 | Tampering / Information Disclosure | `AddDistributedSqlServerCache` in Testing env | mitigate | Guard registration with `if (!builder.Environment.IsEnvironment("Testing"))`; fall back to `AddDistributedMemoryCache()` in Testing so `dotnet test` never writes session rows into the real dev `QuestBoard` database (RESEARCH Pitfall 2) |
| T-33-04 | Tampering | Cache key collisions from wrong collation | mitigate | `Id` column uses `COLLATE SQL_Latin1_General_CP1_CS_AS` (case-sensitive) per official DDL — prevents cache keys differing only by case from colliding |
| T-33-05 | Denial of Service | Session cache table unbounded growth | accept | Built-in `ExpiredItemsDeletionInterval` (30-min default) purges expired rows automatically; no Hangfire cleanup job needed (D-04). Low-value, single-tenant-scale table |
| T-33-SC | Tampering | nuget install (`Microsoft.Extensions.Caching.SqlServer`) | mitigate | Package is first-party Microsoft, part of `dotnet/aspnetcore`, verified OK in RESEARCH Package Legitimacy Audit — no `[ASSUMED]`/`[SUS]`/`[SLOP]` verdict; no human checkpoint required |

ASVS mapping: V3 Session Management — storage durability change only; expiration/invalidation posture unchanged (existing `AddSession(IdleTimeout=24h)` + `ExpiredItemsDeletionInterval` satisfy V3). No high-severity threats introduced.
</threat_model>
