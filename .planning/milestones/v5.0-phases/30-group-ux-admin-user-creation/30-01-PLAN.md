<objective>
Build the group-picker foundation: a new service method to fetch a user's own groups, the `GroupPickerController` that runs the post-login group-context lifecycle, the picker view (desktop + mobile), its stripped-down layout, and the `ActiveGroupName` session constant.

Purpose: This is the entry point of the multi-tenancy UX loop. After login (wired in plan 30-02), every authenticated user passes through `/groups/pick`, which either auto-selects their single group or presents a picker. Selecting a group establishes the session context that the global query filter (Phase 28) and all downstream pages depend on.

Output: `GroupPickerController` (Index GET + SelectGroup POST), `GroupPickerViewModel`, `GroupPicker/Index.cshtml` + `.Mobile.cshtml`, `_Layout.GroupPicker.cshtml`, `IGroupService.GetGroupsForUserAsync` (+ repo impl), and `SessionKeys.ActiveGroupName`.

Implements D-02, D-03, D-05, D-06, D-07, D-08, D-16, D-18 — covers UX-01, UX-02, UX-03, UX-04.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser → GroupPickerController | Authenticated user submits group selection (groupId, returnUrl) — both are untrusted form input |
| session → query filter | ActiveGroupId in session drives Phase 28 global query filter; tampering would expose another group's data |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-30-01 | Spoofing (open redirect) | GroupPickerController.Index / SelectGroup returnUrl | mitigate | Validate returnUrl with `Url.IsLocalUrl(returnUrl)` before `Redirect`; fall back to Home/Index otherwise (replicates AccountController.RedirectToLocal) |
| T-30-02 | Tampering (CSRF) | SelectGroup POST | mitigate | `[ValidateAntiForgeryToken]` on SelectGroup + `@Html.AntiForgeryToken()` in every card form |
| T-30-03 | Elevation of Privilege | GroupPickerController access | mitigate | `[Authorize]` ensures authentication before any group selection; controller has no policy so any authenticated user may pick among THEIR OWN groups only (GetGroupsForUserAsync is user-scoped) |
| T-30-04 | Information Disclosure (cross-group enumeration) | Index GET for non-SuperAdmin | mitigate | Non-SuperAdmin loads only `GetGroupsForUserAsync(userId)` — never `GetAllWithMemberCountAsync`; a user cannot see or select a group they do not belong to |
| T-30-05 | Tampering (forged groupId in SelectGroup) | SelectGroup POST groupId | accept | groupId is validated to exist via GetByIdAsync; a non-member who guesses a groupId could set it in session, but the Phase 28 query filter only scopes data — group membership is independently enforced by AdminHandler/DungeonMasterHandler (Phase 29) reading UserGroups.GroupRole, so a non-member sees no privileged data. Low risk at current scale; full per-pick membership re-check deferred. |
| T-30-SC | Tampering | npm/pip/cargo installs | mitigate | No packages installed this plan; RESEARCH.md confirms zero new dependencies |
</threat_model>
