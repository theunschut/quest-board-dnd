## Summary

Reviewed the fail-fast Production email-config guard, the Resend stats client with 429-retry
logic, the CSRF reflection-sweep test, and `AdminController`'s email-related actions. The
Resend retry/pagination logic, `EmailService`'s exception-safe logging, and the CSRF sweep's
core mechanism are sound. However, the new Production fail-fast check in `Program.cs` reads
configuration keys that do not exist anywhere in this codebase — it will either always throw
on Production startup or silently pass regardless of actual email configuration, which is the
opposite of its stated purpose. There is also a silent-failure regression in `AdminController`
that is inconsistent with the same file's own established pattern, plus a stated-scope gap in
the new CSRF coverage test and a couple of test-coverage gaps worth calling out.

## Critical Issues

### CR-01: Production fail-fast email guard reads the wrong configuration section — always throws or never fires

**File:** `QuestBoard.Service/Program.cs:345-359`
**Issue:** The new startup guard checks:
```csharp
if (string.IsNullOrWhiteSpace(app.Configuration["Email:FromEmail"]))
    missingEmailKeys.Add("Email:FromEmail");
if (string.IsNullOrWhiteSpace(app.Configuration["Email:SmtpServer"]))
    missingEmailKeys.Add("Email:SmtpServer");
```
but `EmailSettings` is bound to the configuration section **`EmailSettings`**, not `Email`:
```csharp
// QuestBoard.Domain/Extensions/ServiceExtensions.cs
services.AddOptions<EmailSettings>().BindConfiguration("EmailSettings");
```
`appsettings.json` uses `"EmailSettings": { "SmtpServer": ..., "FromEmail": ... }`, and
`docker-compose.yml` documents the production override pattern as
`EmailSettings__FromEmail=...` / `EmailSettings__SmtpServer=...`. There is no `Email:*` key
anywhere in the codebase.

Consequence: `app.Configuration["Email:FromEmail"]` and `app.Configuration["Email:SmtpServer"]`
will **always** evaluate to null/empty, no matter how `EmailSettings` is actually configured.
This means:
- With correctly configured `EmailSettings__FromEmail`/`EmailSettings__SmtpServer` env vars,
  Production startup will **always throw `InvalidOperationException`** and the app will never
  boot — a total deployment blocker.
- If someone "fixes" this by adding an unrelated `Email:FromEmail` key just to silence the
  check, the guard would pass while `EmailSettings` (the section actually consumed by
  `EmailService`) could still be completely unconfigured — silently defeating the entire
  purpose of the fail-fast check described in the adjacent comment.

**Fix:**
```csharp
if (app.Environment.IsProduction())
{
    var missingEmailKeys = new List<string>();
    if (string.IsNullOrWhiteSpace(app.Configuration["EmailSettings:FromEmail"]))
        missingEmailKeys.Add("EmailSettings:FromEmail");
    if (string.IsNullOrWhiteSpace(app.Configuration["EmailSettings:SmtpServer"]))
        missingEmailKeys.Add("EmailSettings:SmtpServer");

    if (missingEmailKeys.Count > 0)
    {
        throw new InvalidOperationException(
            $"Email delivery cannot function without the following configuration key(s): {string.Join(", ", missingEmailKeys)}. " +
            "Set them via appsettings.json or environment variable overrides before starting in Production.");
    }
}
```
Better yet, resolve `IOptions<EmailSettings>` from `app.Services` and check
`.Value.FromEmail`/`.Value.SmtpServer` directly, so the guard can never drift from the actual
binding path again.

## Warnings

### WR-01: EditUser's email-change branch swallows a null callback URL instead of failing like every sibling action

**File:** `QuestBoard.Service/Controllers/Admin/AdminController.cs:203-208`
**Issue:** `CreateUser` (lines 125-130) and `SendConfirmationEmail` (lines 311-317) both check
`Url.Action(...)` for null, log an error, and surface a user-facing failure message before
enqueuing the email job. `EditUser`'s email-change branch does not follow the same pattern:
```csharp
var callbackUrl = Url.Action("ConfirmEmailChange", "Account",
    new { userId = user.Id, newEmail = model.Email, token = encodedToken }, Request.Scheme);
jobClient.Enqueue<ChangeEmailConfirmationJob>(j => j.ExecuteAsync(model.Email, user.Name, callbackUrl!, CancellationToken.None));
TempData["Success"] = $"A confirmation email has been sent to {model.Email} for {user.Name}. The address will update once confirmed.";
```
If `Url.Action` returns null (e.g. routing misconfiguration), this null-forgives (`callbackUrl!`)
and enqueues a Hangfire job with a null URL — the confirmation email will render a broken link
(or throw inside the job, depending on `ChangeEmailConfirmationJob`'s handling) — while the
admin is told "a confirmation email has been sent," masking the failure entirely.
**Fix:** Mirror the null-check pattern used elsewhere in this same file:
```csharp
var callbackUrl = Url.Action("ConfirmEmailChange", "Account",
    new { userId = user.Id, newEmail = model.Email, token = encodedToken }, Request.Scheme);
if (callbackUrl == null)
{
    logger.LogError("Failed to generate ConfirmEmailChange callback URL for userId {UserId}", user.Id);
    TempData["Error"] = $"Failed to send confirmation email to {model.Email}. Please try again.";
    return RedirectToAction(nameof(Users));
}
jobClient.Enqueue<ChangeEmailConfirmationJob>(j => j.ExecuteAsync(model.Email, user.Name, callbackUrl, CancellationToken.None));
```

### WR-02: New CSRF coverage test's docstring overstates its guarantee — HttpDelete/HttpPut actions are unchecked

**File:** `QuestBoard.IntegrationTests/Security/AntiForgeryTokenCoverageTests.cs:6-11, 29`
**Issue:** The class doc comment claims the sweep proves "every state-changing `[HttpPost]`
controller action carries CSRF protection," but the filter only inspects
`HttpPostAttribute`:
```csharp
var postActions = controllerType
    .GetMethods(BindingFlags.Public | BindingFlags.Instance | BindingFlags.DeclaredOnly)
    .Where(m => m.GetCustomAttribute<HttpPostAttribute>() != null);
```
Several state-changing actions in this codebase use `[HttpDelete]` instead of `[HttpPost]` —
e.g. `AdminController.DeleteUser`, `AdminController.DeleteQuest`,
`QuestController.RemovePlayerSignup` — and are entirely excluded from this sweep. Today these
happen to carry `[ValidateAntiForgeryToken]` manually, but the test provides **zero regression
protection** if a future `[HttpDelete]`/`[HttpPut]`/`[HttpPatch]` action is added without the
attribute — directly contradicting the stated purpose of the test ("prevent CSRF" on
"state-changing" actions).
**Fix:** Broaden the verb filter to cover all state-changing HTTP verbs:
```csharp
var mutatingActions = controllerType
    .GetMethods(BindingFlags.Public | BindingFlags.Instance | BindingFlags.DeclaredOnly)
    .Where(m =>
        m.GetCustomAttribute<HttpPostAttribute>() != null ||
        m.GetCustomAttribute<HttpDeleteAttribute>() != null ||
        m.GetCustomAttribute<HttpPutAttribute>() != null ||
        m.GetCustomAttribute<HttpPatchAttribute>() != null);
```
and update the assertion message/docstring accordingly.

### WR-03: ResendStatsClient logs diagnostics via Console.Error instead of the injected ILogger, including a raw response-body dump

**File:** `QuestBoard.Service/Services/ResendStatsClient.cs:46, 59, 64, 85`
**Issue:** `ResendStatsClient` takes an `ILogger<ResendStatsClient> logger` in its constructor
but uses it exactly once (line 52, the "no response" case). Every other diagnostic/error path
uses `Console.Error.WriteLine` instead:
```csharp
Console.Error.WriteLine($"[Resend] 429 received, retrying in {delay.TotalSeconds}s ...");
...
Console.Error.WriteLine($"[Resend] {(int)response.StatusCode} {response.StatusCode} — {body}");
...
Console.Error.WriteLine($"[Resend] raw: {json[..Math.Min(500, json.Length)]}");
...
Console.Error.WriteLine($"[Resend] Exception: {ex.GetType().Name}: {ex.Message}");
```
This bypasses the app's structured logging pipeline entirely (no log level, no correlation
with the request/Hangfire job, not captured by whatever log sink is configured for
Production). Line 64 in particular unconditionally dumps up to 500 characters of the raw
Resend API response body to stderr on every successful page fetch — this is unbounded
diagnostic noise in Production logs and bypasses the minimum-log-level configuration that
would normally suppress verbose output.
**Fix:** Replace all four `Console.Error.WriteLine` calls with `logger.Log{Warning,Error,Debug}`
calls at appropriate levels, e.g.:
```csharp
logger.LogWarning("Resend 429 received, retrying in {DelaySeconds}s (attempt {Attempt}/{MaxRetries})",
    delay.TotalSeconds, attempt + 1, MaxRetries);
...
logger.LogError("Resend stats fetch failed with {StatusCode}: {Body}", response.StatusCode, body);
...
logger.LogDebug("Resend raw response (truncated): {Json}", json[..Math.Min(500, json.Length)]);
...
logger.LogError(ex, "Resend stats fetch threw an exception");
```
so the log level, sink, and retention policy are consistent with the rest of the app, and the
raw-body dump can be suppressed in Production via standard log-level configuration.

### WR-04: SessionReminderJobTests has no coverage for the `useYesMaybeVoters: true` branch

**File:** `QuestBoard.UnitTests/Services/SessionReminderJobTests.cs` (entire file)
**Issue:** `SessionReminderJob.ExecuteAsync` has a distinct code path when `useYesMaybeVoters:
true` — it looks up the `ProposedDate` matching `FinalizedDate`, early-returns with a warning
log if none is found, and otherwise filters recipients by `Yes`/`Maybe` votes on that date
instead of `IsSelected`. None of the 8 tests in this file pass `useYesMaybeVoters: true`, so
this entire branch — including the "no matching ProposedDate" guard clause the code explicitly
calls out as protecting against a documented pitfall — has zero regression coverage.
**Fix:** Add at least two tests: one exercising the DM manual-trigger path where a Yes/Maybe
voter (not marked `IsSelected`) still receives the reminder, and one exercising the
early-return when `quest.ProposedDates` has no entry matching `FinalizedDate.Date`.

## Info

### IN-01: EditUser's email-equality reassignment on the unchanged branch is a confusing no-op

**File:** `QuestBoard.Service/Controllers/Admin/AdminController.cs:177-182`
**Issue:**
```csharp
var emailChanged = !string.Equals(user.Email, model.Email, StringComparison.OrdinalIgnoreCase);
user.Name = model.Name;
user.HasKey = model.HasKey;
if (!emailChanged)
    user.Email = model.Email;
```
When `emailChanged` is false, `user.Email` and `model.Email` are already equal by definition
(modulo case), so `user.Email = model.Email` only ever normalizes casing — it never
communicates that intent. A reader has to trace the boolean twice to confirm this isn't a bug
where the "changed" and "unchanged" branches got swapped.
**Fix:** Add a one-line comment clarifying intent, e.g. `// Re-assign to normalize casing;
actual email changes go through the confirmation flow below.`

### IN-02: ResendStatsClientTests does not assert on request URL/pagination cursor behavior

**File:** `QuestBoard.UnitTests/Services/ResendStatsClientTests.cs` (entire file)
**Issue:** The `StubHttpMessageHandler` receives the full `HttpRequestMessage` in `SendAsync`
but the tests never inspect `request.RequestUri` or `request.Headers.Authorization`. This means
a regression that dropped the `after={afterId}` pagination cursor, sent the wrong `limit`, or
stopped attaching the `Bearer` token would not be caught by this test file at all — only the
retry-count and record-count behaviors are verified.
**Fix:** Capture the requested URIs (e.g. `List<Uri>` recorded in `StubHttpMessageHandler`) and
assert the second-page request includes `after={id}` from the first page's last record, and
assert the `Authorization` header is present and correctly scoped per-request.
