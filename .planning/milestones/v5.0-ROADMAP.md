# Milestone v5.0: Multi-Tenancy

**Status:** ✅ SHIPPED 2026-07-02
**Phases:** 26–34.3
**Total Plans:** 48

## Overview

Transform the Quest Board from a single-tenant EuphoriaInn app into a generic, rebrandable multi-group platform. Namespace rename, group schema with EF Core Global Query Filters, SuperAdmin role and management area, group-picker UX, admin-only user creation, auth lockdown with public landing page, first-login password flow, session persistence across app restarts, and a closing codebase cleanup/security pass.

## Milestone Summary

**Decimal Phases:**

- Phase 34.1: Security & Bugs — split off Phase 34's cleanup scope, closing security-adjacent Known Bugs / Security Considerations items
- Phase 34.2: Performance & Architecture — split off Phase 34's cleanup scope, closing Tech Debt / Performance / Architecture items
- Phase 34.3: Group Role Authorization Regression Fix — urgent regression discovered during manual pre-ship testing after Phase 34.2, inserted before ship

**Key Decisions:**

- Full rename (EuphoriaInn → QuestBoard) done as a dedicated zero-behavior-change phase before any schema work, to keep the multi-tenancy work itself easier to review
- Shared-database, shared-schema multi-tenancy via EF Core Global Query Filters — no per-tenant database/schema, no third-party multi-tenancy framework
- Per-group roles live in `UserGroups.GroupRole`; `AspNetUserRoles` used only for the system-wide `SuperAdmin` role
- `IActiveGroupContext.ActiveGroupId` is `int?` — `null` means SuperAdmin (see all groups), read from ASP.NET Core Session
- SuperAdmin management area routed at `/platform`, not `/superadmin`
- Self-registration removed entirely — only Admins/SuperAdmin create accounts, which triggers the passwordless first-login welcome-email flow
- Session state moved off the in-memory default onto `AddDistributedSqlServerCache` so `ActiveGroupId` survives app restarts
- Admin-triggered repeatable email-send buttons rate-limited 3/hour per target user; one-shot automated sends stay exempt
- See `.planning/PROJECT.md` Key Decisions table for the full decision log with outcomes

**Issues Resolved:**

- Stale-checkbox drift in REQUIREMENTS.md (GROUP-04/05/06, TENANT-01/02/03/05 shown unchecked despite being complete) — corrected at milestone close
- Group role authorization regression (~20 inline `IsInRole` call sites orphaned by Phase 27's `AspNetUserRoles` cleanup) — caught during manual pre-ship testing and fixed in Phase 34.3 before any production deploy
- `GroupSessionMiddleware`-era session recovery for authenticated users with an expired/missing active group

**Issues Deferred:**

- `GroupSessionMiddleware` redirects on all HTTP verbs including POST — a POST-body data-loss risk on session expiry mid-submission, flagged by code review but not yet fixed (see PROJECT.md Key Decisions)
- Digest batching for session reminders (EMAIL-04/REMIND-02) — still deferred, same-day quests have never occurred in a year of operation
- Profile picture crop/avatar selection (issue #78) — still paused pending SkiaSharp native-lib verification on the deployment host

**Technical Debt Incurred:**

- None new beyond the pre-existing deferred items above; Phase 34/34.1/34.2 were themselves a dedicated cleanup/hardening pass against debt accumulated in v1.0–v4.0
