<objective>
Scope the `AdminController.Users()` management page to the currently active group, and harden the four role-change POST actions in the same controller against out-of-group targets. Today `Users()` calls `userService.GetAllAsync()` with no group filter, leaking every platform user into a group admin's page (a cross-tenant PII leak). Simultaneously, `PromoteToAdmin`/`DemoteFromAdmin`/`PromoteToDM`/`DemoteToPlayer` accept a raw `userId` with no membership check, so a crafted POST can grant/revoke a group role for a user outside the active group even after the list is scoped.

Purpose: Close a live cross-tenant PII read leak (D-01, D-02) and the matching cross-group authorization bypass on the write path (D-04, D-05), and prove both closed with a regression test (D-03).
Output: A group-scoped read method through the full Domain → Repository layering, a membership guard on the four role-change actions, and a cross-group-isolation integration test.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| authenticated group admin → AdminController.Users() (read) | An authenticated Admin/SuperAdmin requests the Users page; the server must return only rows scoped to the caller's active group. Cross-tenant data is untrusted-to-expose here. |
| authenticated group admin → AdminController role-change POSTs (write) | A `userId` submitted in a POST body is attacker-controllable; the UI only renders Promote/Demote buttons for in-group members, but a crafted request can submit any userId. The server must not act on out-of-group targets. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-38-01 | Information Disclosure | `AdminController.Users()` GET | mitigate | Replace `GetAllAsync()` with `GetAllGroupMembersAsync(groupId.Value)` — the page returns only `UserGroups` members of the active group (Task 2). Regression test T-38-03 proves cross-group users are absent (Task 3). Closes the live cross-tenant PII leak. |
| T-38-02 | Elevation of Privilege / Tampering | `PromoteToAdmin`/`DemoteFromAdmin`/`PromoteToDM`/`DemoteToPlayer` POST | mitigate | Insert a membership guard (`GetGroupRoleByIdAsync(userId, groupId.Value) == null` → silent redirect) before `SetGroupRoleAsync`, so a crafted POST cannot create/modify a `UserGroups` row for a user outside the active group (Task 2). Without this, `SetGroupRoleAsync` silently inserts a new membership row for any submitted userId. |
| T-38-03 | Information Disclosure (regression) | Cross-group isolation invariant | mitigate | Dedicated integration test `Users_WhenAdmin_DoesNotShowUsersFromOtherGroups` asserts an out-of-group user never appears on `/Admin/Users`, with an in-group positive control (Task 3). Follows the codebase convention of a regression test per security fix. |
| T-38-SA | Information Disclosure (accepted trade-off) | SuperAdmin view of `Users()` | accept | Scoping applies uniformly with no SuperAdmin exception (D-02). SuperAdmin loses the "see everyone" view on this specific page until Phase 40 ships the dedicated cross-group Platform Members page. User confirmed explicitly; the removed behavior was an unintentional side effect of the bug, not a designed feature, so it does not violate the "no user-facing functionality removed" constraint. Low risk: SuperAdmin can still switch active group via the group picker. |
| T-38-SC | Tampering | npm/pip/cargo installs | accept | No package-manager installs in this phase — all changes reuse existing types and existing test helpers; no new NuGet/npm dependencies added. No supply-chain surface introduced. |
</threat_model>
