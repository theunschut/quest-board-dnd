<objective>
Repurpose the group-admin "Delete" user flow (SAFE-01) so it removes a user from the active group only, instead of hard-deleting the entire account. Rename the button and confirm dialog to accurately describe the reversible group-removal behavior.

Purpose: The current `AdminController.DeleteUser` calls `userService.RemoveAsync(user)`, which hard-deletes the `UserEntity` row — cascading the user out of *every* group plus their characters, DM profile, and player signups, and throwing an unhandled `DbUpdateException` for any user with quest/shop/transaction/trade/reminder history (five `NoAction` FKs). This plan replaces that call with the already-proven `IGroupService.RemoveMemberAsync(groupId, userId)` primitive — the exact call `GroupController.RemoveMember` already makes.

No special-case handling is added for a removal that leaves a user with zero group memberships (D-03): `GroupSessionMiddleware` (Phase 31) already redirects a groupless authenticated user to `/groups/pick` gracefully — this is confirmed as a non-issue, deliberately not a gap to close in this phase.

Output: A safe, group-scoped removal path with accurate UI copy and integration tests proving no account data is lost and no server error is thrown even for users with rich FK history.
</objective>
