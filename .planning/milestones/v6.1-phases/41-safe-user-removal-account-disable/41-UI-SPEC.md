# Phase 41 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none — server-rendered ASP.NET Core MVC + Razor views, no npm/Node front-end tooling, no shadcn/Tailwind. Confirmed: no `components.json`, no `tailwind.config.*`, no `postcss.config.*` in repo. |
| Component library | Bootstrap 5 (existing `wwwroot/css/site.css` + Bootstrap classes: `card`, `table`, `badge`, `btn`, `modal`, `alert`) |
| Icon library | Font Awesome (`fas fa-*`, already used exhaustively across all Admin/Platform views) |
| Font | System default body text; `'Cinzel', serif` reserved for large display headings only (shop header, landing page) — **not** used for this phase's admin/utility tables |
| Visual motif | "Modern card" glassmorphism pattern (`modern-card` / `modern-card-header` / `modern-card-body`) — semi-transparent frosted panel over the app's dark fantasy background, per `CLAUDE.md` UI/UX Design Guidelines |

This phase adds **zero new front-end dependencies**. It extends three existing Razor view families (`Views/Admin/Users.cshtml` + `.Mobile.cshtml`, a new `Areas/Platform/Views/Users/Index.cshtml` + `.Mobile.cshtml`, and `Areas/Platform/Views/Group/Index.cshtml`) using patterns already proven in Phases 38–40. shadcn gate is **not applicable** — this is not a React/Next.js/Vite project.

## Spacing Scale

Declared values (project uses Bootstrap's rem-based spacing utilities, which map cleanly onto an 8px-derived scale at the default 16px root):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px (`me-1`, `gap-1`) | Icon-to-text gaps inside badges/buttons |
| sm | 8px (`me-2`, `gap-2`) | Inline spacing between buttons, icon-to-label spacing |
| md | 16px (`mb-3`, `p-3`, `.75rem`) | Default form-group/alert spacing, card body padding baseline |
| lg | 24px (`mb-4`, `1.5rem` — `modern-card-body` padding) | Card body internal padding, section spacing |
| xl | 32px (`py-5` used for empty states, `g-4` row gutters) | Empty-state vertical padding, two-column row gutter (Members-page precedent) |
| 2xl | 48px | Not used in this phase |
| 3xl | 64px | Not used in this phase |

Exceptions: none. Table action-button rows use Bootstrap's `btn-sm` (compact touch target, ~31px height) consistent with every existing action column in `Users.cshtml`/`Group/Index.cshtml`/`Group/Members.cshtml` — no 44px minimum touch target override needed since this pattern is already shipped and accepted across the whole admin surface.

## Typography

Table is descriptive of the **existing, already-shipped** type scale this phase must match — not new sizes being introduced.

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body / table cell | 16px (Bootstrap `1rem` default) | 400 (regular) | 1.5 |
| Label / small / badge | 14px (`0.875rem`, e.g. `.vote-summary`, badge text) | 400 regular (600 semibold inside `.modern-card .form-label`) | 1.5 |
| Heading (card header `h2`) | 20px (Bootstrap default `h2` inside `modern-card-header`, visually reduced by the card context) | 600 (semibold) | 1.2 |
| Sub-heading (`h5`, e.g. "Current Members" / "Available Users" column headers) | 20px (Bootstrap `h5` = `1.25rem` ≈ 20px) | 600 (semibold) | 1.2 |

Only 2 weights are in use anywhere in this contract: **400 (regular)** for body/table text, **600 (semibold)** for headings, labels, and badges. Do not introduce a third weight.

## Color

This app uses a **dark fantasy theme**, not a light 60/30/10 SaaS palette — the "dominant surface" is a dark background image/gradient, not a light neutral. Restating the 60/30/10 contract in this theme's actual terms:

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | Dark page background (body background image / `#1a1a2e`–`#0f3460` gradient family, already global, unchanged by this phase) | Page background behind all cards |
| Secondary (30%) | `rgba(255,255,255,0.15)` frosted `modern-card` surface, text rendered in `#F4E4BC` (parchment) per `.modern-card` rules | Card body panels, the two Platform/Admin tables this phase touches |
| Accent (10%) | Bootstrap `#ffc107` (`btn-warning` / `text-warning`, amber-gold) | **Reserved for:** the "Create New User" button/modal trigger (matches Phase 40's `Members.cshtml` precedent), search-submit buttons, and any "in-progress/needs-attention" badge state. Do NOT use amber for the new Disable/Enable buttons — see Destructive/semantic row below. |
| Destructive (2nd semantic) | Bootstrap `#dc3545` (`btn-danger` / `bg-danger`) | Reserved for: **"Remove from Group"** button (renamed Delete, D-02) and its confirm-dialog action, and the **"Disable"** account button/badge (an account-suspension action, treated as destructive-class per existing `btn-danger` usage for role demotions/removals elsewhere in `Users.cshtml`) |

Additional semantic colors already established and reused (not new): `btn-success` / `bg-success` (`#198754`) for **"Enable"** (re-enable is a positive/restorative action — mirrors `btn-success` used for "Add" in `Members.cshtml` and "Create Group"/"Create User" elsewhere), `btn-info` (`#0dcaf0`) reserved for informational/neutral actions (Edit, Resend Welcome Email) — not introduced by this phase but present in the same tables.

Accent reserved for: Create New User trigger, search-submit buttons. Never apply amber/warning to Disable, Enable, or Remove-from-Group — those follow the destructive/success semantic pair above, not the accent color.

## Copywriting Contract

Pre-populated directly from CONTEXT.md's locked decisions (D-02, D-13, specifics) — nothing left ambiguous for the executor.

| Element | Copy |
|---------|------|
| Primary CTA (Admin Users page, unchanged) | "Create User" (existing, untouched by this phase) |
| Primary CTA (new Platform Users page entry point, D-06) | "Manage Users" button in `Group/Index.cshtml` header bar, icon `fa-users-cog` |
| Renamed action — group removal (D-02) | Button label: **"Remove from Group"** (was "Delete"); icon `fa-user-minus` (matches `Members.cshtml`'s existing "Remove Member" icon choice — do not reuse `fa-trash`, which now misrepresents the action) |
| Confirm dialog — group removal (D-02) | **"Are you sure you want to remove this user from the group? They will keep their account and any other group memberships."** — replaces the old "cannot be undone" hard-delete phrasing, since this action is reversible (re-add via Members page) |
| Disable action button | **"Disable"**, icon `fa-user-lock`, `btn-danger` |
| Disable confirmation | **"Disable this account? They will be signed out and unable to log in until re-enabled. No data will be deleted."** |
| Enable action button | **"Enable"**, icon `fa-user-check` (or `fa-unlock`), `btn-success` |
| Enable confirmation | Not required — re-enable is non-destructive/restorative; no confirm dialog needed (consistent with "Add Member"/"Create User" having no confirm step elsewhere in the app) |
| Status badge — active account | `bg-success` badge, text **"Active"**, icon `fa-check-circle` |
| Status badge — disabled account | `bg-danger` badge, text **"Disabled"**, icon `fa-user-lock` |
| Self-disable guard (D-07) | Disable button/control hidden or disabled with tooltip: **"You cannot disable your own account."** — Claude's discretion whether hidden entirely or shown disabled; recommend disabled+`title` attribute for clarity over silently hiding the control |
| Disabled-login error (D-13, locked exact copy) | **"This account has been disabled. Contact an administrator."** |
| Ordinary temporary lockout (unchanged, locked exact copy) | **"Account locked due to too many failed attempts. Try again in 15 minutes."** |
| Empty state — Platform Users page | Heading: **"No users found."**, icon `fa-users fa-3x text-muted`, matching the existing empty-state pattern in `Users.cshtml`/`Group/Index.cshtml` (`py-5 text-center` block) |
| Success flash — group removal | **"User removed from the group."** — mirrors `Members.cshtml`'s existing `RemoveMember` flash string ("Member removed from the group.") for consistency |
| Success flash — disable | **"Account disabled."** |
| Success flash — enable | **"Account re-enabled."** |
| Error flash — self-disable attempt (defense in depth, server-side guard even though button is hidden/disabled client-side) | **"You cannot disable your own account."** |

Destructive actions in this phase and their confirmation approach:

| Action | Confirmation |
|--------|-------------|
| Remove from Group | JS `confirm()` dialog (existing `deleteUser()` pattern in `Users.cshtml`/`Users.Mobile.cshtml`, function renamed to `removeFromGroup()` and copy updated per D-02) |
| Disable account | JS `confirm()` dialog, same pattern — new `disableUser()` function following the identical `fetch()` + `RequestVerificationToken` shape as the existing `deleteUser()`, OR a plain `<form method="post">` + server confirm page if the executor prefers matching `Group/Delete.cshtml`'s full-page-confirm pattern instead of a JS confirm. **Recommendation:** JS `confirm()` for consistency with the immediately-adjacent `Users.cshtml` pattern this phase is editing in the same commit. |
| Enable account | No confirmation required (restorative, non-destructive) |

## Layout Notes (phase-specific, supplementing the generic contract above)

### Admin Users page (`Views/Admin/Users.cshtml` + `.Mobile.cshtml`)
- No layout change — only the existing "Delete" button (desktop table action column + mobile card action row) is relabeled to "Remove from Group" per the Copywriting Contract above. Table columns, badge logic, and all other buttons (Promote/Demote/Edit/Resend) are unchanged.
- Icon change: `fa-trash` → `fa-user-minus`.
- JS function rename: `deleteUser(id)` → `removeFromGroup(id)` (or keep the name and only change the confirm string — executor's call, but the confirm copy itself is locked per D-02).

### New Platform Users page (`Areas/Platform/Views/Users/Index.cshtml` + `.Mobile.cshtml`)
Single-column table (not the two-column Members-page layout — this is a flat cross-group list, per CONTEXT.md D-04), following the exact `modern-card` scaffold already used by `Group/Index.cshtml`:

- `card-header modern-card-header d-flex justify-content-between align-items-center` — title `"User Accounts"` or `"Platform Users"` with `fas fa-users-cog text-danger me-2`, no create-action button in this header (account creation is not this page's job — it lives on the Members page per Phase 40).
- Table columns: Name, Email, Confirmed (badge, existing pattern), Status (Active/Disabled badge, new), Actions (Disable or Enable button, single button per row based on current state — never show both).
- Empty state: unreachable in practice (SuperAdmin always exists), but include the standard `py-5 text-center` block for defensive consistency with every other list view in the app.
- Mobile variant: stacked `user-card-mobile`-style cards mirroring `Users.Mobile.cshtml`'s existing per-user card block — Name + badges on one row, action button(s) below, consistent with the established mobile card pattern (D-05).

### Platform Group Index header (`Areas/Platform/Views/Group/Index.cshtml` + `.Mobile.cshtml`)
- Add one new button in the existing header-bar `d-flex` row, positioned after "Create Group": `<a asp-controller="Users" asp-action="Index" asp-area="Platform" class="btn btn-secondary"><i class="fas fa-users-cog me-2"></i>Manage Users</a>` — `btn-secondary` (neutral navigation action, not a create/accent action) distinguishes it from the amber-adjacent "Create Group" `btn-success`.
- Mobile variant gets the identical button, stacked per the mobile header pattern already used elsewhere in this Area.

### Status badge convention (new for this phase, follow existing badge shape exactly)
```html
<!-- Active -->
<span class="badge bg-success">
    <i class="fas fa-check-circle me-1"></i>Active
</span>

<!-- Disabled -->
<span class="badge bg-danger">
    <i class="fas fa-user-lock me-1"></i>Disabled
</span>
```

### Action button convention (new for this phase, follow existing button shape exactly)
```html
<!-- Disable (shown when Active) -->
<form asp-action="Disable" method="post" class="d-inline">
    <input type="hidden" name="userId" value="@user.Id" />
    <button type="submit" class="btn btn-sm btn-danger" onclick="return confirm('Disable this account? They will be signed out and unable to log in until re-enabled. No data will be deleted.');">
        <i class="fas fa-user-lock me-1"></i>Disable
    </button>
</form>

<!-- Enable (shown when Disabled) -->
<form asp-action="Enable" method="post" class="d-inline">
    <input type="hidden" name="userId" value="@user.Id" />
    <button type="submit" class="btn btn-sm btn-success">
        <i class="fas fa-user-check me-1"></i>Enable
    </button>
</form>

<!-- Self-disable guard: render disabled control with a tooltip instead of the form above -->
<button type="button" class="btn btn-sm btn-danger" disabled title="You cannot disable your own account.">
    <i class="fas fa-user-lock me-1"></i>Disable
</button>
```
