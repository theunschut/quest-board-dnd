<objective>
Build the SuperAdmin-only Platform Users surface (SAFE-02, SAFE-03): a new `[Area("Platform")] [Authorize(Policy = "SuperAdminOnly")]` `UsersController` with `Index`/`Disable`/`Enable` actions, its desktop + mobile Razor views, a status view model, and an entry-point button in the existing Group Index header bar. Includes the self-disable guard (D-07) and confirms peer-SuperAdmin disable is allowed (D-08).

Purpose: Disable/enable is an account-wide concept that must live outside the group-scoped Admin/Members pages (D-04). This plan delivers the cross-group SuperAdmin view listing every platform user with an Active/Disabled badge, single Disable-or-Enable button per row, and wires those actions to Plan 02's `IIdentityService` primitives. A SuperAdmin cannot disable their own account (D-07) but can disable another SuperAdmin (D-08).

Output: New `UsersController`, `PlatformUserViewModel`, `Users/Index.cshtml` + `Index.Mobile.cshtml`, a "Manage Users" button added to `Group/Index.cshtml` + `.Mobile.cshtml`, and a new `UsersControllerIntegrationTests` covering Disable/Enable/self-guard/peer-SuperAdmin/CSRF.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser → UsersController.Disable/Enable | Authenticated SuperAdmin issues an account-state-mutating POST identifying a target userId |
| SuperAdmin → own account | A SuperAdmin could accidentally lock themselves out with no peer able to recover |
