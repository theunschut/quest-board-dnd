# Phase 69 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

**Scope note:** This phase mechanically repeats Phase 66's Markdown editor + rendering pattern onto Contact Description (dedicated Create/Edit forms) and Contact Notes (inline, multi-instance, toggle-edited list on the Details page itself). Every foundational visual decision — design system, spacing scale, `.editor-toolbar` touch-target overrides, `.markdown-content` typography, color palette, third-party registry, and the CodeMirror/Preview text-color specificity fix — was already locked in `66-UI-SPEC.md`/`67-UI-SPEC.md`/`68-UI-SPEC.md` and is **global CSS scoped by class name, not by field**, so it applies automatically the moment Description/Notes wire into the shared `_MarkdownEditor.cshtml` partial and `Html.Markdown()` helper. This document's job is to (1) confirm the inherited contract explicitly against Contact's specific DOM/CSS, and (2) resolve the two things genuinely new to this phase: Notes' independent per-instance rendering surface (multiple editors on one page, D-03 auto-collapse with silent discard) and a heading/list-item contrast check inside `.note-item`'s lighter card-within-card treatment. `69-CONTEXT.md` and `69-RESEARCH.md` already resolved all interaction-mechanism questions (per-note `ElementId`, lazy-init, D-03 registry) — this spec covers the remaining visual/copy surface only. Still not a React/Next.js/Vite project — shadcn gate not applicable, unchanged from Phase 66/67/68.

## Typography

No new typography surface — Contact Description and every Contact Note render through the exact same `.markdown-content` wrapper Quest/Character fields already use (`Html.Markdown()` in `HtmlHelperExtensions.cs` wraps every field's output identically in `<div class="markdown-content">`, regardless of which field, view, or how many instances render on one page). The table documents the current as-built state, inherited unchanged.

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 16px (1rem) | 400 regular | 1.6 |
| Label | 16px (1rem) | 700 bold | 1.2 |
| Heading | stepped 24px→14px (h1 1.5rem down to h6 0.875rem) | 700 bold | 1.2 |
| Display | not used | — | — |

**Accepted Deviation — Heading row (6 sizes, exceeds standard 4-size guideline), carried forward unchanged:**

Same accepted exception documented in `67-UI-SPEC.md`/`68-UI-SPEC.md`: the 6-level `.markdown-content h1`–`h6` stepped scale (24/22/20/18/16/14px) plus Body/Label (16px) totals 6 unique font sizes. This is already-shipped, operator-approved, class-scoped CSS from Phase 66 that applies app-wide to every `Html.Markdown()` render — not introduced by, or specific to, Contact fields. Phase 69 does not modify this CSS and does not re-litigate the deviation.

**Heading color on Contact Details — resolved for both Description and Notes, both platforms (primary technical finding of this research pass):**

- **Desktop Description (`Views/Contacts/Details.cshtml`):** `@Html.Markdown(Model.Description)` will render as a **direct child** of `.card-body.modern-card-body` (the current `<p class="mb-0" style="white-space: pre-wrap;">` sits at that exact nesting depth today — no intermediate label wrapper, unlike Character's `<div class="mb-3"><label>...</label>...</div>` pattern). This means `markdown-content.css`'s `.modern-card-body > .markdown-content h1..h6 { color: #1a1a1a; text-shadow: none; }` selector structurally matches here. **However**, confirmed by direct read of `site.css`: `.modern-card h1, h2, h3, h4, h5, h6 { color: #F4E4BC !important; ... }` (line 1094) carries `!important` while the compensating override does not — `!important` always wins regardless of specificity or DOM depth, so the direct-child override is effectively **dead CSS** here, exactly as `68-UI-SPEC.md` already established for Character (where the override was structurally inapplicable for an unrelated reason, but reached the same practical outcome). Net effect: Description headings render in the same gold-with-shadow (`#F4E4BC`) as every other field on this card. **This is consistent, not invisible** — matches the established, operator-accepted pattern from every prior phase. **No new desktop CSS needed for Description.**
- **Desktop Notes (`.note-item` inside the same `.modern-card`):** each note's `@Html.Markdown(note.Text)` output nests one level deeper (inside `.note-item`, not a direct child of `.modern-card-body`), so the direct-child override never applies regardless of the `!important` question — moot either way. `.modern-card h1..h6 { color: #F4E4BC !important }` still matches (it has no depth restriction), so note headings render gold-with-shadow too, consistent with note body text's existing (pre-migration) gold-with-shadow rendering confirmed by `69-RESEARCH.md` Pitfall 3. `.note-item`'s own background (`rgba(0, 0, 0, 0.03)` — a near-transparent tint sitting on top of `.modern-card`'s glass/dark-overlay stack, not a light/white surface) keeps this legible. **No new desktop CSS needed for Notes.**
- **Mobile Description (`Views/Contacts/Details.Mobile.cshtml`, `contact-detail.mobile.css`):** the existing catch-all `.contact-detail-card p, .contact-detail-card a, .contact-detail-card span:not(.badge) { color: #F4E4BC !important; ... }` (lines 77–82) re-colors Markdig's `<p>`/`<a>` output consistent with the rest of the mobile card, same pattern as Character's mobile precedent. **Confirmed gap, same shape as Character's WR-02 fix:** the catch-all omits `li`, so a Markdown list in Description would fall back to `.contact-info-value`'s muted `rgba(244, 228, 188, 0.7)` opacity instead of full-opacity gold — a visible mismatch against surrounding paragraph text. **Required companion CSS edit:** add `li` to that selector — `.contact-detail-card p, .contact-detail-card a, .contact-detail-card li, .contact-detail-card span:not(.badge)`.
- **Mobile Notes (`.contact-detail-card .note-item`):** already has its own dedicated scope-out block (`contact-detail.mobile.css` lines 87–92) that reverts the parchment catch-all to `color: inherit` for `.note-item`, `.note-item p`, `.note-item span:not(.badge)` — written in anticipation of this exact migration (confirmed by the file's own comment, "note items keep their own bordered/plain treatment... mirrors the desktop note-item convention"). **Confirmed gap:** this scope-out list also omits `li`, meaning a Markdown list inside a note would NOT inherit the scope-out and would instead fall through to the broader `.contact-detail-card p/a/span:not(.badge)` catch-all one rule up — except `li` isn't in that rule either, so it would fall through further to `.note-item`'s own plain-card treatment (dark text on `rgba(255,255,255,0.85)` white background, set at lines 100-103) via normal inheritance, which is actually the CORRECT/intended look (`.note-item` is deliberately styled as a plain white card, not parchment). **Required companion CSS edit:** add `li` to the scope-out selector for defensive correctness and to make the intent explicit — `.contact-detail-card .note-item, .contact-detail-card .note-item p, .contact-detail-card .note-item li, .contact-detail-card .note-item span:not(.badge) { color: inherit !important; text-shadow: none !important; }` — even though the unstyled fallback likely already renders correctly, an explicit rule removes ambiguity and matches the precedent both Character (WR-02) and this phase's own Description fix establish elsewhere in the same file.

**Other inherited notes (unchanged):**

- Heading scale is `.markdown-content h1 { 24px }` → `h6 14px` (`markdown-content.css`) — applies automatically to any Description/Note heading syntax a user types.
- Blockquote: `border-left: 3px solid rgba(255, 193, 7, 0.5)`, `padding-left: 12px`, italic — unchanged, inherited automatically.
- Links: unmodified Bootstrap default (`var(--bs-link-color)`, underlined) — unchanged.
- **Pre-wrap cleanup:** `.markdown-content { white-space: normal }` (Phase 67, `markdown-content.css`) already neutralizes the doubled-spacing mechanism the moment Description/Notes route through `Html.Markdown()`. Per `69-CONTEXT.md`, remove (not move) both inline `style="white-space: pre-wrap;"` attributes — Description's on `Details.cshtml:91`/`Details.Mobile.cshtml:79`, and each note's `.note-view-text` on `Details.cshtml:145`/`Details.Mobile.cshtml:130` — as a companion edit to the read-view restructuring below.
