# Phase 71 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

**Surface note:** Unlike Phases 66–70, the surface this contract governs is not an app view — it is the Markdown-rendered Quest Description block inside 3 transactional HTML email templates (`QuestFinalized.razor`, `SessionReminder.razor`, `WaitlistPromoted.razor`), plus the truncation behavior around it. Every rule below is inline `style=`, matching the existing email convention — `<style>` blocks are stripped by Gmail, confirmed by `_EmailLayout.razor`'s own head comment. All 3 templates are byte-for-byte identical in the markup this phase touches (Row 3 Description block, Row 6 CTA button), so every rule applies identically to all 3 unless noted otherwise.

## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Gap between a list item and the next; blockquote inner top/bottom padding |
| sm | 8px | Gap after a Small heading (h5/h6) |
| md | 12px | Bottom margin between block elements (paragraphs, headings→body, lists, blockquotes) inside the Description column — tighter than the app's own 16px default specifically to fit more structured content inside the fixed 840px card |
| lg | 16px | Top margin above Large/Medium headings (h1–h4); existing Row padding (`padding:16px 20%` on Row 3/5) |
| xl | 20% (existing) | Row 3/4/5 horizontal column padding — pre-existing template convention, untouched by this phase |

Exceptions:
- Percentage-based row padding (`padding:16px 20%`, `padding:5% 0 0 8%`, etc.) is pre-existing template convention across Rows 1–6, not introduced by this phase — left as-is.
- 12px (not one of the app's named 8-pt tokens elsewhere) is used as the standard inter-block margin specifically for this Description column, because the fixed-height card leaves limited vertical budget; it is still a multiple of 4.
- The first block element rendered inside the Description column must have `margin-top:0` regardless of its own declared top margin (see Implementation Notes) — prevents a double gap below the Row 2 subtitle.
- The last block element rendered before the truncation point (or before the natural end of short content) must have `margin-bottom:0` — prevents extra whitespace before the Row 4 gold divider.

### Markdown Block-Element Typography (Email) — new for Phase 71

The Description column's outer wrapper stays `font-size:15px;font-family:Georgia,serif;color:#1a0f08;line-height:1.6;` (font-style:italic on plain body text only — see per-element rules below). **Every block element Markdig can emit inside Quest Description must carry its own full explicit inline `style=` attribute — do not rely on inheritance from the outer wrapper div.** Outlook's Word rendering engine does not reliably cascade color/font-family/line-height through nested block elements; restating every property on every element is the only bulletproof approach, matching how every other element in these 3 templates already restates its own font/color/text-shadow rather than inheriting it.

All colors, fonts, and text-shadow below match the templates' existing palette exactly (`#1a0f08` ink, Georgia serif, the standard legibility text-shadow already applied to every other text node in these templates). Do not introduce new colors.

3-tier heading scale (covers all 6 Markdig heading levels a user could type; consolidated to 3 sizes so the ladder stays visually calm inside a ~360px-wide column):

| Tag | Style |
|-----|-------|
| `h1`, `h2` (Large) | `font-size:20px;font-weight:700;font-style:normal;font-family:Georgia,serif;line-height:1.25;color:#1a0f08;margin:16px 0 8px 0;text-shadow:2px 2px 4px rgba(255,255,255,0.9),1px 1px 6px rgba(0,0,0,0.5);` |
| `h3`, `h4` (Medium) | `font-size:18px;font-weight:700;font-style:normal;font-family:Georgia,serif;line-height:1.25;color:#1a0f08;margin:16px 0 8px 0;text-shadow:2px 2px 4px rgba(255,255,255,0.9),1px 1px 6px rgba(0,0,0,0.5);` |
| `h5`, `h6` (Small) | `font-size:16px;font-weight:700;font-style:normal;font-family:Georgia,serif;line-height:1.3;color:#1a0f08;margin:16px 0 4px 0;text-shadow:2px 2px 4px rgba(255,255,255,0.9),1px 1px 6px rgba(0,0,0,0.5);` |

Headings render **upright (`font-style:normal`), never italic** — this deliberately breaks from the surrounding italic flavor-text style so structural headings read as structure, not narrative prose. This mirrors `markdown-content.css`'s web convention, which also never italicizes headings.

Body-level and inline elements:

| Element | Style |
|---------|-------|
| `p` (paragraph) | `font-size:15px;font-weight:400;font-style:italic;font-family:Georgia,serif;line-height:1.6;color:#1a0f08;margin:0 0 12px 0;text-shadow:2px 2px 4px rgba(255,255,255,0.9),1px 1px 6px rgba(0,0,0,0.5);` |
| `ul` | `margin:0 0 12px 0;padding:0 0 0 20px;list-style-position:outside;` |
| `ol` | `margin:0 0 12px 0;padding:0 0 0 20px;list-style-position:outside;` |
| `li` (inside `ul`) | `margin:0 0 4px 0;font-size:15px;font-weight:400;font-style:italic;font-family:Georgia,serif;line-height:1.5;color:#1a0f08;list-style-type:disc;display:list-item;text-shadow:2px 2px 4px rgba(255,255,255,0.9),1px 1px 6px rgba(0,0,0,0.5);` |
| `li` (inside `ol`) | same as above with `list-style-type:decimal;` |
| `blockquote` | `margin:0 0 12px 0;padding:4px 0 4px 12px;border-left:3px solid #FFD700;font-size:15px;font-weight:400;font-style:italic;font-family:Georgia,serif;line-height:1.6;color:#1a0f08;text-shadow:2px 2px 4px rgba(255,255,255,0.9),1px 1px 6px rgba(0,0,0,0.5);` |
| `a` (autolink / `[text](url)`) | `color:#8B4513;font-weight:700;text-decoration:underline;` — brown, not gold: gold (`#FFD700`) text fails contrast against the light parchment background and is reserved for CTA-button/badge treatments, not body-copy links |
| `strong` | `font-weight:700;` (explicit, even though `<strong>` already renders bold by default — bulletproof-email convention: never rely on a client's default tag styling) |
| `em` | `font-style:italic;` (explicit, same rationale) |
| `hr` (thematic break, if a user types `---`) | `border:none;border-top:1px solid #FFD700;margin:12px 0;` — matches the existing Row 4 gold-divider convention exactly |

Out of scope for this phase's styling (per REQUIREMENTS.md "Out of Scope" — no code/table toolbar buttons exist): `code`, `pre`, `table`/`thead`/`tbody`/`tr`/`th`/`td`, `dl`/`dt`/`dd`, `del`, `sup`, `input` remain in `EmailSanitizer.AllowedTags` (pre-existing) but receive **no new styling** in this phase — they render with each email client's bare default styling if a user manually types the underlying Markdown syntax. This is a deliberate scope boundary, not an oversight.

## Truncation & Read-More Contract

Resolves D-01–D-04 from `71-CONTEXT.md`. The 840px fixed-height poster card (`<td style="height:840px;overflow:hidden;">`) **stays exactly as-is** — it is load-bearing for the poster background image on all 3 templates and is explicitly not being made dynamic (D-01).

1. **No scrolling.** The inner `<div style="height:100%;overflow-y:auto;padding-right:6px;">` wrapper around Description (Row 3, all 3 templates) must be removed/simplified — Outlook's Word rendering engine does not support CSS `overflow`/scroll on `<div>` at all, so today's scroll trick silently clips content in Outlook specifically. Row 3's `<td>` keeps `vertical-align:top`; the Description content div no longer needs `height:100%` or `overflow-y:auto`.
2. **Truncate at a block boundary, preserving Markdown structure.** When rendered Description HTML would exceed the available space, cut after the last complete top-level block element (a full `<p>`, `<ul>`, `<ol>`, `<h1>`–`<h6>`, or `<blockquote>`) — never mid-element. The truncated HTML must remain valid, well-formed markup up to the cut point. Do not fall back to `ExtractPlainText()`-style flat-text truncation (that precedent is deliberately not reused here, per D-03/66-CONTEXT.md).
3. **Starting size heuristic (tune during D-05 live verification, not a hard pixel guarantee):** target roughly 5 top-level block elements or ~650 characters of extracted plain text inside the Description column, whichever limit is reached first — enough for a short heading + a few paragraphs/list items, sized to reliably leave room for Rows 4–6 inside the 840px frame. The exact number is expected to be adjusted once the operator confirms real rendering in Outlook desktop and Gmail webmail (D-05); this UI-SPEC does not lock an exact pixel-perfect cutoff.
4. **Visible truncation indicator, never a silent cutoff.** When truncated, append the exact copy `…continue reading on the quest board` as an inline link immediately after the last preserved block's closing tag (own new trailing element, e.g. a `<p>` containing just the link, using the same `margin:0` bottom-of-column rule), styled per the Copywriting Contract row above. It must read as a natural continuation of the visible content, not as an error banner or warning.
5. **No new button.** The truncation link's `href` is the same `@QuestUrl` already used by the existing Row 6 "View Quest Details" button — do not introduce a second distinct URL or CTA style.
6. **First/last element margin correction.** After truncation (or on short untruncated content), explicitly zero out the first rendered block's `margin-top` and the last rendered block's (or the read-more line's) `margin-bottom`, per the Spacing Scale exceptions above.

## Outlook / Gmail Compatibility Notes

- **Bullet visibility in Outlook desktop is the other named risk this phase closes.** The `list-style-type:disc`/`decimal` + `display:list-item` inline styles specified above are the required minimum. If live D-05 verification in real Outlook desktop still shows missing bullets, add an MSO-conditional-comment fallback bullet character inside each `<li>` (e.g. `<!--[if mso]>&#8226;&nbsp;<![endif]-->` prefix) as a belt-and-suspenders fix — the exact fallback technique is implementation discretion (per `71-CONTEXT.md`), not re-specified here; this note exists so the executor doesn't have to rediscover the pitfall.
- **`style` attribute must be added to `EmailSanitizer.AllowedAttributes`** (`QuestBoard.Domain/Services/MarkdownService.cs`) — every inline style declared in this document depends on that attribute surviving sanitization. `BaseAllowedTags` already includes every tag this contract styles; no new tag allowlisting is needed.
- **Restating properties beats inheritance.** Every rule above is written as a complete, self-contained inline style (color, font, size, line-height, text-shadow all repeated per element) specifically because Outlook's Word engine has known gaps in cascading inherited properties through nested block/list elements. Do not "simplify" by stripping repeated properties in favor of relying on the outer wrapper div's style.
- **Gmail webmail** already correctly ignores `<style>` blocks (confirmed, this is the reason the whole template convention is 100% inline styles) and generally respects standard inline CSS on block/list elements without special-casing — the Gmail risk in this phase is exclusively the truncation/clipping behavior (item 1 above), not the styling gap.
