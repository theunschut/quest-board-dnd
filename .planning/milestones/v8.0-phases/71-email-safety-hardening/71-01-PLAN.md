<objective>
Build the single new email-rendering method `RenderEmailHtml` on `IMarkdownService`/`MarkdownService` that closes both pre-existing gaps for the 3 quest email templates: (1) EMAILMD-02 — every heading/list/blockquote/inline element Markdig emits gets an explicit email-safe inline `style=` plus an Outlook bullet-visibility fallback, and (2) EMAILMD-03 — long structured content is truncated at a complete block boundary with an explicit "…continue reading on the quest board" link instead of being silently clipped by the fixed-height poster card.

Purpose: This is the Domain-layer engine the whole phase rests on. All styling, Outlook-bullet, truncation, and read-more logic live in one cohesive AngleSharp DOM pass here so the 3 Razor templates (wired in Plan 02) only change a single call site.

Output: `RenderEmailHtml(string? markdown, string readMoreUrl, int maxTopLevelBlocks = 5, int maxPlainTextChars = 650)` with full xUnit coverage (satisfies this phase's Wave 0 test requirement).

**Mechanism decision (resolves a source conflict — executor MUST follow this):** `71-UI-SPEC.md` line 137 states `style` must be added to `EmailSanitizer.AllowedAttributes`. Do NOT do that. `71-RESEARCH.md` Pitfall 1 and `71-PATTERNS.md` ("Sanitizer separation") supersede that note: routing styles through the sanitizer risks the `AngleSharp.Css 0.17.0` whole-attribute-drop defect, and `AllowedAttributes` is a HashSet shared with `WebSanitizer` (changing it would regress every Web-target Markdown field across Phases 66-70). `71-CONTEXT.md` marks the injection mechanism as Claude's Discretion. Inject styles via `IElement.SetAttribute("style", ...)` strictly AFTER `EmailSanitizer.Sanitize()` has already run. The UI-SPEC's exact style *strings* remain authoritative and are copied verbatim.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| user Markdown (QuestDescription) → EmailSanitizer | Untrusted user-authored Markdown is converted to HTML and sanitized (existing Phase 65 boundary, unchanged) |
| EmailSanitizer output → RenderEmailHtml DOM pass | Already-sanitized, server-trusted HTML; only compile-time-constant styles/comments and the server-built readMoreUrl are added downstream |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-71-01 | Tampering/Elevation | RenderEmailHtml style injection | high | mitigate | Style strings are compile-time C# constants keyed by tag name; never built from any substring of `markdown` or the sanitized HTML. Enforced by the fixed-value styling unit tests. |
| T-71-02 | Tampering | MSO bullet-fallback comment | medium | mitigate | Comment content is a single hard-coded literal (`[if mso]>&#8226;&nbsp;<![endif]`), never parameterized by request data. |
| T-71-03 | Tampering | Truncation OuterHtml re-serialization | low | accept | `OuterHtml` operates on the already-EmailSanitizer-sanitized DOM; no raw markdown is re-parsed or re-introduced during truncation. |
| T-71-04 | Information Disclosure | read-more link href | low | mitigate | `readMoreUrl` is the server-built QuestUrl (not user content); injected via `SetAttribute`, which DOM-encodes the attribute value. |
| T-71-SC | Tampering | package installs | low | accept | No new packages this phase — Markdig/HtmlSanitizer/AngleSharp already installed and vetted in Phase 65; no `PackageReference` changes. |
