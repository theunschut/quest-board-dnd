# Phase 68 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

**Scope note:** This phase mechanically repeats Phase 66's Markdown editor + rendering pattern onto Character Description and Character Backstory, on all 4 write forms (`Create.cshtml`, `Create.Mobile.cshtml`, `Edit.cshtml`, `Edit.Mobile.cshtml`) and both read views (`Details.cshtml`, `Details.Mobile.cshtml`). Every foundational visual decision — design system, spacing scale, `.editor-toolbar` touch-target overrides, `.markdown-content` typography, color palette, third-party registry — was already locked in `66-UI-SPEC.md`/`67-UI-SPEC.md` and is **global CSS scoped by class name, not by field**, so it applies automatically to Character Description/Backstory the moment they're wired to the shared `_MarkdownEditor.cshtml` partial and `Html.Markdown()` helper. No RESEARCH.md exists for this phase (mechanical repeat, research skipped) — this document's technical reference is `66-UI-SPEC.md` + `67-UI-SPEC.md`/`67-REVIEW.md` plus direct inspection of `Views/Characters/*.cshtml` and `wwwroot/css/character-*.css` performed during this research pass. This document's job is to (1) confirm the inherited contract explicitly, and (2) resolve the one genuinely new risk this phase introduces: Character's card CSS uses a `:not(.badge)` pseudo-class in its mobile catch-all text-color rules that Quest's equivalent rules never used, which changes the CSS-specificity math behind Phase 66's already-shipped editor-color fix. Still not a React/Next.js/Vite project — shadcn gate not applicable, unchanged from Phase 66/67.

**Notes — heading color on Character Details, resolved for both platforms this phase (this is the primary technical finding of this research pass):**

Phase 67's `67-REVIEW.md` (WR-01) found that `.markdown-content h1`–`h6`'s hardcoded `color: #F4E4BC` assumes every render site is a dark-overlay box, and can render "nearly invisible" pale-on-light where that assumption doesn't hold — flagged by `68-CONTEXT.md` as directly relevant here. Verified against Character Details' actual DOM/CSS:

- **Desktop (`Views/Characters/Details.cshtml`):** Description/Backstory render inside `<div class="mb-3"><label>...</label>@Html.Markdown(...)</div>`, nested inside `.card-body.modern-card-body` inside `.card.modern-card` — the same container class every other field on this card (Level, Classes, Sheet Link) already uses. `site.css`'s pre-existing `.modern-card h1, h2, h3, h4, h5, h6 { color: #F4E4BC !important; text-shadow: ...!important; }` and `.modern-card p, li, span, small { color: #F4E4BC !important; ... }` rules (lines 1094–1116, predate this milestone) match any heading/paragraph/list-item nested anywhere inside `.modern-card` regardless of intermediate wrapper depth, and — because they carry `!important` — they win over `.markdown-content h1..h6`'s own non-`!important` color declaration regardless of specificity. Net effect: Description/Backstory headings and body text render in the same gold-with-shadow (`#F4E4BC`) already used by every other field on this card. **This is consistent, not invisible** — do not port Quest/Manage's `.modern-card-body > .markdown-content h1..h6 { color: #1a1a1a }` compensating override into Character Details; that selector requires `.markdown-content` to be a *direct child* of `.modern-card-body` (true on `Quest/Manage.cshtml`, false here — Character Details always nests `.markdown-content` one level deeper inside a `<div class="mb-3">` label wrapper), and even where it is structurally applicable it does not win against the `!important` rule above, so it would be dead CSS if copied here. **No new heading-color CSS is needed for desktop Character Details.**
- **Mobile (`Views/Characters/Details.Mobile.cshtml`, `character-detail.mobile.css`):** the existing catch-all `.character-detail-card p, .character-detail-card a, .character-detail-card span:not(.badge) { color: #F4E4BC !important; text-shadow: ...; }` (lines 82–87) similarly re-colors Markdig's `<p>`/`<a>` output consistent with the rest of the mobile card. **Gap:** the catch-all does not include `li`, so `<ul>/<ol><li>` items from a Markdown list would fall back to the ancestor `.character-info-value`'s *muted* `rgba(244, 228, 188, 0.7)` color (line 57–60) instead of the catch-all's full-opacity `#F4E4BC` — a visible opacity mismatch between list items and surrounding paragraph text in the same rendered block. **Required companion CSS edit:** add `li` to the selector list — `.character-detail-card p, .character-detail-card a, .character-detail-card li, .character-detail-card span:not(.badge)` — in `character-detail.mobile.css`, so list items match full-opacity gold like every other text node in the card.

**Other inherited notes (unchanged):**

- Heading scale is `.markdown-content h1 { 24px }` → `h6 14px` (`markdown-content.css`) — applies automatically to any Description/Backstory heading syntax a user types.
- Blockquote: `border-left: 3px solid rgba(255, 193, 7, 0.5)`, `padding-left: 12px`, italic — unchanged, inherited automatically.
- Links: unmodified Bootstrap default (`var(--bs-link-color)`, underlined) — unchanged.
- **Pre-wrap cleanup:** `.markdown-content { white-space: normal }` (Phase 67, `markdown-content.css`) already neutralizes any inherited `white-space: pre-wrap` from `.character-info-value`/`.form-control-plaintext`, applying automatically the moment Description/Backstory route through `Html.Markdown()`. Per `68-CONTEXT.md`, the two inline `style="white-space: pre-wrap;"` attributes on `Details.cshtml:151,159`'s `<p class="form-control-plaintext">` wrappers are removed as part of the same restructuring that drops the illegal `<p>`-around-block-content pattern (see Copywriting Contract → "Read-view restructuring" row); `character-detail.mobile.css:63-65`'s shared `.character-info-value { white-space: pre-wrap }` rule may be left in place or removed — both are correct per `68-CONTEXT.md`'s Claude's-Discretion note, since `.markdown-content`'s own override neutralizes it either way and the only other consumers of that class (`Owned by`, `Level`) are confirmed single-line values where pre-wrap vs. normal is visually identical.

**Required CSS fix — editor/Preview text-color leak on mobile write forms (new finding this phase, not present in Quest's equivalent CSS):**

Phase 66 fixed a bug where `.modern-card`'s gold-with-shadow text rule leaked into EasyMDE's raw CodeMirror editing surface and Preview pane, via a specificity-bump override in `markdown-editor.css`:
```css
.CodeMirror.CodeMirror,
.CodeMirror.CodeMirror *,
.editor-preview.editor-preview,
.editor-preview.editor-preview * {
    color: #000 !important;
    text-shadow: none !important;
}
```
This is a **global** class selector (not scoped to `.modern-card`), so it was inherited "for free" by Quest Rewards/Recap in Phase 67 with zero new CSS. It is **not** sufficient for Character's mobile write forms. `character-form.mobile.css`'s catch-all (line 22–27) is:
```css
.character-form-card p,
.character-form-card span:not(.badge),
.character-form-card legend {
    color: #F4E4BC !important;
    text-shadow: 2px 2px 4px rgba(0,0,0,0.9), -1px -1px 2px rgba(0,0,0,0.9);
}
```
The `:not(.badge)` pseudo-class contributes a class-level specificity point identical to `.CodeMirror.CodeMirror`'s doubled class, but `.character-form-card span:not(.badge)` additionally carries a type selector (`span`) that `.CodeMirror.CodeMirror *`'s trailing universal selector does not — so for `<span>` elements specifically (CodeMirror's own `cm-header`/`cm-strong`/`cm-emphasis`/`cm-link`/`cm-quote` syntax-highlighting spans), `.character-form-card span:not(.badge)` (specificity 0,2,1) **outranks** `.CodeMirror.CodeMirror *` (specificity 0,2,0) even though both carry `!important` — ties on `!important` and class-count are broken by type-selector count, and the leak rule wins. Net effect: on `Create.Mobile.cshtml`/`Edit.Mobile.cshtml`, styled Markdown syntax (`**bold**`, `# heading`, etc.) typed into the raw editor would render in gold-with-drop-shadow instead of the intended plain black while actively editing — reproducing the exact readability bug Phase 66 fixed for desktop, on mobile Character forms specifically, and confirmed (via grep) to be a genuinely new pattern: Quest's mobile CSS never used `:not(.badge)` in its equivalent catch-all rules, so this specificity gap never existed on Quest's forms.

`.editor-preview` (`<p>` tags in the Preview pane) is unaffected — `.character-form-card p` (no `:not()`, specificity 0,1,1) loses to `.editor-preview.editor-preview *` (0,2,0) on class count alone, so Preview text renders correctly black without any fix needed.

**Fix:** strengthen `markdown-editor.css`'s override to a specificity that cannot be beaten by any `:not()`-qualified single-type-selector catch-all, by tripling the class (mirrors the exact "specificity bump via doubled class" pattern already established in `66-07-SUMMARY.md`, taken one level further):
```css
.CodeMirror.CodeMirror.CodeMirror,
.CodeMirror.CodeMirror.CodeMirror *,
.editor-preview.editor-preview.editor-preview,
.editor-preview.editor-preview.editor-preview * {
    color: #000 !important;
    text-shadow: none !important;
}
```
This is a global, class-scoped edit (same file, same selector family Phase 66 already introduced) — apply it once; it protects every current and future field's editor, not just Character's. Include a live-browser check of the mobile Description/Backstory editors' text color (typing bold/heading syntax) in this phase's verification checkpoint, the same way Phase 66's original fix was only caught during human verification. Note for awareness (not this phase's scope to fix): `contact-form.mobile.css` uses the identical `:not(.badge)` pattern, so Phase 69 (Contact Fields) will need this same fix already in place — confirm it lands from this phase rather than being redone.
